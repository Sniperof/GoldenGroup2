\*\*أفضل إدخال لـ Branch-Based Access Control\*\*

الأنظف عملياً في هذا المشروع هو:



1\. \*\*الإبقاء على permissions كقاموس عالمي\*\*

&#x20;  - مثل `employees.view`, `employees.create`, `jobs.manage`

&#x20;  - لا تربط permission نفسها بالفرع.



2\. \*\*فصل branch scope عن role definition\*\*

&#x20;  - الصلاحية تقول: "ماذا يستطيع أن يفعل؟"

&#x20;  - branch scope يقول: "على أي فروع يستطيع أن يفعل ذلك؟"



3\. \*\*الخيار الموصى به\*\*: `user\_branch\_assignments`

&#x20;  - إذا كان المستخدم قد ينتمي لأكثر من فرع الآن أو مستقبلاً، فهذا هو التصميم الصحيح.

&#x20;  - جدول مقترح: `user\_branch\_assignments(user\_id, branch\_id, is\_primary, scope\_type?)`

&#x20;  - الـ user يملك role/roles، ويملك أيضاً branch assignments.

&#x20;  - عند كل request نحدد `acting\_branch\_id` من header/context، ثم نتحقق:

&#x20;    - هل لدى المستخدم permission المطلوبة؟

&#x20;    - هل `acting\_branch\_id` ضمن branches المسموحة له؟

&#x20;    - وهل resource نفسه ينتمي لنفس الفرع؟



4\. \*\*لماذا لا أوصي بـ role scoped by branch كخيار أساسي؟\*\*

&#x20;  - أنتم بدأتم بهذا فعلاً جزئياً عبر templates/clones في \[migrations/015\_role\_templates\_seed.sql](D:\\OneDrive\\سطح المكتب\\golden-crm-clean\\migrations\\015\_role\_templates\_seed.sql).

&#x20;  - هو مقبول إذا كانت لكل فرع نسخ مستقلة من الأدوار لأسباب تشغيلية.

&#x20;  - لكن لا يجب أن يكون هو الآلية الوحيدة للـ branch isolation، لأن العزل الحقيقي يجب أن يُفرض أيضاً على resource access في الـ middleware/query layer.

&#x20;  - بعبارة أخرى: branch-scoped roles مفيد للإدارة، لكنه غير كافٍ كطبقة enforcement.



5\. \*\*لماذا لا أوصي بـ permission scopes داخل permission string؟\*\*

&#x20;  - مثل `employees.view.branch.7` أو `employees.view.own\_branch`.

&#x20;  - هذا يضخم القاموس، يصعّب الإدارة، ويخلط بين capability وscope.

&#x20;  - مناسب فقط إذا بنيتم policy engine متقدم لاحقاً، وليس الآن.



\*\*الخلاصة\*\*:  

أفضل target عملي هو \*\*RBAC + branch assignments + centralized scope enforcement\*\*.

\*\*التغييرات الموصى بها على الـ Middleware\*\*

1\. `requireAuth` فقط للمصادقة.

2\. `authorize(permission, options)` هو المدخل الموحد لكل authorization.

3\. `options` تحتوي مثلاً:

&#x20;  - `branchMode: 'none' | 'acting' | 'resource'`

&#x20;  - `resourceBranchResolver`

4\. منع أي write إذا لم يُحدد `acting\_branch\_id` عندما يكون المستخدم متعدد الفروع أو HQ.

5\. super admin bypass يبقى مسموحاً، لكن فقط داخل طبقة `authorize` نفسها وليس scattered.



\---



\*\*استراتيجية حماية الـ API\*\*

1\. كل endpoint غير public يجب أن يمر عبر `requireAuth`.

2\. كل endpoint business-sensitive يجب أن يمر عبر `authorize(permission, scope)`.

3\. أي endpoint يقرأ/يعدل بيانات مرتبطة بفرع يجب أن يثبت branch scope بطريقتين:

&#x20;  - branch requested/acting branch

&#x20;  - branch of target resource from DB

4\. لا تعتمدوا على الـ UI لإخفاء الأزرار كتعويض عن حماية السيرفر.

5\. لا تكرروا منطق الحماية في route وservice بصيغ مختلفة؛ اجعلوا service تستخدم context موحداً.
