\# تقرير تحليل: تشكيل الفريق والتبعية الحالية للزبون

\## Golden CRM — تحليل نظام، بدون تعديل



\---



\## A) تشكيل الفريق



\### هل يوجد كيان فريق فعلي؟



\*\*لا. لا يوجد جدول `teams` في قاعدة البيانات.\*\* الفريق في النظام الحالي ليس كياناً مستقلاً، بل هو \*\*بنية مؤقتة مرتبطة باليوم\*\* محفوظة داخل JSONB في جدول `day\_schedules`.



\### كيف يُمثَّل الفريق حالياً؟



\*\*المستوى الأول — TeamSlot في جدول day\_schedules:\*\*



الواجهة الكاملة من \[`shared/types.ts:445`](packages/shared/types.ts):

```typescript

export interface TeamSlot {

&#x20;   supervisor: number | null;      // employee\_id

&#x20;   technician: number | null;      // employee\_id

&#x20;   telemarketers?: number\[];       // مصفوفة employee\_ids

&#x20;   trainee?: number | null;        // employee\_id (migration 044)

}



export interface SoloSlot {

&#x20;   technician: number;             // فني منفرد

}



export interface DaySchedule {

&#x20;   teams: TeamSlot\[];              // قائمة فرق اليوم

&#x20;   solos: SoloSlot\[];             // قائمة الفنيين المنفردين

}

```



\*\*المستوى الثاني — team\_key كمعرّف نصي:\*\*



حقل `team\_key VARCHAR(100)` في `marketing\_visits` وجدول `route\_assignments`. يُستخدم كمعرّف للفريق في التخطيط والجدولة، لكنه مجرد نص وليس مفتاح لجدول حقيقي.



\*\*الجدول الموجز:\*\*



| السؤال | الإجابة |

|--------|---------|

| هل يوجد كيان فريق فعلي (جدول)؟ | \*\*لا\*\* — الفريق JSONB مؤقت في `day\_schedules` |

| هل الفريق مرتبط بفرع؟ | \*\*نعم\*\* — `day\_schedules` مرتبط بـ `branch\_id` |

| هل الفريق يحتوي أعضاء؟ | \*\*نعم\*\* — عبر `employee\_id` داخل `TeamSlot` |

| هل لكل عضو دور داخل الفريق؟ | \*\*نعم\*\* — `supervisor`, `technician`, `trainee`, `telemarketers\[]` |

| هل يدعم مشرفة + فني + متدرب + تيلماركتر؟ | \*\*نعم\*\* — كلهم موجودون في `TeamSlot` |

| هل يمكن فريق فني فقط (بلا مشرفة)؟ | \*\*نعم\*\* — `supervisor: null, technician: X` مسموح |

| هل يمكن فني منفرد بلا فريق؟ | \*\*نعم\*\* — عبر `SoloSlot` منفصل |

| هل الفريق دائم أم مؤقت؟ | \*\*مؤقت\*\* — مرتبط بيوم تشغيل محدد في `day\_schedules` |

| هل يوجد team\_key بدل جدول فريق؟ | \*\*نعم\*\* — `team\_key` نص في `marketing\_visits` و`route\_assignments` |

| أين يُنشأ الفريق في الواجهة؟ | في صفحة الجدولة اليومية عند بناء خطة اليوم |

| أين يُنشأ في API؟ | عبر `/api/schedules` وحفظ `day\_schedules` |

| أين يُحفظ في DB؟ | `day\_schedules.schedule JSONB` — مصفوفة `TeamSlot` و`SoloSlot` |



\*\*ملاحظة مهمة:\*\* جدول `route\_assignments` ليس جدول إسناد للزبائن — هو لتخصيص مناطق العمل (routes + extra\_zones) لكل `team\_key` في التخطيط التسويقي.

\## B) التبعية الحالية للزبون



\### بنية الإسناد في قاعدة البيانات



\*\*جدول `client\_assignments` (migration 042) — العلاقة M2M:\*\*

```sql

client\_assignments (

&#x20;   id           SERIAL PRIMARY KEY,

&#x20;   client\_id    INTEGER NOT NULL REFERENCES clients(id) ON DELETE CASCADE,

&#x20;   hr\_user\_id   INTEGER NOT NULL REFERENCES hr\_users(id) ON DELETE CASCADE,

&#x20;   assigned\_at  TIMESTAMPTZ DEFAULT NOW(),

&#x20;   assigned\_by  INTEGER REFERENCES hr\_users(id) ON DELETE SET NULL,

&#x20;   UNIQUE (client\_id, hr\_user\_id)

)

```



\*\*جدول `clients` — الحقول ذات الصلة:\*\*

```sql

clients (

&#x20;   branch\_id   INTEGER REFERENCES branches(id),   -- الفرع التابع له الزبون

&#x20;   created\_by  INTEGER REFERENCES hr\_users(id),    -- من أنشأ السجل (غير قابل للتغيير)

&#x20;   -- لا يوجد owner\_user\_id أو assigned\_hr\_user\_id بعد migration 042

)

```



\*\*الجدول الموجز:\*\*



| السؤال | الإجابة |

|--------|---------|

| هل يمكن أن يكون للزبون أكثر من شخص مسند؟ | \*\*نعم\*\* — M2M كامل، UNIQUE على (client\_id, hr\_user\_id) |

| هل التبعية محفوظة كإسناد مباشر؟ | \*\*نعم\*\* — في `client\_assignments` بجدول مستقل |

| هل التبعية مرتبطة بالمشرفة فقط؟ | \*\*لا\*\* — أي hr\_user يملك صلاحية `clients.can\_be\_assigned` |

| من يمكن أن يكون مسنداً؟ | ADMIN, BRANCH\_MANAGER, SALES, \*\*TELEMARKETER\*\* — وليس TECHNICIAN |

| هل يمكن أن يكون الزبون تابعاً لفني؟ | \*\*لا حالياً\*\* — TECHNICIAN لا يملك `clients.can\_be\_assigned` |

| هل يمكن أن يكون لعدة مشرفات؟ | \*\*نعم\*\* — بدون حد أقصى |

| هل توجد تبعية عامة للفرع؟ | \*\*ضمنياً\*\* — زبون بلا إسناد يُرى من مستخدمي scope=BRANCH |

| هل created\_by منفصل عن التبعية الحالية؟ | \*\*نعم\*\* — حقل تاريخي لا يعطي أي صلاحية تشغيلية |

| هل خروج الموظف يؤثر على التبعية؟ | \*\*يبقى السجل\*\* — `ON DELETE CASCADE` فقط عند حذف hr\_user نهائياً، لكن `is\_active=FALSE` لا يزيله من assignments |

| هل يوجد مفهوم موظف غير نشط؟ | \*\*نعم\*\* — `hr\_users.is\_active` حقل boolean، لكن deactivation لا تُزيل الإسنادات القائمة |



\### قاعدة `clients.can\_be\_assigned` — من يظهر في قائمة الإسناد؟



من migration 043:



| الدور | يظهر في قائمة الإسناد؟ |

|-------|----------------------|

| ADMIN | ✓ نعم |

| BRANCH\_MANAGER | ✓ نعم |

| SALES | ✓ نعم |

| TELEMARKETER | ✓ نعم |

| CUSTOMER\_SERVICE\_SUPERVISOR | ✗ لا |

| TECHNICIAN | ✗ لا |

| ACCOUNTANT | ✗ لا |

| HR\_MANAGER | ✗ لا |



\*\*ملاحظة مهمة:\*\* `CUSTOMER\_SERVICE\_SUPERVISOR` يملك صلاحية `clients.view\_list` بـ scope=ASSIGNED (يرى فقط الزبائن المسندين له) — لكنه لا يظهر في قائمة الإسناد. يُضاف للإسناد \*\*تلقائياً فقط\*\* إذا أنشأ الزبون بنفسه (auto self-assign).

\## C) الفرق بين منشئ الاسم والتبعية الحالية



\### آلية auto self-assign — \[`clients.ts:382`](packages/api/routes/clients.ts)



```typescript

async function resolveAssignmentUserIds(rawIds, selfId, isSuperAdmin) {

&#x20;   const ids = Array.isArray(rawIds) ? rawIds.map(Number) : \[];

&#x20;   

&#x20;   // Non-super-admin يُضاف تلقائياً لإسناداته الخاصة

&#x20;   if (!isSuperAdmin \&\& !ids.includes(selfId)) {

&#x20;       ids.push(selfId);

&#x20;   }

&#x20;   // التحقق فقط من وجود المستخدم في hr\_users (لا يتحقق من can\_be\_assigned)

&#x20;   const { rows } = await pool.query('SELECT id FROM hr\_users WHERE id = ANY($1)', \[ids]);

&#x20;   return rows.map(r => r.id);

}

```



\*\*الجدول الموجز:\*\*



| السؤال | الإجابة |

|--------|---------|

| هل `created\_by` يُستخدم فقط كمعلومة تاريخية؟ | \*\*نعم\*\* — يُعرض في الواجهة كـ "أُنشئ بواسطة" فقط |

| هل `created\_by` يعطي صلاحية أو تبعية تشغيلية؟ | \*\*لا\*\* — لا يدخل في scope check ولا في `client\_assignments` |

| هل `client\_assignments` هي التي تحدد التبعية؟ | \*\*نعم\*\* — هي المصدر الوحيد للتبعية التشغيلية |

| إذا أدخل فني زبوناً، هل يُسند له؟ | \*\*نعم تلقائياً\*\* (إذا كان غير super-admin فهو يُضاف لإسناداته) — لكن TECHNICIAN ليس في قائمة `can\_be\_assigned` مما يعني أن هذه حالة نادرة إذ لا يملك TECHNICIAN صلاحية `clients.create` أصلاً |

| إذا أدخلت مشرفة زبوناً، هل تصبح مسندة له تلقائياً؟ | \*\*نعم\*\* — auto self-assign إلزامي للمستخدمين غير super-admin |

| إذا رُبط Candidate بزبون موجود، هل منشئ Candidate يدخل ضمن التبعية؟ | \*\*لا\*\* — `linkCandidateToClient` لا تُعدّل `client\_assignments` إطلاقاً |



\*\*الخلاصة العملية:\*\* لكل مستخدم غير super-admin، يُصبح `created\_by` هو نفس أول مُسنَد (بسبب auto self-assign)، لكن هذا تطابق عملي وليس ترابطاً معمارياً. الحقلان مفصولان قصداً: `created\_by` ثابت، `assignments` متغير.

\## D) ما الذي يظهر لمدير الفرع؟



\### نموذج الصلاحيات حسب الـ Scope — \[`clientPolicy.ts:38`](packages/api/policies/clientPolicy.ts)



```typescript

export function getClientListAccessPlan(context: AuthContext) {

&#x20;   if (context.isSuperAdmin) return { scope: 'GLOBAL', ... };

&#x20;   

&#x20;   const grant = context.grants.find(g => g.permission === 'clients.view\_list');

&#x20;   if (!grant) return { scope: 'NONE', ... };

&#x20;   

&#x20;   return { scope: grant.scope, ... }; // BRANCH أو ASSIGNED

}

```



\*\*تأثير الـ Scope على الاستعلام — \[`clients.ts:446`](packages/api/routes/clients.ts):\*\*



```sql

\-- BRANCH scope (مدير الفرع, ADMIN, SALES, TELEMARKETER, TECHNICIAN):

WHERE c.branch\_id = ANY($branches)



\-- ASSIGNED scope (CUSTOMER\_SERVICE\_SUPERVISOR):

WHERE EXISTS (SELECT 1 FROM client\_assignments WHERE client\_id=c.id AND hr\_user\_id=$userId)

&#x20; AND c.branch\_id = ANY($branches)

```



\*\*الجدول الموجز:\*\*



| السؤال | الإجابة |

|--------|---------|

| هل مدير الفرع يرى الزبائن حسب الفرع؟ | \*\*نعم\*\* — scope=BRANCH يعرض كل زبائن الفرع |

| هل يرى الزبائن حسب المشرفة/الفني؟ | \*\*ليس مباشرة\*\* — لا يوجد filter رسمي حالياً حسب assignments |

| هل توجد عدادات أو فلاتر حسب الموظف؟ | \*\*لا\*\* — لا يوجد عداد لـ "زبائن المشرفة X" في الواجهة الحالية |

| هل يمكن تمييز الزبائن غير المسندين؟ | \*\*لا يوجد فلتر صريح\*\* — لكن BRANCH scope يراهم لأنهم في الفرع |

| هل يوجد مفهوم "زبائن الفرع العامة"؟ | \*\*ضمنياً فقط\*\* — زبون لا يملك أي `client\_assignments` = يُرى من مستخدمي scope=BRANCH، لكن لا يوجد حقل `is\_branch\_owned` أو ما شابه |

| هل assignments تُخفى عن ASSIGNED-scope users؟ | \*\*نعم\*\* — يُشطب `assignments: \[]` من الـ response للمستخدمين بـ ASSIGNED scope (backend defense-in-depth) |

\## E) الاستنتاج



| السؤال | الإجابة |

|--------|---------|

| \*\*1. هل يوجد تشكيل فريق فعلي في المشروع؟\*\* | \*\*لا\*\* — لا يوجد جدول `teams` مستقل. الفريق هو بنية مؤقتة JSONB في `day\_schedules` حياتها يوم واحد. |

| \*\*2. هل يوجد team\_key فقط أم جدول فريق كامل؟\*\* | \*\*team\_key فقط كنص\*\* في `marketing\_visits` و`route\_assignments`. لا توجد كيانات فريق قابلة للاستعلام المستقل. |

| \*\*3. هل يمكن تمثيل فريق فيه مشرفة وفني ومتدرب وتيلماركتر؟\*\* | \*\*نعم\*\* — `TeamSlot` يدعم هذه الأدوار الأربعة بالضبط في بنية واحدة. |

| \*\*4. هل يمكن تمثيل فريق فني فقط؟\*\* | \*\*نعم\*\* — `supervisor: null, technician: X` مسموح، وكذلك `SoloSlot` للفني المنفرد. |

| \*\*5. كيف تُحدد التبعية الحالية للزبون؟\*\* | عبر جدول `client\_assignments` (M2M). غير مسند = الزبون موجود في `clients` بلا صفوف في `client\_assignments`. |

| \*\*6. هل التبعية الحالية منفصلة عن منشئ الاسم؟\*\* | \*\*نعم تماماً\*\* — `created\_by` ثابت وتاريخي لا يعطي صلاحية. `client\_assignments` هو المصدر التشغيلي الوحيد. |

| \*\*7. هل يمكن أن تكون التبعية لمشرفة أو عدة مشرفات أو فني أو الفرع؟\*\* | \*\*جزئياً\*\* — تعدد المشرفات: نعم. الفرع (بلا إسناد): نعم ضمنياً. تبعية للفني: \*\*لا\*\* — TECHNICIAN لا يملك `clients.can\_be\_assigned`. |

| \*\*8. ما القريب من فكرتنا؟\*\* | جدول `client\_assignments` M2M هو الأقرب لفكرة "التبعية الحالية". وثنائية `created\_by` / `assignments` صحيحة معمارياً وموجودة. |

| \*\*9. ما الفجوة الأساسية قبل الانتقال إلى نطاق العمل؟\*\* | ثلاث فجوات محددة: |



\---



\### الفجوات الثلاث التفصيلية



\*\*الفجوة 1 — لا يوجد فريق دائم (persistent team):\*\*

الفريق المُشكَّل في جدولة اليوم يختفي بانتهاء اليوم. لا يوجد جدول `teams` يمكن الاستعلام عنه لاحقاً لمعرفة "من هم أعضاء الفريق الحالي؟" أو "ما الفريق الذي ينتمي إليه فني X؟". هذا يعني أنه لا يمكن ربط تبعية الزبون بفريق — فقط بأفراد.



\*\*الفجوة 2 — TECHNICIAN خارج إسناد الزبون:\*\*

`TECHNICIAN` لا يملك `clients.can\_be\_assigned`، أي لا يظهر في قائمة إسناد الزبون. إذا كان نطاق العمل سيُسند زبائن لفنيين، فهذا يحتاج منح صلاحية `clients.can\_be\_assigned` للدور TECHNICIAN، أو إعادة التفكير في آلية الإسناد الميداني.



\*\*الفجوة 3 — لا يوجد "تبعية للفرع" صريحة:\*\*

حالياً، الزبون بلا إسناد يُرى من أي مستخدم BRANCH-scope، لكنه ليس "مسنداً للفرع" بشكل صريح. لا يوجد حقل `is\_unassigned\_branch\_client = TRUE` أو منطق يقول "إذا كانت `client\_assignments` فارغة، فالزبون تابع للفرع". هذا التمييز موجود فكرياً في الكود لكنه غير معلَّم في البيانات.
