\## تقرير المرحلة الرابعة — توليد جهات الاتصال للتيلماركتر



\---



\### A. ما هي جهة الاتصال حالياً؟



يوجد \*\*كيانان\*\* يؤديان أدواراً مختلفة:



\---



\*\*الكيان الأول: `contact\_targets`\*\* — السجل الدوري الدائم



```sql

\-- Migration 045

CREATE TABLE contact\_targets (

&#x20; id                    BIGSERIAL PRIMARY KEY,

&#x20; branch\_id             INTEGER NOT NULL REFERENCES branches(id),

&#x20; target\_type           VARCHAR(50) CHECK (target\_type IN ('client')),

&#x20; target\_id             INTEGER NOT NULL,

&#x20; target\_stage          VARCHAR(50) CHECK (target\_stage IN ('lead')),

&#x20; visit\_type            VARCHAR(50) CHECK (visit\_type IN ('marketing')),

&#x20; source\_type           VARCHAR(50) CHECK (source\_type IN ('lead')),

&#x20; source\_id             INTEGER NOT NULL,

&#x20; supervisor\_hr\_user\_id INTEGER REFERENCES hr\_users(id),

&#x20; zone\_id               INTEGER,

&#x20; status                VARCHAR(50) DEFAULT 'new'

&#x20;                        CHECK (status IN ('new','queued','in\_call\_list','contacted','booked','closed','cancelled')),

&#x20; latest\_call\_outcome      VARCHAR(50),

&#x20; latest\_task\_list\_item\_id VARCHAR(100),

&#x20; latest\_appointment\_id    VARCHAR(100),

&#x20; created\_at            TIMESTAMPTZ DEFAULT NOW(),

&#x20; updated\_at            TIMESTAMPTZ DEFAULT NOW(),

&#x20; UNIQUE (branch\_id, target\_type, target\_id, visit\_type, source\_type)

);

```



\*\*الكيان الثاني: `telemarketing\_task\_list\_items`\*\* — اللقطة اليومية المؤقتة



```sql

\-- Migration 001 + 047 + 048

CREATE TABLE telemarketing\_task\_list\_items (

&#x20; id                VARCHAR(100) PRIMARY KEY,  -- {taskListId}\_client\_{entityId}

&#x20; task\_list\_id      VARCHAR(100) NOT NULL REFERENCES telemarketing\_task\_lists(id) ON DELETE CASCADE,

&#x20; entity\_type       VARCHAR(20)  CHECK (entity\_type IN ('candidate', 'client')),

&#x20; entity\_id         INTEGER NOT NULL,

&#x20; name              VARCHAR(255) NOT NULL,

&#x20; mobile            VARCHAR(50)  NOT NULL,

&#x20; contact\_number    VARCHAR(50),

&#x20; contact\_label     VARCHAR(255),

&#x20; address\_text      TEXT,

&#x20; geo\_unit\_id       INTEGER,

&#x20; status            VARCHAR(20) DEFAULT 'pending' CHECK (status IN ('pending','called','booked')),

&#x20; call\_outcome      VARCHAR(50),

&#x20; contact\_target\_id BIGINT REFERENCES contact\_targets(id) ON DELETE SET NULL

);

```



\*\*الفرق الجوهري بينهما:\*\*



| البُعد | `contact\_targets` | `task\_list\_items` |

|---|---|---|

| الطبيعة | دوري دائم (persistent) | يومي مؤقت (ephemeral) |

| يمتد عبر أيام؟ | نعم | لا — ليوم واحد فقط |

| يحمل تاريخ العمل؟ | لا — تاريخه via `task\_list` | نعم — عبر `task\_list.date` |

| يحمل team\_key؟ | لا | نعم — عبر `task\_list.team\_key` |

| يحمل `call\_outcome`؟ | `latest\_call\_outcome` فقط | نتيجة اليوم الحالي |

| يحمل منطقة؟ | نعم — `zone\_id` | نعم — `geo\_unit\_id` |

| يحمل مشرف؟ | نعم — `supervisor\_hr\_user\_id` | لا |

| مرتبط بزبون؟ | نعم — `target\_id + target\_type='client'` | نعم — `entity\_id + entity\_type='client'` |

| مرتبط بـ Candidate؟ | \*\*لا\*\* — CHECK يمنع ذلك | \*\*نعم\*\* — entity\_type IN ('candidate','client') |

| له حالة؟ | نعم — 7 حالات | نعم — 3 حالات |



\*\*ملاحظة مهمة\*\*: `contact\_targets` مقيد بـ `target\_type='client'` فقط عبر CHECK constraint. لا يمكن إنشاء contact\_target للـ Candidate. أما `task\_list\_items` فيقبل كلاهما — إذن التيلماركتر يرى Candidates في قائمتها اليومية لكن بدون سجل دوري في `contact\_targets`.

\### B. كيف تتولد جهة الاتصال؟



\*\*التوليد يدوي\*\* — يضغط مدير الفرع زر "توليد قائمة الاتصال" في `PlanningContactTargets`.



\*\*مسار `POST /telemarketing/task-lists/generate-from-plan`\*\* (تفصيلي):



\*\*الخطوة 1 — جلب العملاء المؤهلين:\*\*

تستدعي `getPlanningMarketingTargets()` التي تُعيد الـ leads ضمن النطاق الجغرافي للفريق (موضح في تقرير المرحلة الثالثة).



\*\*الخطوة 2 — فحص التكرار:\*\*

```

للعميل في قائمة اليوم:

&#x20; هل entity\_id موجود في task\_list\_items لفريق آخر اليوم؟

&#x20; → نعم: skipped (reason: 'already\_queued\_today', existingTeamKey: 'team\_X')

&#x20; → لا: متابعة

```



\*\*الخطوة 3 — إنشاء أو تحديث `contact\_target`:\*\*

```sql

INSERT INTO contact\_targets (branch\_id, target\_type, target\_id, target\_stage,

&#x20; visit\_type, source\_type, source\_id, supervisor\_hr\_user\_id, zone\_id, status)

VALUES (...)

ON CONFLICT (branch\_id, target\_type, target\_id, visit\_type, source\_type)

DO UPDATE SET supervisor\_hr\_user\_id=..., zone\_id=..., source\_id=..., updated\_at=NOW()

RETURNING id

```



\*\*الخطوة 4 — تحديث حالة `contact\_target`:\*\*

```sql

UPDATE contact\_targets

SET status = CASE WHEN status = 'new' THEN 'queued' ELSE status END,

&#x20;   updated\_at = NOW()

WHERE id = ANY($1::int\[])

```



\*\*الخطوة 5 — إدراج `task\_list\_item`:\*\*

```sql

INSERT INTO telemarketing\_task\_list\_items

&#x20; (id, task\_list\_id, entity\_type, entity\_id, name, mobile,

&#x20;  contact\_number, contact\_label, address\_text, geo\_unit\_id,

&#x20;  status, call\_outcome, contact\_target\_id)

VALUES (...)

ON CONFLICT (id) DO UPDATE SET ...  -- تحديث إن وُجد مسبقاً

```



\*\*الخطوة 6 — تحديث `latest\_task\_list\_item\_id`:\*\*

```sql

UPDATE contact\_targets

SET latest\_task\_list\_item\_id = $itemId, updated\_at = NOW()

WHERE id = $contactTargetId

```



\*\*الخطوة 7 — إعادة الإحصاء:\*\*

```json

{ "counts": { "added": N, "updated": N, "skipped": N }, "skipped": \[...] }

```



\*\*ما يُحفظ وما لا يُحفظ:\*\*



| البيان | هل يُحفظ؟ |

|---|---|

| سبب الإضافة / نوع المهمة | \*\*لا\*\* |

| مصدر النطاق (route\_assignment) | \*\*لا\*\* |

| المنطقة / zone\_id | \*\*نعم\*\* — في `contact\_targets.zone\_id` و`task\_list\_items.geo\_unit\_id` |

| الفريق / team\_key | \*\*نعم\*\* — في `task\_lists.team\_key` |

| التاريخ | \*\*نعم\*\* — في `task\_lists.date` |

| المشرف | \*\*نعم\*\* — في `contact\_targets.supervisor\_hr\_user\_id` |

| منع التكرار في نفس الفريق | \*\*نعم\*\* — UNIQUE على `(task\_list\_id, entity\_id)` ضمنياً عبر id pattern |

| منع التكرار عبر فرق مختلفة | \*\*نعم\*\* — فحص برمجي قبل الإدراج |

\### C. علاقة جهة الاتصال بالمهمة



\*\*الإجابة المباشرة: لا توجد علاقة.\*\*



`contact\_targets` لا يحتوي على أي من الحقول التالية:

\- `task\_type`

\- `reason`

\- `work\_type`

\- `task\_source`

\- `visit\_tasks`

\- `source\_task\_id` (ربط بـ tasks أو emergency\_tickets أو dues)



`telemarketing\_task\_list\_items` كذلك لا يحتوي على أي منها.



\*\*جميع القيم ثابتة ومحددة مسبقاً بـ CHECK:\*\*



```sql

target\_stage = 'lead'       -- لا يتغير

visit\_type   = 'marketing'  -- لا يتغير

source\_type  = 'lead'       -- لا يتغير

```



\*\*لا يوجد ربط مع:\*\*

\- `tasks` table

\- `dues` table

\- `emergency\_tickets` table

\- `maintenance\_requests` table

\- `marketing\_visit\_tasks` (تُنشأ لاحقاً عند إنشاء الموعد فقط)



\*\*السبب الضمني الوحيد\*\* لوجود العميل في القائمة: أنه \*\*lead في النطاق الجغرافي للفريق، مُسند للمشرف، بدون عقد وبدون زيارة سابقة\*\*. لا يوجد "سبب" مُصرَّح به.



\*\*جهة الاتصال الحالية هي: "زبون للاتصال" وليست "مهمة للاتصال".\*\*

\### D. علاقة جهة الاتصال بالنطاق والفريق



```

telemarketing\_task\_lists

&#x20; ├── id          VARCHAR(100) PK

&#x20; ├── team\_key    VARCHAR(100) NOT NULL    ← "team\_0" / "team\_1"

&#x20; ├── date        VARCHAR(50)  NOT NULL    ← "2026-05-02"

&#x20; ├── branch\_id   INTEGER                 ← الفرع (مضاف في generate-from-plan)

&#x20; └── UNIQUE(team\_key, date)



telemarketing\_task\_list\_items

&#x20; ├── task\_list\_id → task\_lists.id         ← يرث team\_key + date

&#x20; ├── geo\_unit\_id  INTEGER                 ← الحي / المنطقة

&#x20; └── contact\_target\_id → contact\_targets.id



contact\_targets

&#x20; ├── zone\_id      INTEGER                 ← geo\_unit\_id الحي

&#x20; └── supervisor\_hr\_user\_id               ← المشرف المسؤول

```



\*\*إجابات محددة:\*\*



| السؤال | الإجابة |

|---|---|

| مرتبطة بـ team\_key؟ | \*\*نعم\*\* — عبر `task\_lists.team\_key` |

| مرتبطة بتاريخ؟ | \*\*نعم\*\* — عبر `task\_lists.date` |

| مرتبطة بفرع؟ | \*\*نعم\*\* — `contact\_targets.branch\_id` و`task\_lists.branch\_id` |

| مرتبطة بمنطقة؟ | \*\*نعم\*\* — `contact\_targets.zone\_id` و`task\_list\_items.geo\_unit\_id` |

| مرتبطة بمشرف؟ | \*\*نعم\*\* — `contact\_targets.supervisor\_hr\_user\_id` |

| مرتبطة بتيلماركتر محدد؟ | \*\*لا\*\* — لا يوجد `telemarketer\_hr\_user\_id` في أي من الجداول |

| يمكن معرفة من أي route\_assignment جاءت؟ | \*\*لا\*\* — لا يُحفظ هذا الربط |

| التيلماركتر ترى فقط قائمة فريقها؟ | \*\*نعم\*\* — `snapshot()` يُفلتر بـ team\_key المرتبط باليوم |

| التيلماركتر ضمن TeamSlot مرتبطة بالقائمة؟ | \*\*لا\*\* — لا يوجد `telemarketer\_id` في القائمة. التيلماركتر تختار الفريق يدوياً من Dropdown |

\### E. شاشة عمل التيلماركتر



\*\*`TelemarketerWorkspace.tsx`\*\* — ثلاثة أعمدة:



\*\*العمود الأيمن (20%) — قائمة العملاء:\*\*

\- Dropdown لاختيار الفريق (من `currentSchedule`)

\- تبويبات تصفية: الكل / المتبقين / محجوز موعد / تم الاتصال / مرفوض

\- بحث بالاسم أو الرقم

\- كارت لكل عميل: Avatar + الاسم + نوع الكيان (client/candidate) + الحالة + نتيجة المكالمة



\*\*العمود الأوسط (55%) — تفاصيل العميل:\*\*

\- بيانات هوية: الاسم، التصنيف، المهنة، التقييم

\- العنوان الكامل مع الوحدة الجغرافية

\- قائمة أرقام الاتصال مع أزرار (اتصال / واتساب / تعيين كرقم رئيسي)

\- تبويب "الرحلة": Timeline تجمع الاقتراحات، العقود، الزيارات، الصيانة، سجلات المكالمات

\- \*\*زر "تسجيل نتيجة التواصل"\*\* — يفتح `OutcomeRecorderModal` — متاح فقط إن لم تكن محجوزاً

\- \*\*زر "جدولة زيارة التسويق"\*\* — يفتح `AppointmentSchedulerModal` — متاح فقط بعد الحجز



\*\*العمود الأيسر (25%) — ملخص الفريق:\*\*

\- KPIs: عدد الزيارات المجدولة، الاتصالات المكتملة، نسبة الحجز

\- جدول مواعيد الفريق



\*\*ما لا ترى التيلماركتر:\*\*

\- سبب الاتصال أو نوع المهمة

\- من أي نطاق عمل جاء الزبون

\- معلومات route\_assignment

\- المهام المطلوبة تنفيذها (تظهر فقط عند تثبيت الموعد وفي الحقل الافتراضي `device\_demo`)

\*\*3. `contact\_targets.latest\_call\_outcome`\*\* — آخر نتيجة عبر الأيام:

```sql

UPDATE SET latest\_call\_outcome = $outcome,
```

\*\*القيم المتاحة:\*\* 17 كود كما في `TelemarketingOutcomeCode`. راجع تقرير المرحلة السابقة للقائمة الكاملة.

\### G. الانتقال من جهة الاتصال إلى موعد



\*\*من أين يُفتح Modal الموعد؟\*\*

من زر "جدولة زيارة التسويق" في `TelemarketerWorkspace` — \*\*يظهر فقط بعد أن يكون `task\_list\_item.status = 'booked'`\*\* أو بعد تسجيل نتيجة `booked\_marketing\_appointment`.



\*\*ما الذي ينتقل من جهة الاتصال إلى الموعد؟\*\*



| الحقل | يصل إلى `telemarketing\_appointments`؟ |

|---|---|

| `entity\_type` + `entity\_id` (client\_id) | \*\*نعم\*\* |

| `team\_key` | \*\*نعم\*\* — يُختار من Dropdown في Modal |

| `task\_list\_id` | \*\*نعم\*\* — في payload |

| `task\_list\_item\_id` | \*\*نعم\*\* — في payload |

| `contact\_target\_id` | \*\*نعم\*\* — يُحفظ في `appointments.contact\_target\_id` |

| `visitTasks` | \*\*نعم\*\* — لكنها ثابتة: `\['device\_demo']` |

| سبب أصلي أو مهمة مصدرية | \*\*لا\*\* |

| route\_assignment أو zone\_id | \*\*لا\*\* |



\*\*سلسلة الجانب الأيسر عند الحفظ:\*\*

```

createAppointment()

&#x20; ├── INSERT telemarketing\_appointments (contact\_target\_id, visit\_tasks=\['device\_demo'])

&#x20; ├── UPDATE task\_list\_items SET status='booked', call\_outcome='booked\_marketing\_appointment'

&#x20; ├── UPDATE contact\_targets SET status='booked', latest\_appointment\_id=...

&#x20; └── createMarketingVisitForAppointment()

&#x20;       ├── INSERT marketing\_visits (team snapshot: supervisor/technician/trainee)

&#x20;       └── INSERT marketing\_visit\_tasks (task\_type='device\_demo', status='pending')

```



\*\*هل يمكن إنشاء أكثر من موعد لنفس جهة الاتصال؟\*\*

لا يوجد UNIQUE constraint يمنع ذلك على مستوى قاعدة البيانات. لكن الواجهة تُخفي الزر بعد الحجز الأول.

\### I. الاستنتاج النهائي



\*\*ما الكيان الحقيقي الذي يمثل جهة الاتصال للتيلماركتر؟\*\*

كيانان بأدوار متكاملة: `contact\_target` هو الذاكرة الدائمة لدورة حياة الزبون كهدف تسويقي، و`task\_list\_item` هو التمثيل اليومي الذي تتعامل معه التيلماركتر فعلياً. التيلماركتر تعمل على `task\_list\_item`، لكن نتائجها تُحدّث كلا الكيانين.



\*\*هل التوليد تلقائي أم يدوي؟\*\*

يدوي تماماً — يتطلب ضغطة زر من مدير الفرع بعد تعيين النطاق.



\*\*هل جهة الاتصال مرتبطة بالنطاق؟\*\*

بشكل غير مباشر — `zone\_id` يُحفظ، لكن لا يوجد ربط بـ `route\_assignment.key` الذي ولّد هذه القائمة.



\*\*هل جهة الاتصال مرتبطة بالفريق؟\*\*

نعم — عبر `task\_lists.team\_key`. لكن التيلماركتر الفردية لا تُعيَّن لجهة الاتصال — هي تختار فريقها يدوياً.



\*\*هل جهة الاتصال مرتبطة بزبون فقط أم بمهمة؟\*\*

بزبون فقط. لا يوجد حقل `task\_type` أو `reason` أو أي ربط بكيان مهمة.



\*\*هل يمكن معرفة سبب الاتصال حالياً؟\*\*

لا. السبب ضمني ومشتق فقط من السياق: العميل في النطاق الجغرافي وغير متعاقد. لا يُصرَّح به في أي حقل.



\*\*هل يمكن حمل أكثر من مهمة في جهة اتصال واحدة؟\*\*

لا — لا توجد بنية لذلك لا في `contact\_target` ولا في `task\_list\_item`.



\*\*هل التيلماركتر ترى المهام المطلوبة أم فقط الزبون؟\*\*

ترى الزبون فقط. المهمة (`device\_demo`) لا تظهر إلا عند لحظة تثبيت الموعد كقيمة افتراضية ثابتة.



\*\*هل نتيجة الاتصال منفصلة عن نتيجة الزيارة؟\*\*

نعم — `call\_outcome` في `task\_list\_items` و`call\_logs` منفصلة تماماً عن `marketing\_visits.outcome` التي تُسجَّل بعد الزيارة الفعلية.



\*\*هل جهة الاتصال تتحول إلى موعد بطريقة واضحة؟\*\*

نعم — المسار محدد: نتيجة `booked\_marketing\_appointment` → `status='booked'` → ينشط زر "جدولة زيارة" → `createAppointment()` ينشئ الموعد وينشئ `marketing\_visit` فوراً.



\*\*ما القريب من الرؤية المنتجية؟\*\*

\- النطاق الجغرافي يعمل (route\_assignments → zone\_ids → leads)

\- دورة حياة `contact\_target` موجودة ومتكاملة

\- ربط `contact\_target\_id` عبر الموعد موجود

\- قائمة اتصال يومية مرتبطة بالفريق والتاريخ موجودة



\*\*ما الفجوة الأساسية قبل الانتقال إلى إدارة المواعيد؟\*\*



| الفجوة | التأثير |

|---|---|

| لا يوجد `task\_type` في جهة الاتصال | التيلماركتر لا تعرف \*\*لماذا\*\* تتصل |

| `contact\_targets` مقيد بـ `lead/marketing` فقط | لا يمكن توليد جهة اتصال لمهام أخرى (صيانة، ذمة، هدية) |

| `marketing\_visit\_tasks` مقيدة بـ `device\_demo` فقط | الزيارة لا تحمل إلا مهمة واحدة ثابتة |

| لا ربط بـ `route\_assignment` في جهة الاتصال | لا يمكن تتبع من أي نطاق جاءت بعد التوليد |

| لا `telemarketer\_id` في `task\_list` | لا يمكن تعيين جهة الاتصال لتيلماركتر محددة |

| لا `attempt\_count` | لا يمكن معرفة عدد محاولات الاتصال |



\*\*الفجوة الأساسية واحدة:\*\* جهة الاتصال الحالية تقول \*\*"من"\*\* يجب الاتصال به، لكن لا تقول \*\*"لماذا"\*\* — وهذا بالضبط ما تحتاجه الرؤية المنتجية لتشغيل دورة: مهمة مفتوحة → اتصال → موعد → زيارة تحمل المهام.
