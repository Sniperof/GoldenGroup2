\## A) وجود مفهوم نطاق العمل أو خط السير



\### هل يوجد كيان واضح؟



\*\*نعم — يوجد جدول `route\_assignments` وهو الكيان الأساسي لنطاق العمل.\*\*



\### تعريف الجدول من migration 001:

```sql

CREATE TABLE route\_assignments (

&#x20; key         VARCHAR(255) PRIMARY KEY,   -- "{date}\_{teamKey}"

&#x20; routes      JSONB DEFAULT '\[]',          -- مسارات (RouteComposition\[])

&#x20; extra\_zones JSONB DEFAULT '\[]'           -- مناطق إضافية (geo\_unit\_id\[])

);

```



\### تركيبة الـ key:

```

"2026-05-02\_team\_0"   ← فريق #0 في يوم 2026-05-02

"2026-05-02\_team\_1"   ← فريق #1 في نفس اليوم

"2026-05-02\_solo\_0"   ← فني منفرد #0 في نفس اليوم

```



\*\*الجدول الموجز:\*\*



| السؤال | الإجابة |

|--------|---------|

| هل يوجد كيان خط سير؟ | \*\*نعم\*\* — جدول `route\_assignments` |

| هل مرتبط بفرع؟ | \*\*لا مباشرةً\*\* — الفرع يُستنتج من team\_key عبر day\_schedules |

| هل مرتبط بـ team\_key؟ | \*\*نعم\*\* — team\_key جزء أساسي من الـ key |

| هل مرتبط بيوم؟ | \*\*نعم\*\* — التاريخ جزء أساسي من الـ key |

| هل يحتوي مناطق ومسارات؟ | \*\*نعم\*\* — `routes\[]` و `extra\_zones\[]` |

| هل محفوظ في DB؟ | \*\*نعم\*\* — persistent، لا يُحذف بعد اليوم |

| الفرق بين route و extra\_zones؟ | `routes\[]` = مسارات مرتبة بنقاط وبداية ونهاية واتجاه. `extra\_zones\[]` = geo\_unit IDs مباشرة بلا ترتيب |

| هل النطاق جغرافي فقط؟ | \*\*لا\*\* — يُستخدم لاستخراج زبائن ومهام كما سنرى |

\## B) علاقة نطاق العمل بالفريق أو team\_key



\### كيف يُولَّد team\_key — \[`planningMarketingTargets.ts:141`](packages/api/services/planningMarketingTargets.ts)



```typescript

const keyMatch = teamKey.match(/^(team|solo)\_(\\d+)$/);

const teamIndex = Number(keyMatch\[2]);

// team\_0 → teams\[0]، team\_1 → teams\[1]، solo\_0 → solos\[0]

```



```typescript

// PlanOverview.tsx:119

const teamKey = `team\_${idx}`;      // idx = موقعه في مصفوفة teams\[]

const assignmentKey = `${date}\_${teamKey}`;

```



\*\*team\_key هو ترتيبي بحت — `team\_0` = الفريق الأول في مصفوفة `day\_schedules.teams`.\*\*



| السؤال | الإجابة |

|--------|---------|

| كيف يُولَّد team\_key؟ | من index الفريق في `day\_schedules.teams\[]` أو `.solos\[]` |

| هل ثابت بما يكفي للربط؟ | \*\*خطر\*\* — إذا تغير ترتيب الفرق في اليوم، يتغير team\_key ويختل الربط |

| هل route\_assignments يستخدمه؟ | \*\*نعم\*\* — في الـ key مباشرةً |

| هل marketing\_visits يستخدمه؟ | \*\*نعم\*\* — حقل `team\_key VARCHAR(100)` مباشر |

| هل telemarketing يستخدمه؟ | \*\*نعم\*\* — `telemarketing\_task\_lists.team\_key` + UNIQUE(team\_key, date) |

| هل يمكن ربط نطاق بفريق مشرفة+فني+تيلماركتر؟ | \*\*نعم\*\* — `TeamSlot` كامل |

| هل يمكن ربط نطاق بفني منفرد (SoloSlot)؟ | \*\*تقنياً نعم\*\* — لكن النظام يُعيد `SOLO\_HAS\_NO\_MARKETING\_LOAD` ولا يعالج نطاق عمل الفني المنفرد |



\*\*team\_key هو الرابط الفعلي الوحيد بين الفريق، النطاق، المهام اليومية، والزيارات — لكن الربط جزئي:\*\*



| الرابط | موجود؟ |

|--------|--------|

| team\_key → route\_assignments | ✓ كامل |

| team\_key → marketing\_visits | ✓ كامل |

| team\_key → telemarketing\_task\_lists | ✓ كامل |

| team\_key → contact\_targets | ✗ غير مباشر |

| team\_key → client\_assignments | ✗ لا يوجد ربط مباشر |

\## C) تحديد النطاق من قبل مدير الفرع



\### الشاشات الموجودة



يوجد \*\*مسار كامل من ثلاث شاشات\*\* في قسم التخطيط:



\*\*شاشة 1 — جدولة الفرق (TeamScheduler):\*\*

\- يُنشئ `day\_schedules` للتاريخ المحدد

\- يختار من قائمة الموظفين النشطين (المؤهلين للجدولة)

\- يُحدد من هو المشرف، الفني، التيلماركتر، المتدرب لكل فريق



\*\*شاشة 2 — تعيين المسارات (RouteAssigner) — \[`RouteAssigner.tsx`](packages/web/src/pages/planning/RouteAssigner.tsx):\*\*



مدير الفرع يختار:

1\. ✓ التاريخ

2\. ✓ الفريق (team\_key مشتق من day\_schedules)

3\. ✓ مسارات مع start/end/direction

4\. ✓ extra\_zones (مناطق جغرافية مستقلة — level 4 أو leaf nodes فقط)

5\. يرى عدادات المهام بعد الحفظ (reload بعد save)

6\. يمكن تعديل النطاق بعد حفظه (PUT على نفس الـ key)



\*\*شاشة 3 — ملخص الخطة (PlanOverview) — \[`PlanOverview.tsx`](packages/web/src/pages/planning/PlanOverview.tsx):\*\*



يعرض لكل فريق:

\- ✓ أسماء الطاقم (المشرف + الفني)

\- ✓ المسار المخصص مع start → end ومحطة واتجاه

\- ✓ عدد المناطق الإضافية (extra\_zones)

\- ✓ عدد المهام (`X مهمة`) و عدد الـ Leads

\- ✓ زر "أهداف الاتصال" للانتقال لتفاصيل الزبائن

\- يتنقل بين التواريخ (أمس / غد)



\*\*ما لا يستطيع مدير الفرع فعله حالياً:\*\*

\- لا يختار زبائن مباشرة عند بناء النطاق

\- لا يختار مهام مباشرة

\- لا يرى عدادات قبل الحفظ (العدادات تأتي بعد الحفظ)

\- لا يختار فرق على مستوى الحي/الناحية في الـ routes إلا عبر route\_points

\## D) علاقة النطاق بالزبائن



\### Query التشغيلي الأساسي — \[`planningMarketingTargets.ts:217`](packages/api/services/planningMarketingTargets.ts)



النطاق يُرجع زبائن تستوفي \*\*كل\*\* هذه الشروط:



```sql

WHERE c.is\_candidate = FALSE

&#x20; AND c.branch\_id = $branchId

&#x20; AND c.neighborhood::int = ANY($zoneIds)     -- عنوان الزبون ضمن النطاق

&#x20; AND EXISTS (                                 -- مسند للمشرف تحديداً

&#x20;     SELECT 1 FROM client\_assignments ca

&#x20;     WHERE ca.client\_id = c.id

&#x20;       AND ca.hr\_user\_id = $supervisorHrUserId

&#x20; )

&#x20; AND NOT EXISTS (SELECT 1 FROM contracts WHERE customer\_id = c.id)  -- بدون عقد

&#x20; AND NOT EXISTS (SELECT 1 FROM visits WHERE v.customer\_id = c.id)   -- بدون زيارة

```



| السؤال | الإجابة |

|--------|---------|

| هل يمكن استخراج زبائن ضمن النطاق؟ | \*\*نعم\*\* — بقيود محددة |

| هل النطاق يعتمد على عنوان الزبون؟ | \*\*نعم\*\* — `c.neighborhood::int = ANY(zoneIds)` |

| هل يعتمد على فرع الزبون؟ | \*\*نعم\*\* — `c.branch\_id = $branchId` |

| هل يعتمد على التبعية الحالية؟ | \*\*نعم — لكن للمشرف فقط\*\* `client\_assignments.hr\_user\_id = supervisorHrUserId` |

| هل يمكن معرفة زبائن فريق معين؟ | \*\*نعم\*\* — عبر API `/api/planning/marketing-targets` |

| هل يمكن التمييز بين تبعية مشرفة، فني، فرع؟ | \*\*لا\*\* — الاستعلام الحالي يبحث عن مسند للمشرف فقط |

| هل يوجد مفهوم "زبائن الفرع غير المسندين"؟ | \*\*لا\*\* — الفلتر يشترط وجود إسناد للمشرف صراحةً |

| هل يوجد عداد زبائن فريدين حسب نطاق؟ | \*\*نعم\*\* — `countsByZone` + `counts.total` في الـ response |



\*\*تحذير معماري مهم:\*\*

الاستعلام يستخدم `c.neighborhood::int` — أي يفترض أن حقل `neighborhood` في جدول `clients` مخزون كـ \*\*ID رقمي لـ geo\_unit\*\*. إذا كان مخزوناً كنص حر، يُعاد `0 leads`. هذا خطر صامت.

\## E) علاقة النطاق بالمهام المفتوحة



\*\*ما الذي يُمثل "المهام" حالياً؟\*\*



المهمة الحالية الوحيدة المعرّفة ضمن النطاق هي: \*\*زبون بدون عقد وبدون زيارة، مسند للمشرف، عنوانه داخل زون النطاق\*\* = Lead قابل للزيارة التسويقية.



لا يوجد حالياً:

\- خدمة تُرجع "مهام مفتوحة" حسب نطاق (صيانة، ذمة، هدية، كرت كفالة...)

\- API يحسب أنواع مهام مختلفة حسب route

\- كيان "task" مرتبط بـ route\_assignment

\- جدول `tasks` أو `open\_tasks` مستقل



\*\*الموجود حالياً:\*\*



| الكيان | علاقته بالنطاق |

|--------|---------------|

| `contact\_targets` | يملك `zone\_id` + `supervisor\_hr\_user\_id` — لكن لا يوجد query يجمع route\_assignment مع contact\_targets مباشرةً |

| `telemarketing\_task\_lists` | مرتبط بـ team\_key + date — لكن يُنشأ للتنفيذ التشغيلي، ليس للتخطيط |

| `marketing\_visits` | مرتبط بـ team\_key — لكن ينشأ بعد تأكيد الموعد |

| `leads` في planning | الاستعلام الرئيسي: زبائن بدون عقد/زيارة ضمن النطاق |



لا يوجد API يحسب targets أو contact\_targets بناءً على route مباشرةً — `contact\_targets.zone\_id` موجود كحقل لكن لا يوجد JOIN مع `route\_assignments` في أي query حالي.

\## F) علاقة النطاق بالتيلماركتر وجهات الاتصال



\### سلسلة الربط الموجودة:



```

route\_assignments \[key = "date\_team\_X"]

&#x20;       ↓ (يحسب leads من زيارة planning)

getPlanningMarketingTargets

&#x20;       ↓ يُظهر قائمة leads في PlanningContactTargets

المستخدم يُضيف lead يدوياً إلى task\_list

&#x20;       ↓

telemarketing\_task\_lists \[team\_key, date, branch\_id]

&#x20;  └── telemarketing\_task\_list\_items \[entity\_id = client\_id]

&#x20;       ↓ (التيلماركتر تتصل وتوثق النتيجة)

telemarketing\_appointments \[team\_key, date, entity\_id]

&#x20;       ↓ (يُنشئ زيارة)

marketing\_visits \[team\_key, contact\_target\_id, client\_id]

```



| السؤال | الإجابة |

|--------|---------|

| هل route\_assignments أو team\_key يُستخدم لتوليد contact\_targets؟ | \*\*لا تلقائياً\*\* — المدير يرى القائمة ويُضيف يدوياً |

| هل التيلماركتر ترى جهات مرتبطة بـ team\_key؟ | \*\*نعم\*\* — `task\_list\_items` مرتبطة بـ `task\_list.team\_key` |

| هل جهة الاتصال تعرف أنها جاءت من نطاق معين؟ | \*\*جزئياً\*\* — `contact\_targets.zone\_id` موجود لكن الربط بالـ route ليس صريحاً |

| هل يوجد ربط بين route\_assignment و contact\_target مباشرة؟ | \*\*لا\*\* — contact\_targets يعرف zone\_id لكن لا يعرف route\_assignment.key |

| هل يوجد قائمة عمل للتيلماركتر مبنية على نطاق الفريق؟ | \*\*نعم بشكل غير مباشر\*\* — `PlanningContactTargets` تعرض leads → المدير يختار من يدخل للـ task\_list |

\### الفجوات الأربع التفصيلية



\*\*الفجوة 1 — team\_key هش (positional):\*\*

team\_key مشتق من ترتيب الفريق في مصفوفة `day\_schedules.teams\[]`. إذا حُذف فريق من منتصف القائمة أو أُعيد ترتيبها، تُكسر العلاقات في `route\_assignments`، `marketing\_visits`، و`telemarketing\_task\_lists` لنفس اليوم. لا يوجد UUID أو معرّف ثابت للفريق.



\*\*الفجوة 2 — الفني المنفرد (SoloSlot) بلا نطاق:\*\*

كل solo\_X يُعيد `SOLO\_HAS\_NO\_MARKETING\_LOAD` ولا يُحسب له أي زبائن. هذا يعني أن الفنيين العاملين منفردين خارج منظومة النطاق والتخطيط كلياً.



\*\*الفجوة 3 — المهام المفتوحة محصورة في نوع واحد:\*\*

النطاق الحالي يُحسب فقط "leads بلا عقد وبلا زيارة" = مهمة التسويق للجهاز الأول. لا يوجد حساب للمهام الأخرى كالصيانة، الذمم، نقل الجهاز، كرت الكفالة... هذه المهام لا يوجد لها كيان مربوط بالنطاق حالياً.



\*\*الفجوة 4 — استعلام الزبائن مقيّد بالمشرف فقط:\*\*

`getPlanningMarketingTargets` يبحث عن زبائن مسندين لـ `supervisorHrUserId`. زبائن الفرع العامة (بلا إسناد لأحد) وزبائن مسندة لتيلماركتر أو SALES لا تظهر في نطاق الفريق حتى لو كانت في نفس المنطقة الجغرافية.
