\### A. كيان الموعد الحالي



جدول `telemarketing\_appointments` تشكّل عبر أربع مراحل من migrations:



```

Migration 001 — الهيكل الأساسي

Migration 014 — إضافة branch\_id

Migration 047 — إضافة contact\_target\_id

Migration 050 — إضافة visit\_tasks + requested\_device\_\*

```



\*\*الشكل النهائي الكامل للحقول:\*\*



| الحقل | النوع | الوصف |

|---|---|---|

| `id` | VARCHAR(100) PK | معرف الموعد |

| `entity\_type` | VARCHAR(20) | CHECK IN ('candidate', 'client') |

| `entity\_id` | INTEGER | معرف الزبون أو المرشح |

| `customer\_name` | VARCHAR(255) | اسم منسوخ من جهة الاتصال |

| `customer\_address` | TEXT | عنوان منسوخ |

| `customer\_mobile` | VARCHAR(50) | رقم منسوخ |

| `team\_key` | VARCHAR(100) | معرف الفريق |

| `date` | VARCHAR(50) | تاريخ الزيارة YYYY-MM-DD |

| `time\_slot` | VARCHAR(50) | وقت الزيارة HH:MM |

| `occupation` | VARCHAR(255) | مهنة الزبون |

| `water\_source` | VARCHAR(255) | مصدر المياه |

| `notes` | TEXT | ملاحظات فنية |

| `visit\_tasks` | JSONB | افتراضي: `\["device\_demo"]` |

| `requested\_device\_model\_id` | INTEGER | FK إلى device\_models |

| `requested\_device\_name` | TEXT | اسم الجهاز |

| `branch\_id` | INTEGER | FK إلى branches |

| `contact\_target\_id` | BIGINT | FK إلى contact\_targets (SET NULL) |

| `created\_at` | TIMESTAMPTZ | وقت الإنشاء |

| `created\_by` | INTEGER | FK إلى hr\_users |



\*\*إجابات محددة:\*\*



| السؤال | الإجابة |

|---|---|

| الموعد مرتبط بزبون؟ | \*\*نعم\*\* — `entity\_id + entity\_type='client'` |

| الموعد مرتبط بـ Candidate؟ | \*\*نعم\*\* — entity\_type يقبل 'candidate' أيضاً |

| الموعد مرتبط بجهة اتصال؟ | \*\*نعم\*\* — `contact\_target\_id` (SET NULL إن حُذف) |

| الموعد مرتبط بـ task\_list\_item؟ | \*\*لا\*\* — لا يوجد حقل `task\_list\_item\_id` في جدول appointments |

| الموعد مرتبط بـ contact\_target؟ | \*\*نعم\*\* — `contact\_target\_id` |

| الموعد مرتبط بـ team\_key؟ | \*\*نعم\*\* — `team\_key` NOT NULL |

| الموعد مرتبط بفرع؟ | \*\*نعم\*\* — `branch\_id` |

| الموعد يحمل visitTasks؟ | \*\*نعم\*\* — `visit\_tasks JSONB` افتراضي `\['device\_demo']` |

| الموعد يحمل سبب/مهمة أصلية؟ | \*\*لا\*\* — لا يوجد `task\_type` أو `source\_task\_id` |

| الموعد له حالة مستقلة؟ | \*\*لا\*\* — لا يوجد حقل `status` في `telemarketing\_appointments` |

| الموعد له تاريخ ووقت؟ | \*\*نعم\*\* — `date + time\_slot` معاً |

| يمكن تعديله أو إلغاؤه؟ | \*\*لا\*\* — لا توجد routes للتعديل أو الإلغاء |

\### B. إنشاء الموعد من شاشة التيلماركتر



\*\*متى يظهر زر جدولة الموعد؟\*\*



```typescript

// TelemarketerWorkspace.tsx

<button

&#x20; disabled={!isBookedForSelected || !!selectedTaskAppointment}

>

&#x20; جدولة زيارة التسويق

</button>

```



يظهر الزر فاعلاً فقط حين:

\- `task\_list\_item.status === 'booked'` أو `callOutcome === 'booked\_marketing\_appointment'`

\- \*\*وليس\*\* لديه appointment مسجل بالفعل



يعني: \*\*يجب تسجيل نتيجة اتصال `booked\_marketing\_appointment` أولاً\*\* — الزر لا يُفعَّل قبل ذلك.



\*\*ما الذي تدخله التيلماركتر في `AppointmentSchedulerModal`؟\*\*



| الحقل | إلزامي؟ | الخيارات / الوصف |

|---|---|---|

| وقت الزيارة (time\_slot) | \*\*نعم\*\* | `<input type="time">` ضمن ساعات العمل |

| مهام الزيارة (visitTasks) | \*\*نعم\*\* | Checkbox — خيار واحد فقط: `device\_demo` |

| مصدر المياه | \*\*نعم\*\* | Dropdown: اسالة / قناني / بئر / تناكر / غير معروف |

| الجهاز المطلوب | لا | Dropdown من device\_models أو إدخال نصي |

| ملاحظات فنية | لا | Textarea |



\*\*ما لا تختاره التيلماركتر:\*\*

\- التاريخ — يأتي من اليوم الحالي

\- الفريق — يأتي من الفريق المحدد في الشاشة

\- الزبون — يأتي من `task\_list\_item` الحالي



\*\*هل يمكنها إضافة أكثر من مهمة؟\*\* لا — هناك checkbox واحد فقط.



\*\*هل يمكنها جدولة موعد لزبون خارج قائمة الاتصال؟\*\* لا — المسار يشترط وجود `taskListId` و`taskListItemId` كشرط صريح في الـ API:

```typescript

if (!appointment.taskListId || !appointment.taskListItemId) {

&#x20;   return res.status(400).json({ error: 'taskListId and taskListItemId are required' });

}

```

\### C. قواعد التعارض والتحقق



\*\*conflict check موجود\*\* — مزدوج (client-side + server-side):



\*\*Client-side في `useTelemarketingStore`:\*\*

```typescript

const isBooked = get().appointments.some(

&#x20;   (apt) =>

&#x20;       apt.teamKey === input.teamKey \&\&

&#x20;       apt.date === input.date \&\&

&#x20;       apt.timeSlot === input.timeSlot,

);

if (isBooked) throw new Error('هذا الموعد محجوز مسبقاً للفريق في نفس الوقت.');

```



\*\*Server-side في API handler:\*\*

```sql

SELECT id FROM telemarketing\_appointments

WHERE team\_key = $1

&#x20; AND date = $2

&#x20; AND time\_slot = $3

&#x20; AND branch\_id = $4

LIMIT 1

```

→ إن وُجد: `409 Conflict` مع رسالة "هذا الموعد محجوز مسبقاً للفريق في نفس الوقت."



\*\*جدول إجابات التعارض:\*\*



| النوع | هل يُمنع؟ |

|---|---|

| نفس team\_key + date + time\_slot | \*\*نعم — ممنوع\*\* |

| نفس الزبون في نفس اليوم بأوقات مختلفة | \*\*لا — مسموح\*\* |

| نفس task\_list\_item له موعدان | \*\*لا\*\* — لا يوجد UNIQUE عليه |

| نفس contact\_target له موعدان | \*\*لا\*\* — لا يوجد UNIQUE عليه |

| فريقان لهما موعد في نفس الوقت | \*\*لا\*\* — الفحص يختص بالفريق الواحد |

| أوقات مختلفة لنفس الفريق في نفس اليوم | \*\*مسموح\*\* — الفحص على الـ time\_slot فقط |



\*\*الخلاصة:\*\* التعارض محصور على مستوى `team\_key + date + time\_slot`. يمكن لنفس الفريق جدولة زيارات متعددة في نفس اليوم بأوقات مختلفة — وهذا منطقي ومقصود.

\### D. علاقة الموعد بالفريق وخط السير



| الحقل | محفوظ في appointment؟ | محفوظ في marketing\_visit؟ |

|---|---|---|

| `team\_key` | \*\*نعم\*\* | \*\*نعم\*\* |

| `date` | \*\*نعم\*\* (`date`) | \*\*نعم\*\* (`scheduled\_date`) |

| `branch\_id` | \*\*نعم\*\* | \*\*نعم\*\* |

| `route\_assignment` | \*\*لا\*\* | \*\*لا\*\* |

| `zone\_id` | \*\*لا\*\* | \*\*لا\*\* |

| `contact\_target\_id` | \*\*نعم\*\* | \*\*نعم\*\* |

| `task\_list\_id` | \*\*لا\*\* (فقط في payload عند الإنشاء) | \*\*نعم\*\* |

| `task\_list\_item\_id` | \*\*لا\*\* (فقط في payload) | \*\*نعم\*\* |



\*\*من أين يأتي snapshot الفريق؟\*\*

يُؤخذ عند إنشاء `marketing\_visit` (وليس عند إنشاء appointment) — تُقرأ `day\_schedules` بـ `date + team\_key` لاستخراج `supervisor/technician/trainee` وتُحفظ في `team\_snapshot JSONB`.



\*\*ماذا يحدث إذا تغير الفريق بعد تثبيت الموعد؟\*\*

الـ `appointment` مرتبط بـ `team\_key` نصياً — لا توجد FK لـ `day\_schedules`. إذا تغير تركيب الفريق في `day\_schedules` لاحقاً فإن:

\- `appointment.team\_key` يبقى كما هو (لا يتأثر)

\- `marketing\_visit.team\_snapshot` يبقى كما هو (snapshot ثابت عند الإنشاء)

\- `marketing\_visit.supervisor/technician/trainee` تبقى القيم الأصلية



الموعد محمي من تغييرات `day\_schedule` بفضل آلية الـ snapshot.

\### E. من الموعد إلى زيارة التسويق



\*\*نعم — `marketing\_visit` يُنشأ تلقائياً\*\* عند نجاح `createAppointment` ضمن نفس transaction.



\*\*شرط الإنشاء:\*\*

\- `entityType === 'client'` ← \*\*إذا كان entity\_type = 'candidate' فلا تُنشأ زيارة\*\*

\- `branchId` غير null



\*\*ما ينتقل من appointment إلى marketing\_visit:\*\*



| الحقل في marketing\_visit | المصدر |

|---|---|

| `id` | `'mv\_' + appointmentId` |

| `branch\_id` | من params |

| `client\_id` | `entityId` من task\_list\_item |

| `visit\_type` | ثابت: `'marketing'` |

| `status` | ثابت: `'scheduled'` |

| `scheduled\_date` | `appointment.date` |

| `scheduled\_time` | `appointment.timeSlot` |

| `source\_type` | ثابت: `'telemarketing\_appointment'` |

| `source\_id` | `appointmentId` |

| `contact\_target\_id` | من params |

| `task\_list\_id` | `taskListId` من payload |

| `task\_list\_item\_id` | `taskListItemId` من payload |

| `team\_key` | `appointment.teamKey` |

| `requested\_device\_model\_id` | `appointment.requestedDeviceModelId` |

| `water\_source` | `appointment.waterSource` |

| `technician\_notes` | `appointment.notes` |

| `customer\_name/address/mobile` | من appointment (منسوخة) |

| `supervisor\_employee\_id` | من day\_schedules snapshot |

| `technician\_employee\_id` | من day\_schedules snapshot |

| `trainee\_employee\_id` | من day\_schedules snapshot |

| `team\_snapshot` | JSON كامل للفريق |

| `created\_by` | المستخدم الحالي |



\*\*ملاحظة مهمة:\*\* `task\_list\_id` و`task\_list\_item\_id` \*\*يُحفظان في marketing\_visit لكن ليس في telemarketing\_appointments\*\*. الـ appointment نفسه لا يعرف من أي task\_list\_item جاء — هذه المعلومة تبقى فقط في marketing\_visit.



\*\*لا يُحفظ:\*\*

\- معرف التيلماركتر التي ثبتت الموعد (الحقل `created\_by` يشير إلى من استدعى الـ API)

\- ربط بـ route\_assignment أو zone\_id



\*\*UNIQUE constraint على marketing\_visits:\*\*

```sql

CONSTRAINT uq\_marketing\_visits\_source UNIQUE (source\_type, source\_id)

```

يمنع إنشاء أكثر من زيارة لنفس الموعد — وهو الضامن الفعلي لعلاقة 1:1 بين appointment وmarketing\_visit.

\### F. المهام داخل الموعد والزيارة



\*\*appointment.visit\_tasks:\*\*

\- نوعه: JSONB مصفوفة

\- قيمته الافتراضية: `\["device\_demo"]`

\- لا يوجد CHECK عليه في هذا الجدول — أي قيمة JSONB مقبولة نظرياً



\*\*marketing\_visit\_tasks:\*\*

```sql

task\_type VARCHAR(50) DEFAULT 'device\_demo'

CHECK (task\_type IN ('device\_demo'))

UNIQUE (visit\_id, task\_type)

```



\*\*ماذا يحدث عند إنشاء marketing\_visit\_tasks؟\*\*

يُنشأ \*\*record واحد ثابت\*\* بـ `task\_type='device\_demo'` بغض النظر عن محتوى `visit\_tasks`:

```sql

INSERT INTO marketing\_visit\_tasks (id, visit\_id, task\_type, status)

VALUES ($visitId + '\_device\_demo', $visitId, 'device\_demo', 'pending')

ON CONFLICT (visit\_id, task\_type) DO NOTHING

```



\*\*المفارقة الجوهرية:\*\*

\- `appointment.visit\_tasks` هي JSONB تقبل أي مصفوفة نظرياً

\- لكن `createMarketingVisitForAppointment` \*\*تتجاهلها\*\* وتُنشئ `device\_demo` فقط

\- الـ CHECK على `marketing\_visit\_tasks` يمنع أي نوع آخر من الوصول للقاعدة



\*\*جدول إجابات المهام:\*\*



| السؤال | الإجابة |

|---|---|

| appointment يحمل أكثر من مهمة؟ | نعم في JSONB — لكن يُتجاهل عملياً |

| الواجهة تعرض أكثر من خيار؟ | \*\*لا\*\* — خيار واحد فقط في VISIT\_TASK\_OPTIONS |

| DB تسمح بأكثر من task\_type؟ | \*\*لا\*\* — CHECK يحصره بـ device\_demo |

| كل مهمة لها status مستقل؟ | نعم — لكن هناك مهمة واحدة فقط |

| كل مهمة لها result مستقل؟ | نعم — `result` + amounts + `closed\_by\_employee\_id` |

| UNIQUE يمنع تكرار المهمة؟ | نعم — `UNIQUE (visit\_id, task\_type)` |

| يمكن ربط مهمة بمهمة مفتوحة أصلية؟ | \*\*لا\*\* — لا يوجد `source\_task\_id` |



\*\*نتائج `marketing\_visit\_tasks` المتاحة:\*\*

\- `cash\_offer\_closed` — تطلب: cashOfferAmount + closedByEmployeeId

\- `installment\_offer\_closed` — تطلب: installmentAmount + months + closedByEmployeeId

\- `cash\_offer\_not\_closed` — تطلب: cashOfferAmount

\- `installment\_offer\_not\_closed` — تطلب: installmentAmount + months

\- `demo\_not\_completed` — تطلب: resultNotes

\### G. حالة الموعد وحالة الزيارة



\*\*خريطة الحالات الكاملة:\*\*



| الكيان | الحالات المتاحة |

|---|---|

| `task\_list\_item.status` | pending → called → \*\*booked\*\* |

| `contact\_target.status` | new → queued → in\_call\_list → contacted → \*\*booked\*\* → closed / cancelled |

| `telemarketing\_appointments` | \*\*لا يوجد حقل status\*\* |

| `marketing\_visit.status` | \*\*scheduled\*\* → completed / not\_completed / postponed\_by\_company / postponed\_by\_customer / cancelled / needs\_reschedule |

| `marketing\_visit\_tasks.status` | \*\*pending\*\* → completed / not\_completed |



\*\*متى تتحول الكيانات إلى "محجوز"؟\*\*



كلها تتحول في نفس transaction عند `createAppointment`:

```

task\_list\_item.status      = 'booked'

task\_list\_item.call\_outcome= 'booked\_marketing\_appointment'

contact\_target.status      = 'booked'

contact\_target.latest\_appointment\_id = appointmentId

marketing\_visit.status     = 'scheduled'  ← يُنشأ مباشرةً

```



\*\*هل appointment له status خاص؟\*\*

\*\*لا.\*\* لا يوجد حقل status في `telemarketing\_appointments`. حالة الموعد تُستنتج من `contact\_target.status` أو `marketing\_visit.status`.



\*\*ماذا يحدث إذا أُجّل الموعد أو أُلغيت الزيارة؟\*\*



عند تحديث `marketing\_visit.status` إلى `postponed\_\*` أو `cancelled`:

\- تُعاد قيمة `marketing\_visit\_tasks` إلى `not\_completed`

\- لا يتحدث `contact\_target.status` تلقائياً

\- لا يتحدث `task\_list\_item.status` تلقائياً

\- `telemarketing\_appointments` لا يتغير



\*\*الخلاصة: لا توجد مزامنة عكسية.\*\* التحديث يسير في اتجاه واحد فقط: appointment → marketing\_visit. أي تغيير في marketing\_visit لا يعكس نفسه على appointment أو contact\_target.

\### H. تعديل الموعد أو إلغاؤه أو تأجيله



\*\*الحالة الحالية: لا يوجد أي شيء.\*\*



| العملية | متاحة؟ |

|---|---|

| تعديل تاريخ/وقت الموعد | \*\*لا\*\* |

| تغيير الفريق | \*\*لا\*\* |

| إلغاء الموعد | \*\*لا\*\* |

| تأجيل الزيارة | \*\*جزئياً\*\* — يمكن تعديل `marketing\_visit.status` إلى `postponed\_\*` |

| حذف الموعد | \*\*لا\*\* |

| audit/سجل تغييرات | \*\*لا\*\* |

\### I. ما الذي تراه الأطراف المختلفة



\*\*التيلماركتر (`TelemarketerWorkspace`):\*\*

\- العمود الأيسر: جدول مواعيد الفريق (`TeamAgendaPanel`) — يعرض مواعيد اليوم مصنفة بـ time\_slot

\- كل task\_list\_item محجوز: يظهر عليه badge "تم الحجز" + وقت الزيارة

\- تُرى فقط مواعيد فريقها في اليوم الحالي (مصفاة بـ team\_key + date)



\*\*مدير الفرع (`MarketingOperationsContent`):\*\*

\- يرى `contact\_targets` مع `latestAppointment` (date + timeSlot)

\- لا يستطيع تعديل أو إلغاء المواعيد من هذه الشاشة

\- يستطيع تصفية الأهداف حسب الحالة (booked / contacted / new ...)



\*\*الفريق الميداني:\*\*

\- يرى `marketing\_visits` المُجدولة — `status='scheduled'`

\- يدخل نتائج الزيارة عبر `MarketingVisitResultModal`

\- يرى: الزبون، الوقت، الجهاز المطلوب، ملاحظات التيلماركتر، مهمة device\_demo



\*\*عرض مركزي لجدول الزيارات؟\*\*

يُمكن الاستعلام عن `marketing\_visits` بـ `branch\_id + scheduled\_date + team\_key` — توجد indexes على كل هذه الحقول. لكن لا توجد صفحة تعرض جدولاً يومياً شاملاً لكل الفرق.

\### K. الاستنتاج النهائي



\*\*1. هل يوجد كيان موعد واضح؟\*\*

نعم — `telemarketing\_appointments` كيان متكامل برابط إلى contact\_target والفريق والفرع.



\*\*2. هل الموعد مرتبط بجهة الاتصال؟\*\*

نعم — `contact\_target\_id` مباشرة. لكن الرابط بـ `task\_list\_item` غير موجود في الجدول — موجود فقط في `marketing\_visit`.



\*\*3. هل الموعد مرتبط بالفريق؟\*\*

نعم — `team\_key` NOT NULL. مع snapshot ثابت للفريق في `marketing\_visit`.



\*\*4. هل الموعد يعرف سبب الاتصال؟\*\*

لا — لا يوجد `task\_type` أو `source\_task\_id` أو `reason`. الموعد يعرف الزبون والفريق والوقت، لكن لا يعرف لماذا تم الاتصال.



\*\*5. هل الموعد يحمل مهام متعددة؟\*\*

نظرياً الـ `visit\_tasks JSONB` تقبل مصفوفة — لكن عملياً هي دائماً `\['device\_demo']` ومقيدة بـ CHECK في قاعدة البيانات.



\*\*6. هل تُنشأ زيارة تلقائياً من الموعد؟\*\*

نعم — في نفس transaction، فوراً. لكن فقط إذا كان entity\_type = 'client'. المرشح Candidate لا يُنشئ زيارة.



\*\*7. هل الزيارة تحفظ رابطها مع الموعد وجهة الاتصال؟\*\*

نعم — `source\_type + source\_id + contact\_target\_id + task\_list\_id + task\_list\_item\_id`. الزيارة هي الكيان الأغنى معلوماتياً في السلسلة.



\*\*8. هل الحالات متناسقة؟\*\*

متناسقة عند الإنشاء. لكن \*\*لا توجد مزامنة عكسية\*\* — تغيير `marketing\_visit.status` لا يعكس نفسه على appointment أو contact\_target. الاتجاه واحد فقط.



\*\*9. هل يمكن تعديل أو إلغاء أو تأجيل الموعد؟\*\*

لا يمكن تعديل أو إلغاء `telemarketing\_appointments`. يمكن تعديل `marketing\_visit.status` إلى `postponed` أو `cancelled` لكن دون أثر راجع.



\*\*10. هل يمكن حالياً تحويل مهمة مفتوحة إلى موعد ثم زيارة؟\*\*

لا — لا يوجد كيان "مهمة مفتوحة" في أي مكان من السلسلة.



\*\*11. ما القريب من الرؤية المنتجية؟\*\*

\- بنية السلسلة موجودة: contact\_target → appointment → marketing\_visit

\- الربط بين الكيانات عبر `contact\_target\_id` و`source\_id` موجود ومتكامل

\- UNIQUE على (source\_type, source\_id) يضمن 1:1 بين appointment وزيارة

\- snapshot الفريق محمي من التغييرات اللاحقة

\- `marketing\_visit.status` يدعم دورة حياة كاملة بما فيها التأجيل



\*\*12. ما الفجوة الأساسية قبل تحليل نتائج الزيارة؟\*\*



| الفجوة | الأثر على الرؤية المنتجية |

|---|---|

| لا يوجد `task\_type` في appointment أو contact\_target | الزيارة لا تعرف لماذا وُجدت |

| `marketing\_visit\_tasks` محصورة بـ `device\_demo` | لا يمكن حمل مهام متعددة أو من أنواع أخرى |

| Candidate لا يُنشئ `marketing\_visit` | إذا تقدم Candidate جديد فلا زيارة له |

| لا يوجد UPDATE أو DELETE لـ appointment | لا يمكن تعديل الموعد بعد إنشائه |

| لا مزامنة عكسية من marketing\_visit إلى contact\_target | إلغاء الزيارة لا يُعيد فتح جهة الاتصال |

| لا يوجد `telemarketer\_id` في القائمة أو الموعد | لا يمكن تتبع من ثبت الموعد بدقة |

| `task\_list\_id/item\_id` غائبان عن appointment | تضيع نقطة الربط لو احتجنا ترابطاً مباشراً |



\*\*الفجوة الجوهرية الواحدة:\*\* الزيارة التسويقية تعرف \*\*من\*\* تزوره و\*\*متى\*\* و\*\*بأي فريق\*\* — لكن لا تعرف \*\*لماذا\*\* ولا \*\*ما المهام المطلوبة\*\*. هذا هو المانع الأساسي قبل أن تتحول الزيارة من "زيارة تسويقية لعرض جهاز" إلى "زيارة تحمل مهام متعددة ومتنوعة".
