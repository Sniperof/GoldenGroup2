\## A) هل يوجد جدول مهام عام؟



\*\*نعم — يوجد جدول `tasks` في قاعدة البيانات، لكنه جدول قديم وغير مكتمل الربط.\*\*



\### تعريفه من \[`migrations/001\_core\_tables.sql:127`](migrations/001\_core\_tables.sql):

```sql

CREATE TABLE IF NOT EXISTS tasks (

&#x20; id            SERIAL PRIMARY KEY,

&#x20; type          VARCHAR(50) NOT NULL

&#x20;                 CHECK (type IN ('emergency', 'dues', 'periodic', 'returns', 'followup')),

&#x20; customer\_name VARCHAR(255) NOT NULL,   -- نص حر، لا FK لجدول clients

&#x20; context       TEXT,

&#x20; location      VARCHAR(255),

&#x20; due\_date      DATE,

&#x20; status        VARCHAR(50) DEFAULT 'pending'

&#x20;                 CHECK (status IN ('pending', 'in-progress', 'completed')),

&#x20; priority      VARCHAR(50) CHECK (priority IN ('high', 'medium', 'low'))

);

```



\*\*الجدول الموجز:\*\*



| السؤال | الإجابة |

|--------|---------|

| هل يوجد جدول موحد للمهام؟ | \*\*نعم\*\* — `tasks` — لكنه جزئي وقديم |

| هل المهمة مرتبطة بزبون؟ | \*\*لا بشكل حقيقي\*\* — `customer\_name` نص فقط، \*\*لا يوجد FK\*\* إلى `clients(id)` |

| هل لها نوع؟ | \*\*نعم\*\* — `emergency`, `dues`, `periodic`, `returns`, `followup` |

| هل لها سبب؟ | \*\*نعم\*\* — حقل `context TEXT` |

| هل لها تاريخ استحقاق؟ | \*\*نعم\*\* — `due\_date DATE` |

| هل لها حالة؟ | \*\*نعم\*\* — `pending / in-progress / completed` |

| هل لها أولوية؟ | \*\*نعم\*\* — `high / medium / low` |

| هل يمكن أن تكون قبل الزيارة؟ | \*\*نعم\*\* — مستقلة عن الزيارة |

| هل ترتبط بزيارة؟ | \*\*لا\*\* — لا يوجد FK لجدول `visits` أو `marketing\_visits` |

| هل يوجد distinction واضح بين task و visit؟ | \*\*نعم معمارياً\*\* — هما جدولان منفصلان بدون ربط |



\*\*الملاحظة الجوهرية:\*\* جدول `tasks` يستخدم `customer\_name` كنص حر بدلاً من `customer\_id` كـ FK. هذا يعني أن المهام القديمة \*\*مقطوعة\*\* من منظومة الزبائن الحديثة.

\## B) مهمة التسويق / عرض جهاز



\### مستويان مختلفان للمفهوم:



\*\*المستوى 1 — المهمة كـ "Lead محسوب" (غير مستقل):\*\*

لا يوجد record مستقل يُنشأ عند إضافة Lead. المهمة التسويقية هي \*\*نتيجة استعلام ديناميكي\*\* من `getPlanningMarketingTargets`:

```sql

\-- زبون بلا عقد + بلا زيارة + مسند للمشرف + ضمن النطاق = "مهمة تسويق"

WHERE NOT EXISTS (SELECT 1 FROM contracts WHERE customer\_id = c.id)

&#x20; AND NOT EXISTS (SELECT 1 FROM visits WHERE v.customer\_id = c.id)

```



\*\*المستوى 2 — المهمة كـ `marketing\_visit\_tasks` (مرتبطة بزيارة):\*\*

تُنشأ تلقائياً عند تثبيت الموعد، وتمثل `device\_demo` داخل الزيارة.



\*\*الواجهة — قائمة مهام الزيارة:\*\*

```typescript

&#x20;   { value: 'device\_demo', label: 'عرض جهاز' },

&#x20;   // خيار واحد فقط

];

// يبدأ دائماً بـ \['device\_demo'] كـ default

```



\*\*DB constraint:\*\*

```sql

\-- migration 051:67

&#x20; CHECK (task\_type IN ('device\_demo'))  -- قيمة واحدة مسموحة فقط

```



| السؤال | الإجابة |

|--------|---------|

| هل توجد مهمة تسويق مستقلة عند إنشاء Lead؟ | \*\*لا\*\* — محسوبة ديناميكياً |

| هل توجد مهمة باسم "عرض جهاز"؟ | \*\*نعم\*\* — `device\_demo` في `marketing\_visit\_tasks` |

| لكن أين تُنشأ؟ | \*\*داخل الزيارة فقط\*\* — بعد تثبيت الموعد، ليس قبله |

| هل يمكن تمييز زبون لديه مهمة تسويق؟ | \*\*نعم\*\* — عبر `planningMarketingTargets` (بدون عقد/زيارة) |

| هل للمهمة حالة قبل الزيارة؟ | \*\*لا\*\* — ليس لها record قبل الموعد |

| هل ترتبط بالتبعية الحالية؟ | \*\*نعم\*\* — تفلتر حسب `supervisorHrUserId` |

\## E) مهام التسليم والخدمة الأخرى

| السؤال | الإجابة |

|--------|---------|

| \*\*1. هل يوجد مفهوم "مهمة مفتوحة" عام؟\*\* | \*\*جزئياً\*\* — يوجد جدول `tasks` لكنه غير مربوط بـ `clients` ولا بالمسار الحديث |

| \*\*2. هل توجد مهمة تسويق مستقلة عند إنشاء Lead؟\*\* | \*\*لا\*\* — محسوبة ديناميكياً (بدون عقد/زيارة) |

\### الفجوات الأربع قبل توليد جهات الاتصال



\*\*الفجوة 1 — لا يوجد "كيان مهمة" مربوط بزبون قبل الزيارة:\*\*

المهام حالياً إما محسوبة (leads) أو منتجة داخل الزيارة (visit\_tasks). لا يوجد جدول يقول "هذا الزبون لديه مهمة مفتوحة من نوع كذا بتاريخ استحقاق كذا" بشكل مستقل عن الزيارة.



\*\*الفجوة 2 — `tasks` القديم مقطوع عن `clients`:\*\*

الجدول الوحيد الذي يحمل اسم "tasks" يستخدم `customer\_name` نصاً حراً بدون FK. لا يمكن استخدامه لمعرفة "ما المهام المفتوحة على الزبون رقم 123".



\*\*الفجوة 3 — `task\_list\_item` يمثل جهة اتصال وليس مهمة:\*\*

التيلماركتر حالياً تتصل بزبون (entity) لا بمهمة محددة. لا يعرف الـ item ما الذي يجب إنجازه بالضبط — هل هو عرض جهاز؟ تحصيل ذمة؟ صيانة؟ كل هذا غائب عن قائمة الاتصال.



\*\*الفجوة 4 — `marketing\_visit\_tasks.task\_type` محصور بـ `device\_demo`:\*\*

البنية التقنية لقائمة المهام داخل الزيارة ناضجة ومعمارياً سليمة، لكن DB constraint يمنع إضافة أنواع جديدة حالياً. إضافة: `periodic\_maintenance`, `collect\_dues`, `deliver\_gift`, `water\_calibration`... إلخ تتطلب توسيع هذا الـ CHECK.
