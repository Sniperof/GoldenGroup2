**8. تصميم الـ API المقترح**

**GET /api/marketing-visits?date=YYYY-MM-DD**
- permission:
  - `marketing_visits.view_list` أو permission مخصصة مشابهة
- scope:
  - acting branch فقط
- response:
  - list visits for selected date
- يعرض:
  - time
  - customer
  - phone
  - address
  - requested device
  - water source
  - team
  - supervisor/technician if available
  - status

**GET /api/marketing-visits/:id**
- permission:
  - `marketing_visits.view_detail`
- scope:
  - acting branch فقط
- response:
  - full visit + task result payload

**PATCH /api/marketing-visits/:id/result**
- permission:
  - `marketing_visits.update_result`
- scope:
  - acting branch فقط
- validation:
  - إذا `status = completed`:
    - `device_demo` result required
  - إذا `cash_offer_closed` أو `cash_offer_not_closed`:
    - `cashOfferAmount` required
  - إذا `installment_offer_closed` أو `installment_offer_not_closed`:
    - `installmentAmount` و `installmentMonths` required
  - إذا result is closed:
    - `closedByEmployeeId` required
  - إذا `status != completed`:
    - `reason/notes` required

**12. المخاطر الرئيسية**

- أكبر خطر هو استخدام جدول `visits` الحالي وإفساد تعريف `Lead`.
- ثاني خطر هو duplicate creation إذا كان إنشاء الزيارة غير idempotent داخل حجز الموعد.
- ثالث خطر هو إدخال صلاحيات معقدة مبكراً. لهذا MVP الأفضل إبقاؤها branch-level فقط.

**13. الأسئلة المفتوحة**

- هل نريد `closedByEmployeeId` أن يكون من employees فقط أم من hr_users؟
- هل team snapshot نحفظه JSON أم حقولاً منفصلة؟
- هل `customerAddress` تؤخذ snapshot كما هي من appointment أم من client وقت القراءة؟
- هل “لم يتم تقديم العرض” تعتبر `status=completed` مع task result `demo_not_completed`؟ أم نعدّها `not_completed`؟  
  المنتج الحالي يوحي أنها ضمن completed flow task-level، لكن تحتاج تثبيت نهائي.
