# تحليل دورة حياة المهام — المراحل والكيانات والانتقالات

> **الجمهور المستهدف:** مدير المنتج، الفريق التقني، أي طرف ثالث يقرأ هذا الملف لأول مرة
> **المشروع:** Golden CRM — منظومة إدارة علاقات العملاء الميدانية
> **تاريخ التحليل:** 2026-05-16
> **الحالة:** مسودة للمراجعة — المرحلة الخامسة مؤجّلة

---

## 1) السياق التشغيلي — لماذا هذا النظام موجود؟

تعمل الشركة من خلال فرق ميدانية تزور الزبائن لتنفيذ مهام متنوعة (عروض، تحصيل، صيانة...). هذا النظام يُنظّم رحلة كل مهمة من لحظة نشوئها حتى إغلاقها النهائي، عبر أربعة كيانات أساسية تعمل بشكل متزامن.

**المبدأ الجوهري:**
> المهمة هي الأصل — تمثّل الحاجة التشغيلية
> الزيارة هي الوسيلة — تمثّل الموعد المجدوَل للتنفيذ
> إلغاء الزيارة لا يُلغي المهمة. إلغاء المهمة يُزيل الحاجة للزيارة.

---

## 2) الكيانات الأربعة — تعريف كل منها

### 2.1 المهمة (Task)
**ما هي:** وحدة العمل التشغيلية الأساسية. تمثّل حاجة محددة يجب تنفيذها لدى زبون معيّن.

**أمثلة:** عرض جهاز للزبون، تحصيل قسط متأخر، إصلاح عطل طارئ.

**خصائصها:**
- ترتبط بزبون واحد وفرع واحد
- قد تملك تاريخ استحقاق (due_date) أو لا تملك
- قد تكون واحدة أو متعددة لنفس الزبون
- هي سبب وجود الزيارة — الزيارة لا تُنشأ بدون مهمة

---

### 2.2 جهة الاتصال (Contact Target)
**ما هي:** سجل يمثّل محاولة استهداف واحدة للتواصل مع الزبون بهدف جدولة زيارة.

**مبدأ أساسي:**
> كل محاولة استهداف = جهة اتصال جديدة مستقلة
> إذا فشلت المحاولة وبدأت جديدة في دورة لاحقة، تُنشأ جهة اتصال جديدة

**علاقتها بالمهمة:** جهة الاتصال قد تجمع عدة مهام لنفس الزبون في محاولة واحدة.

---

### 2.3 الزيارة (Visit)
**ما هي:** الموعد المجدوَل الذي يذهب فيه الفريق الميداني لتنفيذ المهام لدى الزبون.

**خصائصها:**
- فريق واحد + زبون واحد + تاريخ ووقت محددان
- قد تُنفّذ مهمة واحدة أو عدة مهام في زيارة واحدة
- يمكن إضافة مهام جديدة أثناء الزيارة
- مرتبطة دائماً بمهمة واحدة على الأقل

---

### 2.4 خطة العمل (Work Plan)
**ما هي:** الإطار اليومي لفريق معيّن يجمع: تشكيل الفريق + النطاق الجغرافي + الزبائن المستهدفين.

**مكوّناتها الثلاثة (تُبنى بهذا الترتيب):**
```
1. جدول الفرق     ← من هو الفريق اليوم (مشرف + فني + مسوّقون...)
2. نطاق العمل      ← الجغرافيا (مسارات + مناطق إضافية)
3. جهات الاتصال   ← الزبائن المؤهلون ضمن النطاق
```

**مسؤولية الإنشاء:** مدير الفرع حصراً.

## 3) أنواع المهام — تقسيم جوهري

### النوع الأول: مهام مفتوحة (لا تاريخ استحقاق ثابت)
| الخاصية | القيمة |
|---------|--------|
| due_date | لا يوجد |
| ظهور في التخطيط | فوري (عند الإنشاء) |
| تكرار لنفس الزبون | مقيَّد (عادة مهمة واحدة نشطة) |
| مثال | عرض جهاز، صيانة طارئة |

### النوع الثاني: مهام مقيّدة بتاريخ (ذات استحقاق)
| الخاصية | القيمة |
|---------|--------|
| due_date | إلزامي |
| ظهور في التخطيط | عند اقتراب الاستحقاق (ضمن نافذة N يوم) |
| تكرار لنفس الزبون | ممكن (كل قسط = مهمة مستقلة) |
| مثال | قسط عقد، رسوم صيانة دورية |

**قاعدة نافذة التخطيط N:**
> كل نوع مهمة له N خاص به يحدده الأدمن
> المهمة تظهر في التخطيط عند: due_date ≤ اليوم + N
> المهمة "لاحقة" (due_date > اليوم + N) لا تظهر في التخطيط حتى يقترب موعدها

## 5) المرحلة الأولى — قيد الانتظار

**التعريف:** المهمة موجودة في النظام لكنها لم تدخل أي خطة عمل بعد.

**الكيانات النشطة:** المهمة فقط (لا جهة اتصال، لا زيارة)

### 5.1 حالات المهمة في هذه المرحلة

| الحالة | الاسم التقني | المعنى |
|--------|-------------|--------|
| مفتوحة | `open` | مهمة جديدة تنتظر الإسناد |
| بحاجة متابعة | `needs_follow_up` | مهمة عادت من دورة سابقة فاشلة |

**البُعد الزمني (مشتق، لا يُخزَّن):**
| الحالة الزمنية | الشرط | الأثر |
|---------------|-------|-------|
| متأخرة | due_date < اليوم | تظهر في التخطيط بأولوية عالية |
| ضمن النافذة | اليوم ≤ due_date ≤ اليوم+N | تظهر في التخطيط |
| لاحقة | due_date > اليوم+N | **لا تظهر في التخطيط** حتى يقترب موعدها |

### 5.2 من أين تأتي المهمة؟
```
مصادر الإنشاء:
  ├── من نظام الإحالة (referral) → تلقائياً
  ├── يدوياً من الإدارة
  └── تلقائياً من مهمة سابقة أُعيدت جدولتها (بـ origin_ref_id)
```

### 5.3 من يملك صلاحية الإلغاء في هذه المرحلة؟
- **مدير الفرع** فقط يملك صلاحية إلغاء المهمة من هذه المرحلة
- **التلمارك** لا يملك الإلغاء — فقط إعادة المهمة لحالتها الأصلية

## 6) المرحلة الثانية — التخطيط

**التعريف:** المهمة دخلت خطة عمل فريق وتجري المحاولات لجدولة زيارة بشأنها.

**الكيانات النشطة:** المهمة + جهة الاتصال

**المهام التي تدخل هذه المرحلة:**
- المهام المفتوحة (`open`)
- المهام التي تحتاج متابعة (`needs_follow_up`)
- المهام المتأخرة (شرط زمني: due_date < اليوم)
- **لا تدخل:** المهام اللاحقة (due_date > اليوم + N)

### 6.1 مراحل خطة العمل (شرط دخول المهمة للتخطيط)

```
جدول الفرق جاهز
      ↓
نطاق العمل محدد (مسارات + مناطق)
      ↓
حساب الزبائن المؤهلين:
  شروط التأهل (يجب توافر الثلاثة):
    ✓ الزبون ضمن الفرع
    ✓ الزبون مسند تشغيلياً للفريق
    ✓ للزبون مهمة مؤهلة (مفتوحة أو بحاجة متابعة أو متأخرة)
      ↓
توليد جهات الاتصال
```

### 6.2 حالات المهمة في التخطيط

| الحالة | الاسم التقني | الانتقال |
|--------|-------------|---------|
| مسندة | `assigned` | عند إدراجها في حساب الحمل لفريق |
| قيد الجدولة | `in_scheduling` | عند توليد جهة الاتصال وإدراجها في القائمة |
| مجدولة | `scheduled` | عند حجز الموعد بنجاح |

**قاعدة مهمة:** المهمة التي لم تُختَر (لأن زبونها لديه جهة اتصال جاهزة) تعود لـ `last_waiting_status` في قيد الانتظار.

**قيد صارم:** لا يمكن إلغاء المهمة من مرحلة التخطيط.

### 6.3 حالات جهة الاتصال

| الحالة | المعنى | الانتقال إليها |
|--------|--------|---------------|
| جاهزة | جهة الاتصال أُنشئت | عند توليد الخطة |
| ضمن القائمة | أُضيفت لقائمة اتصال اليوم | عند تفعيل القائمة |
| تم التواصل | سُجِّل أي اتصال | أي نتيجة اتصال تنقلها هنا |
| مغلقة | انتهت المحاولة | بنتيجة / يدوياً / تلقائياً |

**قاعدة "تم التواصل":** هذه الحالة مستمرة — جهة الاتصال قد تبقى هنا عدة محاولات قبل أن يُقرر التلمارك إغلاقها.

### 6.4 كيف تُغلَق جهة الاتصال؟

| طريقة الإغلاق | الشرط | التأثير على المهمة |
|--------------|-------|------------------|
| نتيجة الاتصال (حجز موعد) | الزبون قبل موعداً | المهمة → `scheduled` |
| نتيجة الاتصال (رفض، إلغاء، خطأ...) | الزبون لم يقبل | المهمة → `last_waiting_status` |
| يدوي من التلمارك | انتهى وقت اليوم | المهمة → `last_waiting_status` (مع خيار تعديل date/priority) |
| تلقائي بعد m ساعة | Admin-config | المهمة → `last_waiting_status` (صامت، بلا تعديل) |

**الفرق بين اليدوي والتلقائي في الإعادة:**
- **يدوي:** تظهر نافذة تُتيح للتلمارك تعديل due_date وpriority اختيارياً
- **تلقائي (m ساعة):** رجوع صامت بدون نافذة تعديل

### 6.5 قواعد متعددة المهام لجهة اتصال واحدة

```
زبون لديه: ذمة قسط (متأخرة) + ذمة صيانة (ضمن النافذة) + صيانة طارئة

→ جهة اتصال واحدة تجمع المهام الثلاثة
→ الزبون يظهر مرة واحدة في قائمة التلمارك
→ الموعد يُحجز مرة واحدة لكل المهام
→ نتائج كل مهمة مستقلة في الزيارة
```

**مهمة جديدة تُضاف أثناء خطة قائمة:**
```
مدير الفرع يُضيف مهمة → يُعيد حساب الحمل
  → المهمة الجديدة تأخذ نفس حالة المهام الموجودة
  → لا تُولَّد جهة اتصال جديدة (تنضم للقائمة الحالية)
```

## 7) المرحلة الثالثة — التنفيذ

**التعريف:** الفريق في طريقه أو عند الزبون لتنفيذ المهام.

**الكيانات النشطة:** المهمة + الزيارة (جهة الاتصال انتهت دورها)

### 7.1 حالات الزيارة في التنفيذ

| الحالة | الانتقال إليها | الخروج |
|--------|---------------|--------|
| استحقاق الزيارة | تلقائي عند يوم الجدولة | زر "بدء الزيارة" |
| ضمن الزيارة | زر "بدء الزيارة" يدوياً | زر "إنهاء الزيارة" أو v ساعة |
| نهاية الزيارة | زر "إنهاء" أو v ساعة | دخول مرحلة الإغلاق |

**انتقال "مجدولة → استحقاق الزيارة":**
- تلقائي في منتصف الليل (cron)
- فوري إذا كانت الجدولة في نفس اليوم

**حالة خاصة:** الزيارة استحقّت ولم تبدأ
```
├── يدوياً: المشرف يُنهيها + يُدخل سبباً
└── تلقائياً: بعد z ساعة (Admin-config) + سبب ثابت "لم تُنفَّذ"
```

### 7.2 حالات المهمة في التنفيذ

| حالة المهمة | تتوافق مع |
|------------|-----------|
| بانتظار التنفيذ | استحقاق الزيارة |
| قيد التنفيذ | ضمن الزيارة |
| انتهت | نهاية الزيارة |

**قاعدة التزامن:** المهام تتحرك جماعياً مع حالة الزيارة.
**قاعدة استقلال النتائج:** كل مهمة تُسجَّل نتيجتها بشكل مستقل.

### 7.3 الانتقال الفوري للإغلاق

```
مهمة في "قيد التنفيذ" + تُسجَّل نتيجتها
    ↓
تنتقل فوراً لمرحلة الإغلاق (اكتملت)
    ↓ تتجاوز حالة "انتهت"
الزيارة لا تتأثر — تبقى "ضمن الزيارة" حتى إنهائها
```

### 7.4 إضافة مهمة أثناء الزيارة

```
المهمة الجديدة تُنشأ كـ "مفتوحة" في قيد الانتظار
    ↓
تنتقل فوراً إلى حالة الزيارة الحالية:

  زيارة مجدولة         → المهمة: مجدولة
  زيارة استحقت         → المهمة: بانتظار التنفيذ
  زيارة ضمن الزيارة   → المهمة: قيد التنفيذ
```

### 7.5 إلغاء الزيارة في التنفيذ

```
الزيارة تُلغى ←←
      ↓
المهام تعود إلى last_waiting_status في قيد الانتظار
(المهام لا تُلغى — الحاجة لا تزال موجودة)
      ↓
الزيارة: ملغاة
```

### 7.6 إلغاء مهمة في مرحلة التنفيذ

```
لا يمكن إلغاء مهمة بعد تسجيل نتيجتها.
يمكن الإلغاء فقط خلال "مرحلة تسجيل النتائج" (قبل التأكيد النهائي)

إذا أُلغيت المهمة الوحيدة في الزيارة → الزيارة تُلغى تلقائياً
إذا أُلغيت مهمة من عدة → الزيارة تكمل بالمهام المتبقية
```

## 8) المرحلة الرابعة — الإغلاق

**التعريف:** تسجيل النتائج ومراجعتها واعتمادها بشكل نهائي لا رجعة فيه.

**الكيانات النشطة:** المهمة + الزيارة

### 8.1 شرط دخول مرحلة الإغلاق

```
الزيارة لا تدخل الإغلاق إلا إذا:
  ✓ كل المهام لها حالة نهائية: (اكتملت أو ملغاة)
  ✓ شروط الزيارة الإضافية (تُحدد لاحقاً):
      - موقع بدء الزيارة (GPS) مُسجَّل
      - لائحة الأسماء المقترحة منجزة (إن كانت مطلوبة)
      - [شروط أخرى حسب نوع الزيارة]
```

### 8.2 حالات الزيارة في الإغلاق

| الحالة | المعنى | الانتقال |
|--------|--------|---------|
| مكتملة | كل المهام لها نتائج + شروط الزيارة محققة | بعد نهاية الزيارة |
| مغلقة | مراجعة الطرف الثالث واعتماده | بعد المراجعة (نهائي) |
| ملغاة | الزيارة لم تتم (من التنفيذ) | من مرحلة التنفيذ |

### 8.3 حالات المهمة في الإغلاق

| الحالة | المعنى | الانتقال |
|--------|--------|---------|
| اكتملت | النتيجة مُسجَّلة | عند تسجيل النتيجة |
| اغلقت | المراجعة اعتمدتها (نهائي لا رجعة) | بعد اعتماد المراجع |

### 8.4 المراجعة — مبدأ الطرف الثالث

```
من يراجع؟
  من يملك صلاحية visits.review
  → قد يكون نفس المنفّذ (مراجعة ذاتية)
  → قد يكون شخص إداري مختلف

ماذا يفعل المراجع؟
  ✓ يُدقّق في النتائج
  ✓ إذا وجد خطأً → يُعدّل مباشرة (في مكانه)
  ✗ لا يُعيد المهمة لأي حالة سابقة
  ✗ لا يتراجع عن النتيجة

بعد اعتماده:
  ✗ لا يمكن لأحد تعديل النتائج مهما كان

```

### 8.5 قيد الإلغاء في الإغلاق

```
اكتملت (نتيجة مسجَّلة) → لا يمكن إلغاء المهمة
اغلقت (معتمَدة)         → لا يمكن تعديل أي شيء
```

## 10) مصفوفة الانتقالات الكاملة

### 10.1 انتقالات المهمة

```
قيد الانتظار
  مفتوحة ────────────────────────────────────────────────→ ملغاة
  مفتوحة ──[حساب الحمل]──────────────────────────────────→ مسندة
  بحاجة متابعة ──[حساب الحمل]──────────────────────────→ مسندة

التخطيط
  مسندة ──[توليد جهة الاتصال]─────────────────────────→ قيد الجدولة
  مسندة ──[جهة اتصال جاهزة موجودة]──────────────────→ last_waiting_status
  قيد الجدولة ──[حجز الموعد]──────────────────────────→ مجدولة
  قيد الجدولة ──[إغلاق جهة الاتصال بلا حجز]──────────→ last_waiting_status

التنفيذ
  مجدولة ──[يوم الجدولة]──────────────────────────────→ بانتظار التنفيذ
  بانتظار التنفيذ ──[بدء الزيارة]────────────────────→ قيد التنفيذ
  قيد التنفيذ ──[إنهاء الزيارة / v ساعة]──────────────→ انتهت
  قيد التنفيذ ──[تسجيل النتيجة]──────────────────────→ اكتملت (مباشرة للإغلاق)
  [أي حالة قبل النتيجة] ──[إلغاء الزيارة]────────────→ last_waiting_status

الإغلاق
  اكتملت ──[مراجعة واعتماد]──────────────────────────→ اغلقت (نهائي)
```

### 10.2 انتقالات جهة الاتصال

```
جاهزة
  ──[إضافة للقائمة اليومية]────────────────────────→ ضمن القائمة

ضمن القائمة
  ──[أي تسجيل نتيجة اتصال]────────────────────────→ تم التواصل
  ──[يدوي من التلمارك]─────────────────────────────→ مغلقة
  ──[تلقائي بعد m ساعة]────────────────────────────→ مغلقة

تم التواصل
  ──[نتيجة: حجز موعد]──────────────────────────────→ مغلقة (سبب: booked)
  ──[نتيجة: رفض/إلغاء/خطأ]──────────────────────→ مغلقة (سبب محدد)
  ──[يدوي من التلمارك]─────────────────────────────→ مغلقة

مغلقة
  ──────────────────────────────────────────────────→ [نهاية المحاولة]
```

### 10.3 انتقالات الزيارة

```
التخطيط
  [تُنشأ عند الحجز] → مجدولة

التنفيذ
  مجدولة ──[يوم الجدولة]──────────────────────────→ استحقاق الزيارة
  استحقاق ──[بدء الزيارة يدوياً]────────────────→ ضمن الزيارة
  استحقاق ──[z ساعة بلا بدء]─────────────────────→ نهاية الزيارة (لم تُنفَّذ)
  استحقاق ──[إلغاء يدوي + سبب]───────────────────→ ملغاة
  ضمن الزيارة ──[إنهاء يدوي]───────────────────→ نهاية الزيارة
  ضمن الزيارة ──[v ساعة بلا إنهاء]─────────────→ نهاية الزيارة

الإغلاق
  نهاية الزيارة ──[كل المهام + شروط الزيارة]───→ مكتملة
  مكتملة ──[مراجعة واعتماد]───────────────────→ مغلقة (نهائي)
  [أي مرحلة] ──[إلغاء]────────────────────────→ ملغاة
```

## 12) القواعد التشغيلية الأساسية

| # | القاعدة | الوصف |
|---|--------|-------|
| Q01 | المهمة هي الأصل | الزيارة لا تُنشأ إلا بسبب مهمة. الزيارة وسيلة والمهمة الحاجة. |
| Q02 | كل محاولة = جهة اتصال جديدة | كل دورة استهداف مستقلة ولها سجل منفصل. |
| Q03 | إلغاء الزيارة لا يُلغي المهمة | المهام تعود لـ last_waiting_status عند إلغاء الزيارة. |
| Q04 | إلغاء المهمة يُلغي الحاجة | إذا كانت المهمة هي الوحيدة → تُلغى الزيارة تلقائياً. |
| Q05 | الإغلاق نهائي | بعد اعتماد المراجع لا يمكن تعديل أي شيء. |
| Q06 | المراجع لا يُعيد | إذا وجد خطأ يُعدّل مباشرة، المهمة لا تعود لأي حالة سابقة. |
| Q07 | لا إلغاء في التخطيط | المهمة لا تُلغى من مرحلة التخطيط. |
| Q08 | لا إلغاء بعد النتيجة | بعد تسجيل النتيجة لا يمكن إلغاء المهمة. |
| Q09 | المهام تتزامن جماعياً | كل مهام الزيارة تتحرك معاً عبر حالات التنفيذ. |
| Q10 | النتائج مستقلة | كل مهمة تُسجَّل نتيجتها بشكل مستقل. |
| Q11 | الرجوع اليدوي مع تعديل | عند رجوع التلمارك يدوياً: خيار تعديل date/priority. |
| Q12 | الرجوع التلقائي صامت | عند انتهاء m ساعة: رجوع بلا نافذة تعديل. |
| Q13 | المهمة اللاحقة لا تُخطَّط | due_date > اليوم+N → لا تظهر في التخطيط. |
| Q14 | المهمة المتأخرة تُخطَّط فوراً | due_date < اليوم → تدخل التخطيط بأولوية. |
| Q15 | N لكل نوع مهمة | كل نوع مهمة له نافذة تخطيط مستقلة يضبطها الأدمن. |

## 16) مسرد المصطلحات

| المصطلح | التعريف |
|--------|--------|
| due_date | تاريخ استحقاق المهمة — متى يجب تنفيذها |
| N (planning window) | عدد الأيام قبل الاستحقاق تظهر فيها المهمة في التخطيط |
| m ساعة | مهلة الإغلاق التلقائي لجهة الاتصال بلا تواصل |
| z ساعة | مهلة إنهاء الزيارة المستحقة بلا بدء |

| origin_ref_id | رابط المهمة الجديدة بالمهمة السابقة عند إعادة الجدولة |
| visits.review | صلاحية مراجعة نتائج الزيارة واعتمادها |
| planning.manage | صلاحية إنشاء وإدارة خطط العمل |
