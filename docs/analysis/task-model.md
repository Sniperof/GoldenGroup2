# موديل المهمة الموحد — المرجع الشامل

> **الحالة:** الأساس مُنفَّذ ✅ — يستمر التوسع مع كل نوع مهمة جديد
> **تاريخ آخر تحديث:** 2026-05-16
> **المصادر:** نقاشات مباشرة مع صاحب المنتج + PDF مهام الزيارات + فحص قاعدة البيانات الفعلية
> **Migrations المرتبطة:** 101 (عنوان التركيب) — 102 (حقول المرحلة الزمنية) — 103 (حقول مرحلة الانتظار) — 105 (إعادة تسمية الحالات وتوسعتها لـ 11)

## 1) المبدأ الجوهري

```
الزبون
  └── عقد 1  →  جهاز واحد  +  عنوان تركيب خاص
  └── عقد 2  →  جهاز واحد  +  عنوان تركيب خاص
  └── عقد 3  →  جهاز واحد  +  عنوان تركيب خاص

المهمة ترتبط بـ: الزبون + العقد (وليس بالزبون فقط)
العقد هو المصدر الحقيقي لـ: الجهاز + عنوان الزيارة
```

**القاعدة المحورية المتفق عليها:**
> كل مهمة تعيش في جدول `open_tasks` الواحد.
> لا يوجد جدول مستقل لكل نوع مهمة.
> التفاصيل الخاصة بنوع المهمة تُخزَّن في جداول نتائج منفصلة فقط.

## 2) الحقول الأساسية لكل مهمة

هذه الحقول موجودة في **كل مهمة** بدون استثناء.

### 2.1 حقول الهوية — من هي هذه المهمة؟

| الحقل | النوع | الإلزامية | الوصف |
|-------|-------|:---------:|-------|
| `id` | INTEGER | ✓ | معرف فريد تلقائي |
| `task_type` | VARCHAR | ✓ | نوع المهمة (من قائمة الـ 20) |
| `task_family` | VARCHAR | ✓ | العائلة التشغيلية |
| `client_id` | INTEGER FK | ✓ | الزبون المرتبط بالمهمة |
| `contract_id` | INTEGER FK | ✗* | العقد المرتبط — **null فقط لنوعين محددين** |
| `branch_id` | INTEGER FK | ✓ | الفرع المسؤول عن التنفيذ |

> **\* قاعدة `contract_id`:** يكون `null` فقط في:
> - `device_demo` — مهمة ما قبل البيع، لا عقد بعد
> - `gift_delivery` — قد تكون هدية وسيط بدون عقد
>
> كل المهام الـ 18 الأخرى تحتاج عقداً.

### 2.2 حقول دورة الحياة — أين هي المهمة الآن؟

| الحقل | النوع | الإلزامية | الوصف |
|-------|-------|:---------:|-------|
| `status` | VARCHAR | ✓ | الحالة الحالية في دورة الحياة (إحدى 11 قيمة) |
| `last_waiting_status` | VARCHAR | ✗ | آخر حالة انتظار قبل التخطيط — تُكتب تلقائياً عند الخروج من قيد الانتظار |
| `priority` | VARCHAR | ✗ | الأولوية: `high` / `medium` / `low` (اختيارية — قد لا تُحدَّد) |
| `due_date` | DATE | ✗ | **تاريخ الاستحقاق** — صارم — فواته يجعل المهمة "متأخرة" رسمياً |
| `expected_date` | DATE | ✗ | **الموعد المتوقع** — ليّن — فواته يُبقي المهمة "مفتوحة" مع إشارة بصرية |
| `waiting_reason_id` | INTEGER FK | ✗ | سبب الانتظار المنظَّم (FK لـ `system_lists` فئة `telemarketing_reschedule_reason`) |
| `waiting_reason_text` | TEXT | ✗ | نسخة نصية مقروءة من سبب الانتظار |
| `attempt_count` | INTEGER | ✓ | عدد محاولات الاتصال (يُزاد تلقائياً عند كل `call_made`) — افتراضياً 0 |
| `last_attempt_at` | TIMESTAMPTZ | ✗ | تاريخ آخر محاولة اتصال |

**تمييز مهم — ثلاثة مفاهيم مختلفة لا يجوز خلطها:**

| المفهوم | النوع | المعنى | التأثير عند الفوات |
|---------|-------|--------|-------------------|
| `due_date` — تاريخ الاستحقاق | حقل تاريخ | موعد إلزامي للتنفيذ | المهمة → **متأخرة** رسمياً |
| `expected_date` — الموعد المتوقع | حقل تاريخ | وعد ليّن مع الزبون | المهمة → **مفتوحة** + إشارة بصرية فقط |
| `needs_follow_up` — بحاجة متابعة | حالة (status) | المهمة عادت من دورة سابقة فاشلة | لا علاقة له بأي تاريخ |

> **القاعدة:** `needs_follow_up` هي **مكانة في دورة الحياة** — تعني أن المهمة جُرّبت مرة وأُعيدت.
> أما `expected_date` فهو **إشارة زمنية ليّنة** — يعني "نتوقع العودة لهذا الزبون بهذا التاريخ".
> الاثنان مستقلان تماماً ويمكن أن يجتمعا في نفس المهمة.

**مثال:**
```
مهمة عرض جهاز:
  status = 'needs_follow_up'     ← عادت بعد أن رفض الزبون الأسبوع الماضي
  expected_date = 2026-05-23     ← وعد بالنظر في الأسبوع القادم
  due_date = null                ← لا يوجد استحقاق صارم
```

---

**قيم `status` المعتمدة (11 قيمة):**

| الحالة | الاسم التقني | المرحلة |
|--------|-------------|---------|
| مفتوحة | `open` | قيد الانتظار |
| بحاجة متابعة | `needs_follow_up` | قيد الانتظار |
| مسندة | `assigned` | التخطيط |
| قيد الجدولة | `in_scheduling` | التخطيط |
| مجدولة | `scheduled` | التخطيط |
| بانتظار التنفيذ | `waiting_execution` | التنفيذ |
| قيد التنفيذ | `in_execution` | التنفيذ |
| انتهت | `ended` | التنفيذ |
| مكتملة | `completed` | الإغلاق |
| مغلقة نهائياً | `closed` | الإغلاق |
| ملغاة | `cancelled` | خروج من أي مرحلة |

> **ملاحظة ترحيل:** القيم الحالية في قاعدة البيانات مختلفة (7 قيم).
> الانتقال مطلوب — سجَّل في قسم المهام المعلقة [T01].

### 2.2.1 الطبقات الثلاث للمهمة (المفهوم المحوري)

> **هذا تعريف معماري لا يجوز خلطه.** كل مهمة تُوصَف بثلاث طبقات مستقلة تماماً، لكل منها سؤال ودور مختلف.

```
┌─────────────────────────────────────────────────────────────────┐
│ الطبقة 1 — المرحلة (Phase)                                       │
│   السؤال: أين هي المهمة في رحلتها الكلية؟                       │
│   القيم: قيد الانتظار → التخطيط → التنفيذ → الإغلاق             │
│   التخزين: مشتقّة من status (لا تُخزَّن كحقل مستقل)              │
│   الاستخدام: تحدد من له صلاحية التصرف                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ الطبقة 2 — الحالة (State)                                       │
│   السؤال: ماذا يحدث للمهمة الآن داخل المرحلة؟                   │
│   القيم: open / needs_follow_up / assigned / ... (11 قيمة)      │
│   التخزين: حقل status في open_tasks                            │
│   الاستخدام: تحدد الإجراءات الممكنة الآن                       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ الطبقة 3 — البُعد الزمني (Temporal)                              │
│   السؤال: كم الوقت المتبقي للاستحقاق؟                          │
│   القيم: متأخرة / ضمن النافذة / لاحقة / مفتوحة بلا تاريخ        │
│   التخزين: مشتقّ لحظياً من due_date و N (لا يُخزَّن)             │
│   الاستخدام: تحدد ظهور المهمة في حساب الحمل والقوائم            │
└─────────────────────────────────────────────────────────────────┘
```

**جدول المرحلة → الحالات الممكنة:**

| المرحلة | الحالات الفنية الممكنة |
|---------|----------------------|
| قيد الانتظار | `open`, `needs_follow_up` |
| التخطيط | `assigned`, `in_scheduling`, `scheduled` |
| التنفيذ | `waiting_execution`, `in_execution`, `ended` |
| الإغلاق | `completed`, `closed`, `cancelled` |

**جدول البُعد الزمني — يُحسب لحظياً (لا يُخزَّن):**

| التصنيف الزمني | الشرط |
|---------------|-------|
| متأخرة | `due_date < اليوم` |
| ضمن النافذة | `اليوم ≤ due_date ≤ اليوم + N` (N حسب نوع المهمة) |
| لاحقة | `due_date > اليوم + N` |
| مفتوحة بلا تاريخ | `due_date IS NULL` |

**مثال على مهمة واحدة بثلاث طبقات متزامنة:**
```
المرحلة:  قيد الانتظار
الحالة:   needs_follow_up    (رجعت من اتصال فاشل أمس)
الزمني:   متأخرة             (due_date كان قبل 3 أيام)
```

### 2.2.2 قاعدة حساب الحمل ضمن نطاق العمل

> **القاعدة الذهبية:** **المهمة "لاحقة" لا تدخل حساب الحمل.**

**السبب:** المهمة اللاحقة ليست قابلة للعمل الآن — لا داعي لإثقال خطة الفريق بها.

**معايير دخول المهمة في حساب الحمل:**

```sql
المهمة تدخل الحمل إذا تحقق كل من:
  ✓ المرحلة: قيد الانتظار  →  status IN ('open', 'needs_follow_up')
  ✓ ضمن نطاق العمل           →  الزبون مسند للفرع/الفريق
  AND البُعد الزمني يسمح:
      due_date IS NULL                        ← مفتوحة بلا تاريخ
      OR due_date <= CURRENT_DATE + N         ← متأخرة أو ضمن النافذة
                                                 (تُستثنى "لاحقة")
```

**الانتقال الطبيعي بين التصنيفات الزمنية:**

```
مهمة جديدة، due_date = بعد شهر، N = 7
  ↓ التصنيف: لاحقة          → خارج حساب الحمل
الزمن يمر، تبقى 6 أيام للاستحقاق
  ↓ التصنيف: ضمن النافذة     → تدخل الحمل تلقائياً
يُنجَزها الفريق، تُسجَّل النتيجة
  ↓ تخرج من قيد الانتظار، قد تُولِّد مهمة لاحقة (مثل صيانة تالية)
```

**ملاحظة:** لا حاجة لـ scheduled job يحوّل الحالات — فلتر `WHERE` كافٍ. التصنيف يُحسب لحظياً من `CURRENT_DATE` + `due_date` + `N`.

## 4) سلاسل المهام — الترابط التشغيلي

المهام ليست مستقلة دائماً. بعضها يُنشئ بعضاً تلقائياً:

```
device_demo  ──[وافق الزبون]──→  device_purchase
                                        │
                          ┌─────────────┼─────────────┐
                          ↓             ↓             ↓
                   device_delivery  device_installation  [أقساط]
                                         │         installment_collection ×N
                                    device_activation
                                         │
                              [يبدأ عداد الصيانة الدورية]
                                         │
                              periodic_maintenance (1)
                                         │
                              periodic_maintenance (2)
                                        ...

emergency_maintenance ──[سحب الجهاز]──→ device_retrieval
                                               │
                                        device_repair
                                               │
                                        device_return
```

**قاعدة `origin_ref_id`:**
كل مهمة نشأت تلقائياً من مهمة أخرى تحمل `origin_ref_id` يشير للمهمة الأم.
هذا يبني سجلاً كاملاً لتاريخ الجهاز منذ العرض حتى آخر صيانة.

### ✅ G09 — "لاحقة" لم تكن تُستثنى من حساب الحمل *(مغلقة)*
