# الدليل المرجعي لتوثيق المهام — Golden CRM

> **الجمهور:** صاحب المنتج وفريق الأعمال
> **الغرض:** كل مهمة جديدة في النظام تُوثَّق بنفس هذه البنية قبل أن يبدأ التنفيذ
> **المبدأ:** مهمة غير موثقة = مهمة لا يمكن تنفيذها

---

## لماذا هذا الملف؟

كل مهمة ميدانية تنفذها الفرق تحتاج إلى إجابات واضحة على ثلاثة أسئلة:

1. **من أين تأتي هذه المهمة، ومتى؟**
2. **ماذا يجب أن يحدث أثناء تنفيذها؟**
3. **كيف تنتهي، وما الذي يأتي بعدها؟**

هذا الملف يجبر كل مهمة على الإجابة على هذه الأسئلة بشكل موحد.

---

## البنية الموحدة لكل مهمة

كل مهمة تُوثَّق في قسمين:

**القسم الأول: البيانات الثابتة** — وهي نفسها لكل مهمة بدون استثناء
**القسم الثاني: بيانات التنفيذ** — وهي خاصة بكل نوع مهمة

---

## القسم الأول: البيانات الثابتة (مشتركة بين كل المهام)

هذه المعلومات تُسجَّل تلقائياً لكل مهمة في النظام، بغض النظر عن نوعها:

| البيان | الوصف | مثال |
|--------|-------|------|
| **نوع المهمة** | ما هي هذه المهمة | عرض جهاز / تحصيل قسط / صيانة طارئة |
| **الزبون** | لمن تُنفَّذ هذه المهمة | أحمد علي (رقم 4521) |
| **الفرع** | أي فرع مسؤول عن تنفيذها | دمشق / طرطوس |
| **الحالة** | أين المهمة الآن ضمن 11 حالة معتمدة | open / needs_follow_up / scheduled / completed |
| **المرحلة** | قراءة مشتقة من الحالة | قيد الانتظار / التخطيط / التنفيذ / الإغلاق |
| **البعد الزمني** | قراءة مشتقة من التواريخ ونافذة N | متأخرة / ضمن النافذة / لاحقة |
| **الأولوية** | مدى إلحاحية التنفيذ | عالية / متوسطة / منخفضة |
| **تاريخ الاستحقاق** | موعد صارم للتنفيذ `due_date` | 15-06-2026 |
| **الموعد المتوقع** | موعد لين للعودة أو المتابعة `expected_date` | 20-06-2026 |
| **سبب الانتظار** | سبب منظم أو نصي لعودة المهمة للانتظار | طلب الزبون متابعة لاحقة |
| **عداد المحاولات** | عدد محاولات التواصل أو المعالجة | 3 |
| **سبب الإنشاء** | لماذا أُنشئت هذه المهمة | عميل جديد / متابعة / إعادة جدولة |
| **مصدر الإنشاء** | كيف ظهرت في النظام | يدوي / تلقائي / من نتيجة مهمة سابقة |
| **المهمة الأم** | هل جاءت بعد مهمة أخرى؟ | ناتجة عن مهمة تشغيل جهاز رقم 3201 |
| **ملاحظات** | أي معلومة إضافية مفيدة | — |
| **بيانات الزبون وقت الإنشاء** | لقطة ثابتة من معلومات الزبون لحظة إنشاء المهمة | العنوان والموبايل والمنطقة |
| **تاريخ الإنشاء** | متى دخلت المهمة النظام | 2026-05-16 |
| **من أنشأها** | المستخدم الذي سجّلها | مدير الفرع / النظام تلقائياً |

## القسم الثاني: بيانات التنفيذ (خاصة بكل نوع)

هذا هو القسم الذي يختلف من مهمة لأخرى. يجب أن يُجيب على:

### أ) شرط النشوء — متى تُنشأ هذه المهمة؟

مثال (عرض جهاز):
> تُنشأ يدوياً من الإدارة عند وجود زبون مرشح لعرض جهاز، أو تلقائياً من نتيجة اتصال في قائمة التلمارك

مثال (صيانة دورية):
> تُنشأ تلقائياً بعد مهمة تشغيل الجهاز بمدة محددة (يحددها الأدمن)

مثال (تحصيل قسط):
> تُنشأ تلقائياً من العقد بتاريخ استحقاق كل قسط

---

### ب) ما الذي يجب تسجيله أثناء التنفيذ؟

هذه هي الحقول الخاصة بكل مهمة. تُملأ من الفريق الميداني في الزيارة.

**مثال: عرض جهاز**
- نتيجة فحص المياه (TDS قبل / كلور / بعد شبك الفلتر)
- نوع الجهاز المعروض
- نوع العرض المالي (كاش / تقسيط / عدد الأقساط)
- استجابة الزبون (قبل / رفض / متابعة لاحقاً)
- اسم الموظف الذي تم التسكير معه

**مثال: تركيب جهاز**
- الاكسسوارات المستخدمة
- الاكسسوارات المُعادة للشركة
- الاكسسوارات التي سُلِّمت للزبون دون استخدام
- ملاحظات التركيب

**مثال: تحصيل قسط**
- المبلغ المستحق
- المبلغ الذي دفعه الزبون فعلياً
- طريقة الدفع
- رقم الإيصال
- في حال دفع جزئي: الموعد الجديد للمبلغ المتبقي

---

### ج) كيف تنتهي هذه المهمة؟ (النتائج الممكنة)

كل مهمة يجب أن تنتهي بواحدة من نتيجتين رئيسيتين فقط:

| النتيجة | المعنى |
|---------|--------|
| **منفذة** | تم تحقيق هدف المهمة |
| **غير منفذة** | لم يتحقق الهدف — يجب ذكر السبب |

ثم تفصيل ضمن كل نتيجة حسب نوع المهمة.

**مثال: صيانة دورية**
- منفذة: تم تبديل قطعة واحدة على الأقل
- غير منفذة: رفض الزبون التبديل (مع ذكر السبب) / طلب تأجيل

**مثال: صيانة طارئة**
- منفذة: تم حل المشكلة (بتبديل قطع أو بدونها)
- منفذة جزئياً: الزبون وافق على حل مؤقت برفض التكلفة الكاملة
- غير منفذة (بسبب الفني): ليس لديه القطعة / لم يحدد المشكلة → سحب الجهاز
- غير منفذة (بسبب الزبون): التكلفة عالية / التأجيل / قرر شراء جهاز جديد

---

### د) ماذا يحدث بعد انتهاء المهمة؟ (الأثر اللاحق)

| الأثر | هل يحدث تلقائياً؟ |
|-------|:-----------------:|
| إنشاء مهمة جديدة | ✓ أو ✗ |
| تحديث حالة الجهاز | ✓ أو ✗ |
| تحديث حالة الكفالة | ✓ أو ✗ |
| فتح ذمة مالية | ✓ أو ✗ |
| إرسال إشعار | ✓ أو ✗ |

**مثال: تشغيل الجهاز**
> ينشئ النظام تلقائياً أول مهمة صيانة دورية بتاريخ استحقاق = تاريخ التشغيل + N شهر

**مثال: توقيع عقد شراء**
> ينشئ النظام تلقائياً مهام: تسليم الجهاز + تركيبه + تشغيله بتواريخ محددة

## مثال مكتمل — مهمة عرض الجهاز

### البيانات الثابتة
| البيان | القيمة |
|--------|--------|
| نوع المهمة | عرض جهاز |
| تاريخ الاستحقاق | لا يوجد (مهمة مفتوحة) |
| الأولوية | متوسطة |
| سبب الإنشاء | عميل جديد |
| مصدر الإنشاء | يدوي / تلقائي من التلمارك |
| المهمة الأم | لا يوجد (مهمة بداية) |

### شرط النشوء
تُنشأ عند وجود زبون مرشح لعرض جهاز فلتر مياه.
لا يمكن أن يكون للزبون أكثر من مهمة عرض جهاز نشطة في نفس الوقت.

### ما يُسجَّل أثناء التنفيذ
- نتيجة فحص المياه (TDS قبل / بعد / كلور)
- نوع الجهاز المعروض
- العرض المالي المقدم (كاش أو تقسيط)
- استجابة الزبون
- اسم الموظف المسؤول عن التسكير

### النتائج الممكنة
| النتيجة | التفصيل |
|---------|---------|
| منفذة — عرض مقدم وقبل | انتقل للعقد (مهمة شراء جهاز) |
| منفذة — عرض مقدم ورفض | أُغلقت، لا متابعة |
| منفذة — عرض مقدم وطلب متابعة | مهمة متابعة بتاريخ لاحق |
| غير منفذة | لم يوصل للعرض، سبب إلزامي |

### الأثر اللاحق
| النتيجة | الأثر |
|---------|-------|
| وافق على الشراء | النظام يُنشئ مهمة (شراء جهاز) تلقائياً |
| طلب متابعة | مهمة (عرض جهاز) جديدة بتاريخ المتابعة |
| رفض نهائي | لا شيء — تُغلق المهمة |

## ملاحظة دستورية

دورة حياة كل مهمة وحالاتها العامة لا تُعاد تعريفها في هذا القالب. المرجع الحاكم هو:

- `docs/constitution/domains/tasks.md`

أي نوع مهمة جديد يملأ تفاصيل النشوء والتنفيذ والنتائج والأثر اللاحق، لكنه يبقى ملتزماً بنموذج الحالات والمرحلة والبعد الزمني المعتمد هناك.

## كيف نُجهّز مهمة جديدة؟

لإضافة أي من المهام العشرين للنظام، نحتاج فقط إلى ملء هذا النموذج:

```
اسم المهمة:
متى تُنشأ؟ (شرط النشوء):
هل لها تاريخ استحقاق؟ نعم / لا
هل يمكن وجود أكثر من مهمة لنفس الزبون؟ نعم / لا

ما الذي يُسجَّل أثناء التنفيذ؟
  1.
  2.
  3.
  ...

النتائج الممكنة:
  - منفذة:         [التفصيل]
  - غير منفذة:     [التفصيل + أسباب إلزامية]

الأثر اللاحق:
  - إذا منفذة →   [مهمة جديدة / تحديث / لا شيء]
  - إذا غير منفذة → [...]
```

**عند اكتمال هذا النموذج لكل مهمة — يمكن التنفيذ فوراً دون نقاشات إضافية.**
