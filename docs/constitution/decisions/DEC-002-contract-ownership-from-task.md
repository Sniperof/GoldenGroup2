# قرار معماري DEC-002: ملكية البيعة وربط العقد بالمهمة

> **التاريخ:** 2026-05-27

> **المعتمد:** إبراهيم عبيد
> **الفئة:** عقود مبيعات (Contracts) + مهام ميدانية (Open Tasks)
> **الإصدار:** 2.0 (مصحح — تم إزالة team_id لأن الفرق يومية JSONB وليست كيانات ثابتة)

---

## 1. الملخص التنفيذي (Executive Summary)

تحديد قاعدة ملكية البيعة: **العقد يُربط كاملاً بالمهمة المسببة (`source_open_task`)، والملكية الفردية للمشرفة/الفني اللي أنهى المهمة.**

أي موظف بصلاحية إنشاء عقد (بما فيه المدير) — إذا دوره مش مشرف/فني — ملكيته للفرع فقط، وهو مجرد منشئ للعقد.

> ⚠️ **تصحيح مهم:** الفرق (`teams`) تُخزّن كـ JSONB داخل `day_schedules` وتتغير يومياً. لا يوجد `teams` table كيان ثابت. لذلك لا يمكن ربط العقد بـ "فريق" — الملكية فردية فقط.

---

## 2. السيناريو العملي (Practical Scenario)

```
مشرفة "سارة" من فرع حمص:
  │
  ├─ مهمة device_demo (task_id=501) ← assigned_employee_id = سارة
  │
  └─ الزبون وافق → المدير "خالد" ضغط "إنشاء عقد"
       │
       └─ العقد C-5002:
            ├─ source_open_task_id = 501 ← ربط كامل بالمهمة
            ├─ created_by = خالد ← للتدقيق فقط
            ├─ branch_id = فرع حمص ← الإيراد للفرع
            ├─ الملكية الفردية = سارة ← من assigned_employee_id
            └─ contract_referrer_id = الوسيط المختار (اختياري)
```

**النتيجة:** خالد (مدير) أنشأ العقد بس ملكيته للفرع. سارة (مشرفة) هي يلي "باعت" — بياخد الملكية الفردية.

---

## 3. القواعد الثابتة (Immutable Rules)

| # | القاعدة | التوضيح |
|---|---------|---------|
| **R-1** | **العقد = تابع للمهمة** | `contracts.source_open_task_id` هو المصدر الوحيد لمعرفة "مين باع" |
| **R-2** | **الملكية الفردية = من assigned_employee_id** | المشرفة/الفني اللي أنهت المهمة هو صاحب الملكية |
| **R-3** | **الأدوار المؤهلة = مشرف + فني فقط** | أي دور آخر (مدير، إداري، telemarketer...) → ملكيته للفرع |
| **R-4** | **created_by = للتدقيق فقط** | من ضغط "إنشاء عقد" — لا يؤثر على الملكية |
| **R-5** | **الإيراد = دائماً للفرع** | `contracts.branch_id` = مصدر الإيراد مهما كان صاحب الملكية |

## 4. جدول الملكية (Ownership Matrix)

| الدور (Role) | إنشاء عقد | الملكية الفردية |
|--------------|-----------|-----------------|
| **مشرف (SUPERVISOR)** | ✅ | ✅ (باسمه) |
| **فني (TECHNICIAN)** | ✅ | ✅ (باسمه) |
| **مدير (MANAGER)** | ✅ | ❌ (للفرع) |
| **موظف آخر** | ✅ (إذا صلاحية) | ❌ (للفرع) |

## 5. الجداول والحقول المطلوبة (Schema Changes)

### 5.1 حقل جديد على `contracts`

| الحقل | النوع | NULL? | الوصف |
|-------|-------|-------|-------|
| `contract_referrer_id` | `INTEGER` | ✅ | معرف الوسيط اللي سبب البيعة (من `clients.referrers`) |
| `contract_referrer_type` | `VARCHAR(30)` | ✅ | نوع الوسيط: `Client` / `Employee` / `Personal` / `Direct` |

### 5.2 جدول جديد: `referrer_rewards` (مستقبلي — جدول الهدايا)

| الحقل | النوع | NULL? | الوصف |
|-------|-------|-------|-------|
| `id` | `SERIAL` PK | ❌ | المعرف |
| `referrer_id` | `INTEGER` | ❌ | FK → referrers(id) |
| `contract_id` | `INTEGER` | ✅ | FK → contracts(id) |
| `reward_type` | `VARCHAR(50)` | ❌ | نوع الهدية: `gift_item` / `cash` / `discount` |
| `reward_value` | `NUMERIC` | ✅ | قيمة الهدية |
| `reward_status` | `VARCHAR(50)` | ❌ | `pending` / `delivered` / `cancelled` |
| `delivered_by` | `INTEGER` | ✅ | FK → hr_users(id) |
| `delivered_at` | `TIMESTAMPTZ` | ✅ | تاريخ التسليم |
| `notes` | `TEXT` | ✅ | ملاحظات |
| `created_at` | `TIMESTAMPTZ` | ✅ | NOW() |

> ⚠️ **ملاحظة:** لا يتم إنشاء `referrer_rewards` حالياً — يُترك للمستقبل عندما تُحدّد آلية الهدايا.
