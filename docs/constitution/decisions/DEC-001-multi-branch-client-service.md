## 1. ملخص المعضلة (Executive Summary)

**النظام الحالي يفرض أن كل زبون "تابع لفرع واحد" (`clients.branch_id`).** هذا يعمل بشكل صحيح في معظم الحالات. لكن واقع العمل في سوق متعدد الفروع (متعدد المحافظات) أثبت وجود سيناريو معقد:

- **الزبون** ممكن يكون مسجل بفرع دمشق، بس يشتري جهاز من فرع حمص.
- **الجهاز** بيُركّب بحمص، وبالتالي الصيانة الدورية والطوارئ لازم تيجي من حمص.
- **العقد** تابع لفرع البيع (حمص)، والذمم المالية تابعة لحمص.
- **الزبون** نفسه ما بيظهر بقائمة زباين حمص — فالفريق الميداني بيخدم جهاز بلا معلومات صاحبه.

هاد بيخلق فجوة تشغيلية أساسية: **الخدمة الميدانية ممكن تكون بفرع مختلف عن تسجيل الزبون، والنظام ما بيدعم هالفصل بالشكل الصحيح.**

### 1.1 لماذا هالمعضلة لم تظهر سابقاً؟

لأن النظام بُني على افتراض أن:
1. الزبون بيشتري من نفس الفرع يلي مسجل فيه.
2. الجهاز بيركّب بنفس عنوان الزبون (نفس المحافظة).
3. كل شي بيصير بفرع واحد.

لكن بالواقع: **الزبون ممكن ينتقل لمحافظة تانية، أو يشتري جهاز كهدية لأهله بمحافظة تانية، أو يكون عنده بيت صيفي بمحافظة تانية.**

## 2. عملية التشخيص (Diagnostic Process)

هاد القسم بيسجّل سلسلة الأسئلة والأجوبة يلي قادتنا لهالقرار.

### السؤال 1: هل عنا زيارة واحدة ولا زيارتين؟

**الإجابة:** النظام المعتمد هو زيارة موحدة واحدة (`field_visits`). فيه جداول قديمة legacy (`visits`, `marketing_visits`) لساتها موجودة بالـ DB بس مش مستخدمة.

```
field_visits (الموحدة)
  ├─ visit_type = 'marketing' أو 'emergency'
  ├─ branch_id = فرع الزيارة
  └─ visit_tasks (المهام داخل الزيارة)
       └─ source_open_task_id ← open_tasks
```

### السؤال 2: هل القالب الموحد للمهام كلها متعلقة بنوع زيارة واحد؟

**الإجابة:** نعم. كل المهام بتُنشأ كـ `open_tasks` بأنواع مختلفة (`device_demo`, `delivery`, `installation`, `emergency_maintenance`...). لما بيصير وقت التنفيذ الميداني: `open_task` → `visit_task` → `field_visit`.

```
open_tasks (المهمة)
  ↓ device_id (Phase 3 — migration 194)
installed_devices (الجهاز)
  ↓ contract_id
contracts (العقد)
  ↓ customer_id
clients (الزبون)
```

### السؤال 3: هل في تفريق بين الزبائن والمقترحين؟

**الإجابة:** نعم. تفريق واضح:
- **المرشح (Candidate)** = مقترح قبل أي شراء → `candidates` → `converted_to_lead_id` → يصير زبون
- **الزبون (Client)** = كيان مالي نشط → `clients` → عقود + أجهزة

### السؤال 4: هل تم الفصل بين العقد والجهاز؟

**الإجابة:** نعم — الفصل اكتمل بتاريخ 2026-05-26 عبر 6 مراحل:

| المرحلة | المهجرة | الوصف |
|---------|---------|-------|
| Phase 2A | 190 | إنشاء `installed_devices` |
| Phase 2B | 192 | قراءات API من `installed_devices` |
| Phase 2C | 193 | كتابات مباشرة على `installed_devices` |
| Phase 3 | 194 | ربط `open_tasks.device_id` |
| Phase 4 | 196 | إنشاء `device_warranties` |
| Phase 5 | 197 | إنشاء `device_installed_parts` |
| Phase 6 | 195 | حذف 13 حقل legacy من `contracts` |

**النتيجة:** `contracts` = الكيان المالي. `installed_devices` = الكيان الفيزيائي. كل جهاز مركب له `branch_id` خاص فيه — قابل للاختلاف عن `clients.branch_id`.

### السؤال 5: شو بيصير لما الزبون يشتري من فرع مختلف؟

**السيناريو اللي كشف المشكلة:**

```
الزبون أحمد (دمشق)
  ├─ clients.branch_id = 1 (دمشق) ← ثابت
  ├─ العقد C-5002: contracts.branch_id = 2 (حمص) ← فرع البيعة
  ├─ الجهاز: installed_devices.branch_id = 2 (حمص) ← فرع الخدمة
  ├─ المهام: open_tasks.branch_id = 2 (حمص) ← من الجهاز
  ├─ الزيارات: field_visits.branch_id = 2 (حمص) ← من المهام
  └─ الذمم: dues.branch_id = 2 (حمص) ← من العقد
```

**المشكلة الحقيقية:**
- خالد (مدير حمص) بيفتح "قائمة زباين حمص" → أحمد **ما بيظهر**
- لأن `GET /api/clients` بيفلتر بس: `WHERE clients.branch_id = 2`
- أحمد `branch_id = 1` → **مخفي**

**الفريق الميداني بحمص بيطلع لعند أحمد:**
- بيعرفوا العنوان من `installed_devices` ✓
- بس ما عندهم رقم تلفون أحمد بالنظام ❌
- ما بيقدرو يفتحوا "بروفايل الزبون" ليشوفوا تاريخه ❌

### السؤال 6: هل المهام والصيانة تقع على الجهاز والذمم تقع على العقد؟

**الإجابة:** نعم — هاد التقسيم الدقيق:

| الشي | وين بيقع | `branch_id` = |
|------|---------|---------------|
| الصيانة الدورية | `open_tasks` | من `installed_devices.branch_id` |
| الصيانة الطارئة | `open_tasks` | من `installed_devices.branch_id` |
| التحصيل | `dues` | من `contracts.branch_id` |
| البيعة الجديدة | `contracts` | فرع البيعة |

**القاعدة:** الخدمة تتبع الجهاز، المال يتبع العقد.

## 5. الخيارات المُدرَسة (Options Evaluated)

### الخيار A: Cross-Branch Lookup (البحث المتقاطع)

**الفكرة:** الزبون يضل "تابع لفرع واحد" بالـ DB. بس المستخدمين المصرّح لهم بيقدرو يبحثوا بكل الفروع.

**التنفيذ:**
- إضافة `?includeCrossBranch=true` على `GET /api/clients`
- صلاحية جديدة: `clients.cross_branch_lookup`
- البحث يصير: `WHERE clients.branch_id = $branchId OR (exists devices/contracts بالفرع)`

**المميزات:**
- بسيط — ما بيغيّر architecture
- الزبون بيظل "تابع لفرع" واحد بالتقارير

**العيوب:**
- كل فرع بيشوف "كل زباين اللي عندهم أجهزة عنده" — ممكن يصير فوضي
- ما بيحل مشكلة "الصيانة الدورية بتتولد بفرع غلط"

---

### الخيار B: الفصل الكامل (Device-First Branching) ⭐ المُختار

**الفكرة:** نفصل مصدر الحقيقة:
- **الزبون** (`clients`) → `branch_id` = فرع التسجيل (ثابت)
- **الخدمة** (`installed_devices`, `open_tasks`, `field_visits`) → `branch_id` = فرع الخدمة (قابل للتغيير)
- **المال** (`contracts`, `dues`) → `branch_id` = فرع البيعة

**القاعدة الذهبية:**
> كل ما هو "خدمة ميدانية" (صيانة، طوارئ، زيارة، تحصيل) — فرعه من الجهاز.  
> كل ما هو "إداري/مالي/تسويقي" — فرعه من العقد أو الزبون.

**التنفيذ التقني:**

```
GET /api/clients?branchId=2 (حمص)
  ↓
WHERE c.branch_id = 2                    ← زباين مسجلين بحمص
   OR EXISTS (
     SELECT 1 FROM installed_devices d
     WHERE d.customer_id = c.id
       AND d.branch_id = 2              ← + زباين عندهم أجهزة بحمص
   )
   OR EXISTS (
     SELECT 1 FROM contracts ct
     WHERE ct.customer_id = c.id
       AND ct.branch_id = 2             ← + زباين عندهم عقود بحمص
   )
```

**الصلاحيات:**

| العملية | على مين؟ | الصلاحية | النطاق |
|---------|---------|----------|--------|
| عرض قائمة الزباين | `clients` | `clients.view_list` | `BRANCH` + استثناء الجهاز/العقد |
| عرض بروفايل زبون | `clients` | `clients.view` | `BRANCH` أو "له جهاز/عقد بالفرع" |
| تعديل بيانات زبون | `clients` | `clients.edit` | `BRANCH` فقط (فرع التسجيل) |
| عرض مهام الجهاز | `open_tasks` | `open_tasks.view` | `BRANCH` (من `device_id`) |
| فتح بلاغ طوارئ | `open_tasks` | `open_tasks.edit` | `BRANCH` (من `device_id`) |

**المميزات:**
- ✅ فريق الخدمة بيشوف الزبون يلي بيخدم جهازه
- ✅ الزبون يضل "تابع لفرع التسجيل" بالتقارير الإدارية
- ✅ لا إعادة هيكلة كبيرة
- ✅ الصيانة الدورية بتتولد من `installed_devices.branch_id` (صح)

**العيوب:**
- ⚠️ فرع حمص بيشوف "اسم ورقم" زبون دمشق (read-only)
- ⚠️ محتاج تعديل على 4-5 routes بالـ API

### ما هو "فرع الخدمة"؟ (Definition of Service Branch)

**فرع الخدمة = الفرع اللي بيقدم الخدمات الميدانية للجهاز بشكل فعلي.**

```
الزبون أحمد (مسجل بدمشق)
    │
    ├─── فرع التسجيل = دمشق (ثابت دائماً)
    │
    └─── الجهاز المركب بحمص
              │
              └─── فرع الخدمة = حمص (الفريق الميداني بحمص بيصلحوه)
              │
              ├─── الصيانة الدورية ← من فريق حمص
              ├─── الصيانة الطارئة ← من فريق حمص
              ├─── التوصيل ← من فريق حمص
              ├─── التركيب ← من فريق حمص
              └─── الطوارئ ← من فريق حمص
```

**يعني:** فرع الخدمة هو الفرع اللي بيطلع عند أحمد لما:
- الجهاز بيعطل (طوارئ)
- بدّو صيانة دورية (كل 3 شهور)
- بدّو يحصّل دفعة
- بدّو يغيّر قطعة

**ما بيعني:** فرع الخدمة مش بالضرورة فرع البيعة. يمكن أحمد اشترى الجهاز بحمص (فرع البيعة = حمص)، بس بعد سنة راح يسكن بدمشق ونقل الجهاز معو. هون **فرع الخدمة بيصير دمشق** (لأن الجهاز هلا بدمشق)، بس فرع التسجيل بيضل حمص (أو دمشق حسب أول تسجيل)، وفرع البيعة بيضل حمص (العقد ما بيتغيّر).

---

### هل فرع الخدمة بيحسم كل مهام الجهاز؟

**الإجابة:** نعم — بس مع استثناء المهام التسويقية قبل البيع.

```
الجهاز المركب
    │
    ├─── مهام قبل البيع (تسويقية) ← فرع البيعة
    │     └─ device_demo ← فريق المبيعات بيعملوها
    │     └─ gift_delivery ← فريق المبيعات بيعملوها
    │     └─ device_purchase ← فريق المبيعات بيعملوها
    │
    ├─── مهام بعد البيع (ميدانية) ← فرع الخدمة
    │     ├─ device_delivery ← فريق الخدمة
    │     ├─ device_installation ← فريق الخدمة
    │     ├─ device_activation ← فريق الخدمة
    │     ├─ emergency_maintenance ← فريق الخدمة
    │     └─ periodic_maintenance ← فريق الخدمة
    │
    └─── مهام مالية (تحصيل) ← فرع العقد
          └─ dues_collection ← فريق التحصيل التابعة للعقد
```

### جدول المهام وفروعها

| نوع المهمة | عائلة المهمة | فرع التنفيذ | اللي بيحدده |
|------------|-----------|------------|------------|
| `device_demo` | `marketing` | فرع البيعة | الزبون ما عندو جهاز بعد |
| `device_delivery` | `delivery` | فرع الخدمة | من `installed_devices.branch_id` |
| `device_installation` | `service` | فرع الخدمة | من `installed_devices.branch_id` |
| `device_activation` | `service` | فرع الخدمة | من `installed_devices.branch_id` |
| `emergency_maintenance` | `emergency` | فرع الخدمة | من `installed_devices.branch_id` |
| `periodic_maintenance` | `maintenance` | فرع الخدمة | من `installed_devices.branch_id` |
| `dues_collection` | `collection` | فرع العقد | من `contracts.branch_id` |

### القاعدة المرجعية

> **فرع الخدمة بيحسم كل المهام الميدانية المتعلقة بالجهاز بعد البيع.**  
> **فرع البيعة بيحسم المهام التسويقية (قبل البيع) والمهام المالية (التحصيل).**

### مثال عملي

```
الزبون عمر مسجل بدمشق، شاف جهاز بحمص:

1. مهمة device_demo ← فريق المبيعات بحمص بيعملوها
   (الزبون مش مستهدك بعد، في طور تسويقي)

2. مهمة device_delivery ← فريق الخدمة بحمص بيعملوها
   (الجهاز بيتوصل من فريق حمص)

3. مهمة emergency_maintenance ← فريق الخدمة بحمص بيعملوها
   (الجهاز بحمص وبيصلح بحمص)

4. مهمة dues_collection ← فريق التحصيل بحمص بيعملوها
   (الذمة مالية تابعة للعقد بحمص)
```

### هل فرع الخدمة وفرع المالية هن نفس الشي؟

**الإجابة:** بالواقع العملية — **نعم، بس مش دائماً.**

```
حالة 1: الزبون بيشتري وبيركّب بنفس الفرع (المعظم)
┌─────────────────────────────────────────────────────────────────┐
│  فرع حمص                                               │
│  ├─── العقد: C-5002 ← فرع البيعة = حمص           │
│  ├─── الجهاز: installed_devices.branch_id = حمص   │
│  └─── الذمم: dues.branch_id = حمص                 │
│                                                       │
│  نتيجة: فرع البيعة = فرع الخدمة = فرع المالية ✓  │
└─────────────────────────────────────────────────────────────────┘

حالة 2: الزبون بيشتري من حمص وبينتقل الجهاز لدمشق (النادر)
┌─────────────────────────────────────────────────────────────────┐
│  فرع حمص (البيع) + فرع دمشق (الخدمة)                    │
│                                                       │
│  ├─── العقد: C-5002 ← فرع البيعة = حمص           │
│  │     └─ الذمم: dues.branch_id = حمص           │
│  │                                                     │
│  └─── الجهاز: installed_devices.branch_id = دمشق   │
│        └─ الصيانة الدورية ← فريق دمشق      │
│        └─ الطوارئ ← فريق دمشق              │
│                                                       │
│  نتيجة: فرع البيعة (حمص) ≠ فرع الخدمة (دمشق)  ❌  │
└─────────────────────────────────────────────────────────────────┘
```

### قاعدة الفرق

> **فرع البيعة وفرع الخدمة بيكونوا نفس الشي بالواقع العملية (المعظم).**  
> **لكن بالنظام نحن مفصلينهم بمفهوم مختلفة — لأن الجهاز قابل للانتقال بين الفروع، بس العقد ما بيتغيّر.**

### متى بيتغير فرع الخدمة؟

| الحالة | فرع الخدمة يصير | السبب |
|---------|------------------|-------|
| بيع جديد بحمص | حمص | الجهاز بيُركّب بحمص |
| الزبون بينتقل لدمشق | من حمص → دمشق | الجهاز انتقل مع الزبون |
| الزبون بيطلب صيانة من دمشق | حمص (لأن الجهاز فيها) | الصيانة بتتولد من الجهاز |
| الزبون بيطلب صيانة من أي فرع | حمص | المهمة بتتولد لفرع الجهاز |

### التعديلات المطلوبة

#### 1. `routes/clients.ts` — GET /

```sql
-- قبل:
WHERE c.branch_id = $branchId

-- بعد:
WHERE c.branch_id = $branchId
   OR EXISTS (
     SELECT 1 FROM installed_devices d
     WHERE d.customer_id = c.id AND d.branch_id = $branchId
   )
   OR EXISTS (
     SELECT 1 FROM contracts ct
     WHERE ct.customer_id = c.id AND ct.branch_id = $branchId
   )
```

#### 2. `routes/clients.ts` — GET /:id (authorize)

```typescript
// قبل:
if (client.branch_id !== userBranchId) return 403;

// بعد:
if (client.branch_id !== userBranchId) {
  const hasDeviceOrContract = await pool.query(
    `SELECT 1 FROM installed_devices WHERE customer_id = $1 AND branch_id = $2
     UNION
     SELECT 1 FROM contracts WHERE customer_id = $1 AND branch_id = $2
     LIMIT 1`,
    [clientId, userBranchId]
  );
  if (hasDeviceOrContract.rows.length === 0) return 403;
}
```

#### 3. `routes/openTasks.ts` — توليد المهام الدورية

```sql
-- قبل (غلط):
INSERT INTO open_tasks (branch_id, ...)
SELECT c.branch_id, ... FROM clients c WHERE ...

-- بعد (صح):
INSERT INTO open_tasks (branch_id, device_id, ...)
SELECT d.branch_id, d.id, ...
FROM installed_devices d
WHERE d.status = 'active'
  AND d.contract_warranty_end_date > NOW()
  -- جدولة حسب warranty_visits
```

#### 4. frontend — Client List

إضافة عمود جديد بالـ UI: **"فرع التسجيل"** — لما مدير حمص يشوف أحمد، يعرف أنه "زبون دمشق عنده جهاز عندنا".
