# دستور الكيان: التقارير والمؤشرات (Reporting & Analytics Domain Constitution)

> **الحالة (Status):** Active Draft / Architectural Blueprint
> **التاريخ:** 2026-07-01
> **المرجع الأعلى لطبقة التقارير و KPIs والفلاتر في النظام.** يُعرّف البنية الموحّدة التي تُبنى عليها كل لوحات المؤشرات والتقارير القابلة للتصدير وفلاتر الجداول عبر كل أقسام Golden CRM.

## 0. التوصية المعمارية الجوهرية (The Spine)

**لا تُبنى ثلاث طبقات منفصلة (تقارير + KPIs + فلاتر). تُبنى طبقة تحليلات واحدة لها ثلاث واجهات استهلاك تتشارك «عقد نطاق» (Scope Contract) واحدًا.**

**السبب:** المتطلبات الثلاثة تقرأ نفس البيانات بنفس قواعد التقييد (زمن + فرع + إسناد). المنظومة تملك أصلًا محرك التقييد المطلوب — نطاقات الصلاحيات `GLOBAL / BRANCH / ASSIGNED`، و`X-Branch-Id`، وخطّاف `useBranchListScope`، وسياسات `authorize()`. بناء ثلاث طبقات منفصلة يكرّر منطق التقييد ثلاث مرّات ويخاطر بتسريب بيانات عبر الفروع. الطبقة الموحّدة تعيد استخدام الموجود وتضمن أن **بطاقة KPI والتقرير وجدول القائمة يُظهرون نفس الرقم دائمًا**.

```
                    ┌─────────────────────────────────────────┐
                    │   عقد النطاق الموحّد (Scope Contract)      │
                    │   الزمن + النطاق التنظيمي + فلاتر القسم     │
                    └─────────────────────────────────────────┘
                                      │
              ┌───────────────────────┼───────────────────────┐
              ▼                       ▼                       ▼
        ┌──────────┐           ┌──────────┐            ┌──────────┐
        │ KPI Cards│           │ التقارير  │            │  فلاتر   │
        │ (داشبورد)│           │ (تصدير)   │            │ الجداول  │
        └──────────┘           └──────────┘            └──────────┘
              └───────────────────────┼───────────────────────┘
                                      ▼
                    مصادر الحقائق المُقيّدة (Scoped Fact Sources)
       financial_movements · field_visits/visit_tasks · contact_targets
              contracts · open_tasks · installed_devices · candidates
```

## 1. الأساس المشترك (Cross-Cutting Foundation) — يُبنى مرّة واحدة

### 1.1 عقد النطاق (Scope Contract)

بُعدان إلزاميان + فلاتر اختيارية خاصة بكل قسم:

| البُعد | القيم | يُترجم إلى | يعيد استخدام |
|---|---|---|---|
| **الزمن (Time)** | اليوم / الأسبوع / الشهر / الربع / مدى مخصّص + مقارنة بالفترة السابقة (Δ) | `... BETWEEN :from AND :to` على عمود زمن المجال | — |
| **النطاق التنظيمي (Org Scope)** | (أ) كل الفروع ومقارنتها · (ب) فرع واحد · (ج) سجلاتي المُسندة | (أ)=`GLOBAL` · (ب)=`BRANCH` عبر `X-Branch-Id`/`BranchSwitcher` · (ج)=`ASSIGNED` عبر `useBranchListScope` | مستويات الصلاحية الثلاثة الموجودة حرفيًا |
| **فلاتر القسم (Domain Filters)** | تختلف لكل قسم (الحالة، النوع، الموظف، الموديل…) | شروط `WHERE` إضافية | تعريفات لكل قسم في §2 |

> **مبدأ:** البُعد التنظيمي يطابق نطاقات الصلاحية الثلاثة، فيُغذّي مفتاح النطاق نفسه الذي يُقيّد القوائم اليوم. أي مستخدم لا يستطيع رؤية مؤشر/تقرير على نطاق لا يملك صلاحية عرضه عليه — التقييد يُفرض في طبقة `authorize()`، لا في الواجهة.

### 1.2 مصادر الحقائق القانونية (Single Source per Metric)

لكل مؤشر **مصدر حقيقة واحد** يمنع تضارب الأرقام:

| المجال | جدول الحقيقة | عمود الزمن | عمود الفرع | ملاحظات |
|---|---|---|---|---|
| المال/التحصيل | `financial_movements` | `occurred_at` | `occurred_branch_id` | append-only؛ `kind ∈ charge/payment/refund/discount`؛ الرصيد = Σ(charge+refund)−Σ(payment+discount) |
| الزيارات | `field_visits` + `visit_tasks` | `scheduled_date` | `branch_id` | state machine: `scheduled→in_progress→ended→closed` |
| التسويق الهاتفي | `contact_targets` + سجلات الاتصال | `created_at`/الإسناد | عبر الزبون/الفرع | دورة الهدف: target→appointment→visit |
| العقود/المبيعات | `contracts` | `created_at` | `branch_id` / `service_branch_id` | `status`, `final_price`, `sale_subtype='definitive'` |
| الأجهزة المركّبة | `installed_devices` + الكفالات | حسب الحدث | فرع الخدمة | الصيانة الدورية + انتهاء الكفالة |
| الزبائن | `clients` + `client_rating_history` | `created_at` / `changed_at` | `branch_id` | الملكية عبر `client_assignments` (M2M) |
| الأسماء المقترحة | `candidates` | `created_at` | عبر الإحالة/الفرع | `status` ∈ New/Suggested/FollowUp/Contacted/Qualified/Junk |
| التوظيف | الشواغر/الطلبات/المقابلات | `created_at` | — | قمع التوظيف |

### 1.3 الطبقات التقنية (Reuse-first)

- **الباك-إند:** خدمة `metricsService` + نمط Endpoint موحّد:
  `GET /api/reports/:domain/:metric?from&to&scope&branchId&<domainFilters>`
  يمرّ عبر `authorize()` نفسها فيُفرَض النطاق تلقائيًا (لا تقييد يدوي مكرّر). كل metric دالة نقية (SQL aggregation) تقرأ من مصدر الحقيقة في §1.2.
- **الفرونت-إند — 4 مكوّنات مشتركة:**
  1. `<ScopeFilterBar>` — الزمن + النطاق التنظيمي (موحّد لكل صفحة تقارير/داشبورد).
  2. `<KpiCard>` — رقم + دلتا مقابل الفترة السابقة + سبارك-لاين. **يحلّ محل بطاقات `Dashboard.tsx` الحالية ذات الدلتا الوهمية (`+12%` ثابتة).**
  3. `<ReportView>` — جدول مُجمّع + تصدير Excel/CSV + رسم بياني.
  4. توسعة `SmartTable` بفلاتر عمودية قابلة للحفظ (Saved Views).

## 2. مصفوفة الأقسام (تقارير · KPIs · فلاتر)

> الترتيب حسب القيمة التشغيلية المتوقّعة. كل قسم يرث عقد النطاق في §1.1؛ الجدول أدناه يضيف ما يخصّه فقط.

### أ) التسويق الهاتفي والمواعيد (Telemarketing & Appointments)
- **تقارير:** إنتاجية المتصل (محاولات/تحويلات/مواعيد) · مسار جهات الاتصال (target→appointment→visit→contract) · أسباب الفشل · معدّل الالتزام بالموعد.
- **KPIs:** عدد المحاولات · معدّل التحويل لموعد % · مواعيد محقّقة/ملغاة · زمن البقاء في كل حالة.
- **فلاتر الجدول:** المتصل المسند · حالة الهدف · النتيجة · الفترة · الفرع · المنطقة.

### ب) التخطيط وعمل الفرع (Planning)
- **تقارير:** تغطية النطاقات (مخطّط/منفّذ) · توزيع المسارات على الفرق · فجوة الخطة مقابل التنفيذ.
- **KPIs:** نسبة تغطية المناطق % · مسارات بلا تعيين · أحياء غير مغطّاة · حمولة الفريق اليومية.
- **فلاتر:** المنطقة/خط السير · الفريق · اليوم · حالة الجدولة.

### ج) الزيارات الميدانية (Field Visits)
- **تقارير:** إنجاز الزيارات حسب الحالة · زمن الدورة · زيارات GPS خارج النطاق · نتائج الزيارة.
- **KPIs:** زيارات اليوم المكتملة % · متوسط مدة الزيارة · زيارات متأخرة · معدّل النتائج الناجحة.
- **فلاتر:** الحالة · الفريق/الفني المسند · النوع (تسويقية/طارئة/دورية) · المصدر (`field_initiated`/مخطّطة) · الفرع.

### د) العمليات والمهام (Operations & Tasks)
- **تقارير:** إنتاجية المهام حسب النوع · مهام متأخرة/SLA · قطع غيار مستهلكة · أسباب نتائج المهام (قوائم migr 345–349).
- **KPIs:** مهام مفتوحة/مكتملة لكل نوع · متوسط زمن الإغلاق · نسبة الإنجاز ضمن الموعد · مهام تحتاج متابعة.
- **فلاتر:** نوع المهمة · الحالة · المسند · الأولوية/الامتثال · الفرع · المدى الزمني.

### هـ) العقود والمبيعات (Contracts & Sales)
- **تقارير:** مبيعات حسب الفرع/البائع/نوع البيع · قمع البيع · عقود ملغاة وأسبابها (migr 339) · هدايا موعودة (migr 344).
- **KPIs:** عدد العقود · قيمة المبيعات (Σ`final_price`) · متوسط قيمة العقد · معدّل الإلغاء % · عقود مسوّدة عالقة.
- **فلاتر:** الحالة · نوع البيع · البائع/مالك البيع · الفرع · الفترة.

### و) المالية والتحصيل (Finance & Collections) — على `financial_movements`
- **تقارير:** أعمار الذمم (Aging) · محصّل مقابل مستحق · كشف حساب الزبون · تدفّق نقدي حسب `source_type`.
- **KPIs:** إجمالي المستحق (Σcharge−Σpayment) · المحصّل خلال الفترة · نسبة التحصيل % · ذمم متأخرة · مبالغ مرتجعة.
- **فلاتر:** نوع الحركة · المصدر · الفرع (`occurred_branch_id`) · الفترة.

### ز) الأجهزة المركّبة والصيانة (Installed Devices)
- **تقارير:** أعمار الأجهزة وكفالاتها · أجهزة مستحقة صيانة دورية · حركة الجهاز (ledger) · معدّل الأعطال حسب الموديل.
- **KPIs:** أجهزة مركّبة فعّالة · كفالات تنتهي قريبًا · أجهزة متأخرة عن الصيانة · أجهزة مفكوكة.
- **فلاتر:** الموديل · حالة الجهاز · حالة الكفالة · الفرع · المنطقة.

### ح) السجلات/الزبائن والأسماء المقترحة (CRM) — **قسم التعمّق، انظر §3**

### ط) التوظيف والموظفون (HR & Workforce)
- **تقارير:** قمع التوظيف (شاغر→طلب→مقابلة→قبول) · أداء الفرق (إنجاز/زيارات/مبيعات لكل موظف).
- **KPIs:** شواغر مفتوحة · طلبات قيد المعالجة · معدّل القبول · حمولة/إنتاجية الموظف.
- **فلاتر:** الشاغر · الحالة · القسم · الفرع.

## 3. التعمّق الأول: الزبائن والأسماء المقترحة (CRM Deep-Dive)

> نقطة البدء المعتمدة بعد الأساس (P0). مبنية على حقول `clients`, `candidates`, `client_rating_history`, `client_assignments`, `referral_sheets`, `direct_suggestions` الفعلية. كل صف widget أدناه يحقّق عقد §8.2 كاملًا (سؤال/صيغة/مصدر واحد/حبيبة/صلاحية/غاية) ولا يدخل الكتالوج إلا ما اجتاز اختبار §8.3 الثلاثي: غير مكرّر + يخدم قرارًا أو سير عمل أو إنجاز فريق + مصدره واحد.
>
> **ملاحظة تسمية:** الشاشة التي يسمّيها المستخدمون «سجلات الأسماء المقترحة» في الشريط الجانبي (`/candidates`) هي جدول `candidates` نفسه. أي widget هنا بعنوان «الأسماء المقترحة» يقصد هذا الجدول.

### 3.3 فلاتر وحقل بحث جدول الزبائن (`/clients`)

**حقل البحث الحر:** نص واحد يغطي `name`/`first_name`/`last_name`/`nickname` + `mobile` + أرقام `contacts` الإضافية — بحث متعدّد الأعمدة بحقل واحد.

**الفلاتر البنيوية (١٤ فلترًا).** «شرط الظهور» = «عام» يعني: يظهر لأي مستخدم يملك صلاحية القائمة الأساسية (`clients.view_list`) بلا شرط إضافي، وخياراته (إن كانت قائمة موظفين/فئات مُشتقّة) تُبنى ديناميكيًا من نطاق البيانات المرئي فقط — لا Lookup شامل خارج النطاق.

### 3.6 تقرير مركّب: القمع الكامل من ترشيح إلى عقد (Cross-Entity Funnel Report)

تقرير واحد (لا widget، بل `<ReportView>` من §1.3) يربط ثلاث مراحل بمصدر واحد لكل مرحلة، دون ازدواج حساب:
`referral_sheets/direct_suggestions (توليد الاسم) → candidates.status (التأهيل) → clients.is_candidate + converted_to_lead_id (التحويل) → contracts (البيع الفعلي، §2.هـ)`.
الغاية: قرار استراتيجي واحد — أين نخسر الفرص فعليًا: عند التوليد، أم التأهيل، أم الإغلاق؟ يُبنى بعد استقرار Tier 1 من §3.2/§3.4.

### 3.7 التوزيع الجغرافي وخط السير (Geo & Route Rollup) — يشمل الزبائن والمرشّحين معًا

> بُعد أفقي يُطبَّق على القسمين معًا، لا widget بذاته بل **تقرير `<ReportView>` واحد قابل للتبديل** بين «الزبائن» و«الأسماء المقترحة» بنفس شكل الجدول.

| # | التقرير | الصيغة | الحبيبة | الغاية | نوع الرسم |
|---|---|---|---|---|---|
| 1 | `clients.by_route` | `COUNT(*) GROUP BY route_id` عبر `route_points` (مطابقة `neighborhood::int = route_points.geo_unit_id` حيث `level=4`) | لكل خط سير × نطاق | قرار: أي خطوط السير تغطّي أعلى كثافة زبائن فعليًا | Bar أفقي مرتّب + جدول تفصيلي |
| 2 | `candidates.by_route` | `COUNT(*) GROUP BY route_id` عبر `route_points.geo_unit_id = candidates.geo_unit_id`، مع `conversion_rate` لكل خط | لكل خط سير × نطاق | قرار: أي خطوط السير تُنتج مرشّحين قابلين للتحويل فعليًا، لا فقط عددًا | Bar مزدوج (عدد + معدّل تحويل) |
| 3 | `clients.by_geo_area` / `candidates.by_geo_area` | `COUNT(*) GROUP BY governorate/district/neighborhood` (زبائن) أو `geo_unit_id` (مرشّحون) | لكل مستوى جغرافي × نطاق | قرار: تخطيط تغطية على مستوى أدق من خط السير (حي/منطقة) | Bar أفقي مرتّب (Top N)، ويطابق `clients.top_geo_areas` #8 §3.2 |

> **لماذا خط السير مهم بالتحديد:** خط السير هو وحدة التخطيط الميداني الفعلية (يُسنَد لفريق، له جدول زيارات). معرفة كثافة الزبائن/المرشّحين *لكل خط سير* — لا فقط لكل حي — تربط مؤشرات CRM مباشرة بقرار «أين نرسل الفريق التالي» (يتقاطع مع دومين Planning §2.ب).

### 3.8 أداء الملكية حسب الموظف/المشرف (Ownership Performance by Employee)

> يجيب مباشرة عن: «ماذا يملك ويحقّق كل موظف من السجلات المسندة إليه، ضمن ما تسمح به صلاحيته؟» — مختلف عن `referral_sheets.team_quality_leaderboard` (#8 §3.4) الذي يقيس *جودة أوراق الإحالة* تحديدًا، لا الملكية الخام.

| # | Widget key | العنوان | الصيغة | الحبيبة | الغاية | نوع الرسم | الأولوية |
|---|---|---|---|---|---|---|---|
| 1 | `clients.ownership_breakdown` | الزبائن المملوكون لكل موظف | `COUNT(*) GROUP BY client_assignments.hr_user_id` + نسبة `Committed` لكل موظف | لكل نطاق × موظف | إنجاز فريق: حجم وجودة محفظة كل موظف مسؤول عنها | Bar أفقي مرتّب (Leaderboard) | ⏳ Tier 1 |
| 2 | `candidates.ownership_breakdown` | المرشّحون المملوكون لكل موظف | `COUNT(*)`, `COUNT(converted_to_lead_id NOT NULL)` `GROUP BY owner_user_id` ضمن الفترة | لكل فترة/نطاق × موظف | إنجاز فريق: مَن يحوّل مرشّحيه فعليًا لا فقط يجمعهم | Bar مزدوج (عدد + معدّل تحويل) | ⏳ Tier 1 |

> **ملاحظة نطاق:** هذان التقريران مفيدان فعليًا لصاحب اتساع `BRANCH`/`GLOBAL` (مشرف/مدير يقارن بين أعضاء فريقه). لصاحب اتساع `ASSIGNED` تنهار المجموعة تلقائيًا لصف واحد (نفسه) بحكم تقاطع §8.1 — سلوك آمن ومتوقَّع، لا استثناء أمني، لكنه يُفرِغ الـ widget من قيمته كـ«مقارنة»؛ لذلك الأنسب تصنيفه بصريًا كبطاقة «أدائي الشخصي» عند عرضه لمستخدم `ASSIGNED`.

### 3.9 دليل اختيار نوع الرسم البياني (Chart-Type Decision Guide)

| طبيعة المؤشر | نوع الرسم الموصى به | مثال من أعلاه |
|---|---|---|
| رقم واحد + اتجاهه عبر الزمن | رقم + Sparkline (ضمن `<KpiCard>`) | زبائن جدد، معدّل التحويل |
| توزيع مراحل متتالية (قمع) | **Funnel Chart** | `candidates.stage_funnel` |
| حصص فئات قليلة (٢–٤) من كل | Donut/Pie | نسبة الالتزام، التصنيف LEAD/OP/FOP، نوع المُحيل/الوسيط |
| ترتيب فئات/أفراد بعدد كبير نسبيًا | Bar أفقي مرتّب تنازليًا (Top N) | القنوات، المناطق، خطوط السير، Leaderboard الفرق/الموظفين |
| مقارنة عدد مقابل معدّل لكل فئة | Bar مزدوج (محورين) | `candidates.by_route`، `candidates.ownership_breakdown` |

| مؤشر لحظي بلا مقارنة زمنية ذات معنى | رقم مجرّد بلا دلتا | إجمالي المحفظة الفعّالة |
| حاجة تدقيق/متابعة سجلات فردية | جدول تفصيلي (`<ReportView>`) خلف الرقم | زبائن بلا نشاط، أوراق دون الهدف |

### 3.10 ثغرات وقيود تؤثّر على دقّة المؤشرات

- **تضارب مسمّى حالة المرشّح** (`New` في الداتابيز مقابل `Prospect` في طبقة التطبيق) — يجب توحيده قبل بناء قمع التحويل (#2 §3.4) وإلا اختلّت العدّة (راجع `candidates.md §9`).
- **غياب ترانزأكشن تحويل المرشّح لزبون** — قد يُنشأ زبون دون تحديث `converted_to_lead_id`، فيُحتسب التحويل ناقصًا (#3 §3.4). يُعالج كشرط دقّة قبل اعتماد KPI «معدّل التحويل».
- **غياب سجل تاريخي لحالة المرشّح** (`candidate_status_history`) — يمنع حساب «متوسط زمن البقاء بكل مرحلة» (آخر صف §3.4) بدقّة؛ `client_rating_history` (migr 343) يقدّم نموذجًا جاهزًا للتكرار على `candidates`.
- **`direct_suggestions` غير موحّد مع `candidates`** — مسار منفصل (راجع مهمة الدستور المعلّقة «بيانات الأسماء المقترحة ولوائح الأسماء» في `INDEX.md`)؛ لذلك قمعه (#10 §3.4) يُحتسب مستقلًا لا كجزء من قمع `candidates.stage_funnel` حتى تُحسم مهمة التوحيد.
- **تعريف «النشاط» لمؤشر #9 §3.2 (`clients.inactive_count`) مفتوح** — أي نافذة زمنية وأي أحداث (زيارة/مهمة/عقد/اتصال) تُحتسب؟ محجوب حتى يُحسم (قرار §5.1).
- **مستوى `candidates.geo_unit_id` غير مؤكَّد** — لا عمود `level` مرافق (خلافًا لـ `route_points`)؛ افتراضنا أنه يُلتقَط عادة على مستوى الحي (`level=4`) مثل خطوط السير، لكن يلزم تحقّق فعلي على بيانات الإنتاج قبل اعتماد `candidates.by_route` (§3.7 #2).
- **`clients.governorate/district/neighborhood` مخزّنة كـ `VARCHAR`** لا `INTEGER` (خلافًا لـ `geo_units.id` و`route_points.geo_unit_id`) — أي join جغرافي (`by_route`, `by_geo_area`) يتطلّب `cast` صريحًا؛ نفس الملاحظة الموثّقة أصلًا في `clients.md`.
- **`occupation` حقل نصّي حرّ لا قائمة نظام (system list)** — مؤشر `clients.top_occupations` (#11 §3.2) سيُظهر تشتّتًا (نفس المهنة بصياغات مختلفة) ما لم تُطبَّع القيم أو تُحوَّل مستقبلًا لقائمة إدارية موحّدة.

## 6. الداشبورد العام القابل للتخصيص (Customizable Widget Dashboard)

### 6.1 المبدأ — قشرة Widgets رقيقة فوق طبقة المؤشرات
الداشبورد ليس مكوّنًا جديدًا، بل **طبقة تخطيط نحيفة** فوق مصفوفة §2. كل widget = مؤشر/تقرير من المصفوفة يُرسم داخل `<KpiCard>`/`<ReportView>` ويرث `<ScopeFilterBar>`. الخطر الذي نتجنّبه: أن يتضخّم إطار التخصيص (سحب/تحجيم/شبكات) ويبتلع الوقت قبل وجود محتوى — لذلك نبدأ نحيفًا.

### 6.2 عمق التخصيص (MVP المعتمد)
- إضافة/حذف widgets من كتالوج + **إعادة ترتيب بالسحب (drag-reorder)** عبر `@dnd-kit` (أخفّ من `react-grid-layout`).
- **محفوظ لكل مستخدم**، مع **تخطيط افتراضي مشتقّ من الدور** (لا داشبورد فارغ للمستخدم الجديد).
- **مؤجّل بوضوح:** التحجيم الحرّ والشبكة المرنة (resize grid) — يُضاف لاحقًا دون كسر النموذج.

### 6.3 نموذج البيانات
عمود `user_dashboard_layout JSONB` على تفضيلات المستخدم (لا جدول صف-لكل-widget — غير ضروري للترتيب/الإضافة):
```json
[ { "key": "crm.new_clients",      "size": "sm", "scope": null },
  { "key": "crm.conversion_rate",  "size": "sm", "scope": null },
  { "key": "crm.candidate_funnel", "size": "lg", "scope": { "branchId": 3 } } ]
```
`scope: null` = يتبع شريط النطاق العام؛ كائن = نطاق مثبّت (Pin) لهذا الـ widget وحده.

### 6.4 كتالوج الـ Widgets (Registry — فرونت)
مصفوفة مركزية واحدة، كل عنصر:
```ts
{ key, titleAr, permission, department, defaultSize, component }
```
- الكتالوج يُفلتر بالصلاحية (`hasPermission(permission)`): لا يرى المستخدم — حتى عند التخصيص الذاتي — إلا widgets يملك صلاحية عرض مصدرها.
- دفاع بطبقتين: الـ endpoint يفرض النطاق ثانيةً عبر `authorize()`، فلا تسريب بيانات عبر تخصيص يدوي.

### 6.5 المكوّنات
| المكوّن | الدور |
|---|---|
| `<DashboardPage>` | تحميل التخطيط، رسم الشبكة، شريط النطاق، زر «+ إضافة widget» |
| `<ScopeFilterBar>` | زمن/فرع/مُسند — يطبّق على كل الـ widgets |
| `<WidgetFrame>` | إطار: عنوان + مقبض سحب + حذف + (Pin نطاق) + «آخر تحديث» + زر تحديث |
| `<WidgetCatalogModal>` | الـ widgets المسموحة مجمّعة حسب القسم |
| `<KpiCard>` / `<ReportView>` | محتوى الـ widget الفعلي |

### 6.6 التخطيط الافتراضي لكل دور
خريطة `role → [widget_keys]` تُستخدم عند غياب تخطيط محفوظ، ثم يُسمح للمستخدم بالتعديل والحفظ.

### 6.7 خطوات البناء (P0 + أول widgets CRM)
1. Migration: `user_dashboard_layout JSONB` + جدول الكاش (§7.2).
2. Endpoints: `GET/PUT /api/me/dashboard-layout` + `GET /api/reports/:domain/:metric` + `POST .../refresh`.
3. مكوّنات الأساس: `ScopeFilterBar` + `KpiCard` + `WidgetFrame` + هيكل الـ registry.
4. `DashboardPage` بترتيب dnd + كتالوج + حفظ.
5. أول 3–4 widgets CRM موصولة بـ endpoints حقيقية (زبائن جدد · معدّل التحويل · نسبة الملتزمين · مؤهّلون لم يُحوّلوا) — **تستبدل الدلتا الوهمية في `Dashboard.tsx`**.

## 7. معيار جلب البيانات والتحديث (Data Fetch & Caching Standard) — **إلزامي**

> هدف هذا المعيار: ألّا يضغط الداشبورد على الداتابيز/السيرفر بجلب بيانات كبيرة باستمرار. **البيانات تُجلب فقط للـ widgets المستخدمة فعليًا، وتُخدم من كاش يُحدَّث كل فترة يحددها الأدمن أو يدويًا.**

### 7.1 الجلب عند الطلب فقط (On-Demand Only)
- لا تُحسب مؤشرات أي widget **غير مُضاف** إلى تخطيط مستخدم. الواجهة تطلب فقط widgets التخطيط الفعّال (ويفضَّل الظاهرة على الشاشة).
- لا precompute شامل لكل المؤشرات؛ الحساب مدفوع بالاستخدام الحقيقي.

### 7.2 كاش المؤشرات على الخادم (Server-Side Metric Cache)
جدول `metric_cache`:
```
metric_key · scope_signature · value JSONB · computed_at · computed_by
```
- `scope_signature` = بصمة مُطبَّعة من (المؤشر + الزمن + الفرع/الإسناد) لتمييز نسخ الكاش.
- طلب الـ widget يُخدم من الكاش مباشرةً دون لمس جداول الحقيقة.

### 7.3 فترة تحديث يحددها الأدمن + تحديث يدوي
- إعداد `dashboard_metric_refresh_hours` (كل X ساعة) عبر `/api/system-settings` (يعيد استخدام صلاحية `settings.manage`).
- منطق الخدمة: إذا `now − computed_at < interval` → يُعاد الكاش؛ غير ذلك → يُعاد الحساب ويُحدَّث الكاش.
- **تحديث يدوي:** زر «تحديث» على كل widget → `POST /api/reports/:domain/:metric/refresh` يفرض إعادة الحساب فورًا (محكوم بنفس صلاحية العرض).

### 7.4 وظيفة التحديث المجدولة تمسّ المستخدَم فقط
وظيفة دورية (cron) تمرّ على **المجموعة الفريدة `(metric_key, scope_signature)` المستخرجة من كل تخطيطات المستخدمين المحفوظة** فتُعيد حساب ما هو مستخدَم فعليًا فقط — لا حساب لمؤشر/نطاق لا يستعمله أحد. هذا هو التطبيق العملي لمبدأ §7.1 على مستوى الخادم.

### 7.5 ضبط تضخّم الكاش (Cardinality Control)
- الكاش يُكتب فقط للنطاقات **الخشنة** (لكل فرع / كل الفروع) والـ presets الزمنية القياسية (اليوم/الأسبوع/الشهر/الربع).
- المدى الزمني المخصّص (custom range) والاستعلامات النادرة تُحسب **حيًّا** مع حدّ معدّل (rate-limit) ولا تُكتب في الكاش، منعًا لانفجار عدد النسخ.

### 7.6 عرض الحداثة (Freshness UX)
كل widget يعرض «آخر تحديث: HH:MM» (من `computed_at`) + زر تحديث يدوي، ليعرف المستخدم أن الرقم مُخزَّن لا لحظيًّا. (تحسين لاحق اختياري: stale-while-revalidate.)

## 8. حوكمة الـ Widgets: الصلاحيات وقيمة المعلومة (Widget Governance) — **إلزامي**

> مبدأ حاكم: لكل widget جمهورٌ محدّد بالصلاحية لا يتجاوزه، وخلف كل مؤشر/KPI/مخطط **معلومة واحدة غير مكرّرة تخدم قرارًا أو متابعة سير عمل أو قياس إنجاز فريق**. ما لا يحقق ذلك لا يدخل الكتالوج.

### 8.1 التنظيم على مستويات الصلاحية (Permission-Tiered Access)
وصول الـ widget محكوم ببوابتين مستقلّتين:
1. **بوابة الرؤية (Visibility):** مفتاح الصلاحية — هل يحقّ له رؤية هذا المؤشر أصلًا؟
2. **بوابة الاتساع (Breadth):** النطاق الذي يُعرض به = **تقاطع** `permission.allowed_scopes` ∩ النطاقات الممنوحة للمستخدم لتلك الصلاحية. **يُمنع اختراع نطاقات في الـ registry** — تُقرأ من الصلاحية نفسها.

| الطبقة | الجمهور | النطاق | مثال widget |
|---|---|---|---|
| تنفيذية | إدارة عليا (يملك `GLOBAL`) | كل الفروع + مقارنة | مبيعات كل الفروع |
| فرعية | مدير فرع (يملك `BRANCH`) | فرع واحد | مبيعات فرعي |
| تشغيلية | مشرف/فني/متصل (يملك `ASSIGNED`) | سجلاتي/فريقي فقط | مهامي المتأخرة |

**قواعد:**
- المؤشر الواحد قد يظهر في عدّة طبقات، لكن كل مستخدم يراه **فقط بالاتساع الذي يسمح به منحُه**.
- إن لم يملك المستخدم أيًّا من نطاقات المؤشر المسموحة → يختفي من الكتالوج (لا يقدر حتى على إضافته).
- `<ScopeFilterBar>` يعرض فقط مستويات النطاق التي يملكها المستخدم لتلك الصلاحية — فلا يمكنه اختيار اتساع لا يحقّه.

### 8.2 عقد تعريف الـ Widget (Widget Definition Contract) — شرط دخول الكتالوج
يوسّع الشكل المختصر في §6.4. **لا يُسجَّل widget دون تصريح كامل بهذه الحقول:**

| الحقل | المعنى |
|---|---|
| `key` | معرّف فريد |
| `question` | القرار/السؤال الذي يجيب عنه — **إلزامي؛ بلا قرار = يُرفض** |
| `formula` | الصيغة الدقيقة للحساب |
| `source` | جدول الحقيقة + الأعمدة (§1.2) — **مصدر واحد** |
| `grain` | حبيبة القياس (لكل زبون/مهمة/يوم/فرع…) |
| `permission` + `allowedScopes` | بوابة الرؤية + الاتساع (تُقرأ من الصلاحية) |
| `audience` | الطبقة (تنفيذية/فرعية/تشغيلية) |

### 8.3 قاعدة عدم التكرار (Single-Canonical Rule)
- **مؤشر قانوني واحد لكل (مفهوم × حبيبة).** أي «نسخة» أخرى = تغيير نطاق/فلتر للمؤشر نفسه، لا مؤشر جديد.
- يُمنع حساب الرقم نفسه بطريقتين (تأكيد §1.2: مصدر حقيقة واحد).
- **مراجعة إدخال إلزامية** قبل تسجيل أي widget — يجب أن يحقّق الثلاثة معًا:
  1. غير مكرّر لمؤشر قائم.
  2. يخدم قرارًا إداريًا أو متابعة سير عمل أو قياس إنجاز فريق.
  3. مصدره واحد وصيغته محدّدة.
  ما لا يحقّق الثلاثة لا يدخل الكتالوج (لا مؤشرات استعراضية / vanity metrics).

### 8.4 تتبّع القيمة إلى الغاية (Decision Traceability)
كل widget يُصنَّف بغايته الصريحة: **قرار إداري / متابعة سير عمل / قياس إنجاز فريق**. يظهر التصنيف في توثيق الكتالوج، فيُمكن تدقيق أن الداشبورد بأكمله موجَّه لغاية — وأي widget بلا غاية واضحة يُزال.
