# معيار هندسة الصلاحيات والنطاق

> **الحالة:** ملزم وفعال
> **آخر تحديث:** 2026-06-11
> **الأولوية:** عند التعارض، تتقدم هذه الوثيقة على وثائق الصلاحيات الأقدم. الكود وقاعدة البيانات يبقيان الحقيقة التشغيلية، وأي اختلاف مكتشف يجب توثيقه ومعالجته، لا نسخه كنمط جديد.

## 1. الهدف

هذه الوثيقة هي نقطة البداية الإلزامية عند بناء أو تعديل أي صلاحية أو نطاق أو فلتر فرع أو تحقق ملكية. الهدف هو منع الخلط بين:

- إظهار قسم في القائمة الجانبية.
- السماح باستخدام بيانات مرجعية داخل عملية روتينية.
- تنفيذ إجراء على كيان.
- الوصول إلى سجل محدد ضمن فرع أو ملكية معينة.

المعادلة الأمنية المعتمدة هي:

```text
identity + permission + scope + subject = decision
```

لا يعتبر القرار مكتملا إذا غاب أحد عناصر هذه المعادلة عن عملية تحتاجه.

## 2. التعريفات الملزمة

### Permission

قدرة وظيفية دقيقة مثل `clients.update` أو `departments.manage`. اسم الدور ليس صلاحية، وظهور الصفحة ليس صلاحية.

### Scope

حدود تطبيق الصلاحية:

| Scope | المعنى |
|---|---|
| `GLOBAL` | جميع الفروع والسجلات التي تسمح بها الصلاحية |
| `BRANCH` | السجل تابع إلى فرع موجود ضمن تعيينات المستخدم الفعالة |
| `ASSIGNED` | السجل مسند إلى المستخدم الحالي، وفرعه مسموح أيضا |

### Subject

السجل أو السياق الحقيقي للعملية. يجب أن يحمل على الأقل `branchId` عند وجود عزل فروع، و`assignedUserId` عندما يدعم الدومين الملكية أو الإسناد.

### UI capability

التحقق في الواجهة لتحسين تجربة الاستخدام فقط. لا يمنح وصولا أمنيا ولا يعوض تحقق الخادم.

## 3. قواعد لا تقبل الاستثناء الضمني

1. لا يعتمد أي قرار أمني جديد على `hr_users.role` النصي. يستخدم `roleId` للوصول إلى المنح، ثم permission وscope وsubject لاتخاذ القرار.
2. `isSuperAdmin` مسار تجاوز صريح ومركزي فقط. لا تنشأ تجاوزات مخفية بأسماء أدوار مثل `ADMIN` أو `MANAGER`.
3. لا يكفي `requirePermission()` لحماية `/:id`. هو يتحقق من القدرة العامة فقط، ثم يجب تحميل السجل وتنفيذ تحقق subject.
4. كل endpoint يتعامل مع سجل قائم يجب أن يحمّل السجل قبل التعديل أو الحذف أو عرض التفاصيل، ثم يمرر فرعه وإسناده إلى policy الدومين.
5. كل list endpoint يطبق scope داخل query أو repository. يمنع جلب كل البيانات ثم فلترتها في الواجهة.
6. `BRANCH` يعتمد على تعيينات الفروع الفعالة، وليس على `branchId` قديم داخل JWT وحده.
7. `ASSIGNED` يتطلب تطابق المستخدم المسند، إضافة إلى صلاحية الفرع. لا يعني مجرد أن المستخدم داخل الفرع.
8. إخفاء زر أو صفحة أو عنصر Sidebar ليس حماية أمنية.
9. البيانات المرجعية اللازمة لإتمام عملية مسموحة تستخدم صلاحية lookup مستقلة عند الحاجة، ولا تتطلب تلقائيا صلاحية إدارة القسم كاملا.
10. يمنع إنشاء منطقين مختلفين لنفس القرار بين REST وtRPC. كلاهما يستدعيان service/policy مشتركة.
11. الصلاحية غير المعرفة أو scope غير المسموح به يؤديان إلى deny by default.
12. أي bypass مؤقت يجب أن يكون موثقا، محدود المدة، ومصحوبا باختبار يمنع توسعه.

## 4. نمط بناء الصلاحية

### 4.1 تعريف القدرة

قبل كتابة الكود أجب عن الأسئلة التالية:

- ما الإجراء الحقيقي: view, list, create, update, delete, assign, approve, manage أم lookup؟
- هل الصلاحية تخص صفحة إدارية أم عملية روتينية؟
- هل الدومين يدعم `GLOBAL`, `BRANCH`, `ASSIGNED` فعلا؟
- ما subject المطلوب لاتخاذ القرار؟
- من يملك السجل، وكيف يمثل الإسناد في قاعدة البيانات؟

لا تنشأ صلاحية باسم عام إذا كانت تجمع قرارات مختلفة أمنيا.

### 4.2 قاعدة البيانات وتعريفات الصلاحيات

- إضافة permission تتم عبر migration قابلة للتكرار بأمان.
- يحدد `allowed_scopes` وفقا لنموذج subject الحقيقي، لا وفقا للأدوار الحالية فقط.
- لا تعاد تسمية permission مباشرة دون خطة انتقال للمنح والكود والواجهة والتقارير.
- المصدر المستهدف للمنح والنطاقات هو `role_permission_grants`.
- يمنع إدخال اعتماد أمني جديد على `role_permissions`; ستزال بعد اكتمال الانتقال والتحقق.

### 4.3 الخادم

المسار الصحيح لعملية على سجل:

```text
authenticate
  -> capability check
  -> load subject
  -> domain policy(permission, subject)
  -> execute operation
```

مثال مفاهيمي:

```ts
router.patch(
  '/:id',
  requirePermission('clients.update'),
  async (req, res) => {
    const client = await clientsRepository.findById(req.params.id);
    await clientPolicy.authorize(req.user, 'clients.update', {
      branchId: client.branchId,
      assignedUserId: client.assignedUserId,
    });

    // Update only after subject authorization succeeds.
  },
);
```

بالنسبة للقوائم، تحول policy النطاق إلى شروط query مركزية مثل الفروع المسموحة أو المستخدم المسند.

### 4.4 الواجهة

- تستخدم permissions/grants لإظهار العمليات المتاحة وتحسين UX.
- قائمة الصلاحيات تأتي من قاعدة البيانات عبر API، ولا تنشأ قائمة صلاحيات أمنية ثابتة في الواجهة.
- خرائط العناوين والوصف الثابتة مسموحة للعرض فقط، ويجب أن تملك fallback لأي permission جديدة.
- التخزين المحلي cache للعرض، وليس مصدر حقيقة. يجب تحديث الجلسة بعد تغيير الدور أو المنح دون اشتراط logout/login كحل دائم.
- نجاح تحقق الواجهة لا يلغي تحقق الخادم.

## 5. فصل الإدارة عن الاستخدام الروتيني

يجب فصل الأنواع التالية بوضوح:

| النوع | مثال | الغرض |
|---|---|---|
| Navigation | إظهار صفحة الأجهزة في Sidebar | الوصول إلى سطح إداري |
| Manage | `device_models.manage` | إنشاء وتعديل وحذف تعريفات الأجهزة |
| Lookup | `reference_data.lookup` أو قدرة lookup متخصصة | قراءة الحد الأدنى اللازم لإكمال نموذج مسموح |
| Operation | إسناد جهاز أو إنشاء مهمة | تنفيذ workflow محدد |

مثال ملزم: المشرفة التي يسمح لها بإضافة مهمة جهاز ضمن فرع دمشق لا يجب أن تحتاج صلاحية إدارة تعريفات الأجهزة. endpoint الخاص بالنموذج يعرض فقط الأجهزة الصالحة للعملية والفرع والقسم، عبر lookup مناسب، بينما تبقى إدارة تعريفات الأجهزة محمية بصلاحيتها المستقلة.

### 5.1 نمط الحقول التشغيلية المقيّدة بالنطاق (Scoped Operational Fields)

داخل أي عملية (مثل إنشاء عقد)، توجد حقول تعرض **سجلات للاختيار منها** (وحدة جغرافية، زبون، موديل جهاز…). كل حقل من هذا النوع هو lookup محكوم بمفتاح lookup مخصص، **ويجب تقييد خياراته بنطاق السجل المالك للعملية** — في الواجهة (أي الخيارات تُحمّل) وعلى الخادم (إعادة التحقق عند الحفظ معاً). فلترة الواجهة ليست حماية.

المعادلة: **خيارات الحقل = السجلات المصرّح بها لـ (فرع/قسم العملية) ∩ (نطاق مفتاح lookup للمستخدم)**.

ثلاث حالات مرجعية ظهرت أولاً في **نموذج العقد** ويجب تعميمها على بقية النماذج (المهام، الزيارات، التسليم، التركيب، إلخ):

| الحقل | مفتاح lookup | النطاقات | مصدر الخيارات (الواجهة) | تحقّق الخادم (deny-by-default) |
|---|---|---|---|---|
| **عنوان التركيب** (جغرافيا) | `geo_units.lookup` | GLOBAL/BRANCH/ASSIGNED | `useBranchGeoUnits(branchId)` → `GET /geo-units` مفلتر بتغطية الفرع، نشط فقط | `assertGeoUnitInScope` + قيد مستوى الحي (level 4) |
| **الزبون** | `clients.view_list` | GLOBAL/BRANCH/ASSIGNED | `api.clients.list()` مفلتر بالنطاق (ASSIGNED=زبائنه، BRANCH=الفرع) | policy الزبون عند الحفظ |
| **موديل الجهاز** | `device_models.task_lookup` | GLOBAL/BRANCH/ASSIGNED | `GET /device-models` مفلتر (BRANCH=أجهزة أقسام الفرع، ASSIGNED=قسمه فقط) | `assertDeviceModelInScope` |

الآليات المشتركة الملزمة:

1. **الواجهة تطلب الخيارات بفرع العملية المالك** (فرع الزبون/الفرع الفعّال)، لا بسياق الجلسة الغامض. والنطاق grant-driven: نطاق مفتاح lookup (GLOBAL/BRANCH/ASSIGNED) هو ما يحدّد الاتساع — لا تُشتق السعة من اسم الدور.
2. **الخادم يعيد التحقق دائماً** عند الحفظ عبر مساعد `assert*InScope` (نمط `geoScopeService`/`deviceScopeService`). معرّف خارج النطاق ⇐ `403` ورفض.
3. **اللوكب الاختياري يفشل بهدوء:** طلب lookup ثانوي يفشل بـ`403` (مثلاً موظف بلا صلاحية صاحب البيعة) يجب ألا يُسقط تحميل النموذج كله — يُلتقط الخطأ ويُعاد `[]`، وإلا تبدو الحقول الأساسية (الزبون/الجهاز/العنوان) فارغة بلا سبب ظاهر.
4. **قيود جودة البيانات** (مثل: عنوان التركيب يجب أن يكون مستوى الحي) تُفرض في المنتقي (`minSelectableLevel`) وعلى الخادم معاً.

قائمة تعميم لأي حقل مقيّد جديد:
- [ ] حدّد فرع/قسم العملية المالك للحقل.
- [ ] استخدم مفتاح lookup المخصص للحقل، لا مفتاح إدارة/عرض إداري.
- [ ] الواجهة تحمّل الخيارات مقيّدة بفرع العملية؛ واللوكب الاختياري fail-soft.
- [ ] الخادم يعيد التحقق من المعرّف المحفوظ (`assert*InScope`) deny-by-default.
- [ ] اختبارات سلبية: معرّف خارج النطاق يُرفض؛ غياب الفرع ⇐ قائمة فارغة لا تسريب.

### 5.2 حارس تصعيد إسناد الأدوار (Role-Assignment Escalation Guard)

تطبيقاً للقاعدة §3-2 («`isSuperAdmin` هو مسار التجاوز الوحيد؛ لا تصعيد مخفي»): تفويض إنشاء/تعديل المستخدمين للفروع (`admin.roles.users.manage` بنطاق `BRANCH`) يجب ألا يفتح باب تصعيد عبر إسناد دور يحمل صلاحيات لا يملكها المُسنِد.

القاعدة المعتمدة (وسط بين الأمان والتفويض): **مُسنِد غير سوبر-أدمن يستطيع إسناد دور فقط إذا كان يملك — بنطاق مساوٍ أو أوسع — كل صلاحية «حسّاسة/إدارية» يحملها ذلك الدور.** الصلاحيات التشغيلية (clients/telemarketing/jobs/tasks/field_visits…) **تُفوَّض بحرية ضمن الفرع** ولا تشترط أن يملكها المُسنِد — فمدير الفرع لا يحتاج كل صلاحية تشغيلية كي يُسند دوراً تشغيلياً لموظفي فرعه.

- المجموعة الحسّاسة `SENSITIVE_PERMISSION_KEYS` (admin.*‏، users.branch_assignments.*‏، branches.edit/manage، departments.manage، catalog.manage، device_models.manage، spare_parts.manage، devices.discounts.manage، settings.*) معرّفة مركزياً في `roleAssignmentGuard.ts`.
- التنفيذ: `assertRoleWithinActorScope(authContext, roleId)` — يُستدعى في **كل مسارات إسناد الدور** (tRPC `createHrUser`/`updateHrUser`، REST `/api/admin/hr-users`، و`employees/:id/system-account`)؛ فشل ⇒ `403`.
- الواجهة: `roles.list` يُرجع علم `assignable` لكل دور (محسوب بنفس الحارس)، وقائمة الأدوار في نموذج المستخدم تعرض القابلة للإسناد فقط (UX؛ الخادم يبقى الحماية).
- نتيجة جانبية مفيدة: الحارس يكشف الأدوار التشغيلية المُلوّثة بصلاحيات إدارية (بذور خاطئة) — يرفض تفويضها حتى تُنظَّف. لذا **يجب ألا تحمل الأدوار التشغيلية (مشرف/فني) صلاحيات من المجموعة الحسّاسة.**

## 7. الخطة المرحلية

### المرحلة 0: تثبيت العقد والجرد

- اعتماد هذه الوثيقة وقفل القواعد الأساسية.
- تشغيل inventory لكل permission وendpoint وscope وsubject.
- تصنيف الاستثناءات الحالية قبل توسيع النظام.

### المرحلة 1: توحيد مصدر الحقيقة

- جعل `role_permission_grants` مصدر المنح والنطاقات للخادم والواجهة.
- اشتقاق permission keys المعروضة في الجلسة من grants نفسها.
- إيقاف الكتابة الجديدة إلى `role_permissions`.
- حذف الجدول القديم فقط بعد مقارنة النتائج وترحيل البيانات واختبارات regression.

### المرحلة 2: توحيد إدارة الأدوار

- إنشاء service وpolicy مشتركتين لإدارة الأدوار والمنح.
- جعل REST وtRPC يستدعيان نفس الطبقة خلال الانتقال.
- إلغاء سطح REST القديم بعد التأكد من عدم وجود مستهلكين.

### المرحلة 3: تقوية الجلسة وJWT

- الوصول إلى token مصغر يحمل `userId` و`sessionVersion` فقط قدر الإمكان.
- تحميل الدور الحالي وحالة super-admin والفروع والمنح من مصدر حديث أو cache قصير مضبوط.
- إبطال الجلسات أو رفع `sessionVersion` عند التغييرات الأمنية الحساسة.
- إزالة الاعتماد على role وbranch القديمين داخل JWT كحقيقة أمنية.

### المرحلة 4: توحيد سياسات subject

- فصل capability middleware عن record authorization بأسماء وواجهات واضحة.
- اعتماد عقد policy موحد للدومينات.
- نقل branch وownership filters من routes المتفرقة إلى policies/repositories.

### المرحلة 5: ترحيل الدومينات تدريجيا

الترتيب المقترح حسب الحساسية:

1. الزبائن.
2. المهام والمهام المفتوحة.
3. الزيارات الميدانية.
4. العقود.
5. الموظفون والهيكل التنظيمي.
6. البيانات المرجعية وأقسام الإدارة.

يطبق كل دومين كوحدة صغيرة مع اختباراته، ولا ينفذ refactor شامل دفعة واحدة.

### المرحلة 6: تحديث الواجهة والجلسة

- توفير endpoint/session refresh يعكس تغيير المنح مباشرة.
- إبقاء تعريفات الصلاحيات ديناميكية من قاعدة البيانات.
- فصل صلاحيات navigation وlookup وoperation وmanage.

### المرحلة 7: التدقيق الآلي والاستمرار

- فشل CI عند استخدام permission غير معرفة.
- فشل CI عند منح scope غير مسموح.
- إضافة تدقيق للـrecord routes التي تملك capability gate دون subject policy.
- إبقاء Excel وملفات inventory متزامنة مع كل تغيير.

## 8. مصفوفة الاختبار الإلزامية

لكل permission ذات scope، يجب اختبار الحالات المناسبة التالية:

| الحالة | النتيجة المتوقعة |
|---|---|
| دون permission | `403` |
| permission مع scope غير مسموح | رفض المنح أو `403` |
| `BRANCH` والفرع خارج التعيينات | `403` أو عدم ظهور السجل في list |
| `BRANCH` والفرع ضمن التعيينات | نجاح |
| `ASSIGNED` والسجل لغير المستخدم | `403` أو عدم ظهوره |
| `ASSIGNED` والسجل للمستخدم ضمن فرع مسموح | نجاح |
| `GLOBAL` | نجاح عبر الفروع وفق القدرة |
| super-admin | نجاح عبر المسار الصريح |
| مستخدم معطل أو تعيين فرع منتهي | رفض |

يجب اختبار list وdetails وcreate وupdate وdelete بصورة منفصلة عندما تكون موجودة، لأن نجاح فلتر القائمة لا يثبت حماية route التفاصيل.

الأدوار المرجعية للاختبارات اليدوية والمصفوفة هي:

- المشرفة.
- الفني.
- مدير الفرع.

يسجل لكل دور هل يملك الصلاحية وما scope الممنوح، ولا يستنتج السلوك من اسم الدور.

## 9. تحديث ملف Excel والآثار المصاحبة

أي إضافة أو حذف أو إعادة تسمية أو تغيير scope يتطلب في نفس الخطوة:

1. تحديث `صلاحيات_النظام.xlsx`.
2. تلوين الإضافة بالأخضر والحذف بالأحمر بصورة واضحة.
3. تحديث عمود/ورقة اختبار المشرفة والفني ومدير الفرع، مع permission وscope المتوقعين.
4. تحديث inventory ونتائج audit ذات الصلة.
5. توثيق migration والملفات التي تنفذ القرار.
6. إبقاء التلوين كسجل مراجعة حتى اعتماد الخطوة، ثم أرشفته بطريقة لا تفقد تاريخ التغيير.

## 10. تعريف الإنجاز

لا تعتبر الصلاحية مكتملة إلا بعد تحقق جميع البنود التالية:

- permission معرفة في migration أو تعريفات قاعدة البيانات.
- allowed scopes منطقية ومقيدة.
- backend capability check موجود.
- subject policy موجودة لكل record operation.
- list filtering يطابق السياسة نفسها.
- UI gate موجود عند الحاجة لكنه غير مستخدم كحماية.
- REST وtRPC لا يملكان قرارين متعارضين.
- الاختبارات الإيجابية والسلبية ناجحة على staging.
- audit لا يظهر permission مجهولة أو endpoint غير محمي.
- Excel وinventory والدستور محدثة.

## 11. التنفيذ التجريبي الحالي

`departments.manage` هو التنفيذ الصغير المرجعي الأول: إدارة الأقسام منفصلة عن lookup الروتيني، والتحقق يراعي فرع subject. يستخدم هذا المثال كنقطة مقارنة، لا كدليل على اكتمال ترحيل بقية الدومينات.

### 2026-06-13: اعتماد نمط الحقول التشغيلية المقيّدة بالنطاق (§5.1) في نموذج العقد

- وُثّق النمط العام في [§5.1](#51-نمط-الحقول-التشغيلية-المقيّدة-بالنطاق-scoped-operational-fields) بثلاث حالات مرجعية بدأت في نموذج العقد ويجب تعميمها لاحقاً: العنوان، الزبون، الجهاز.
- **العنوان:** `geo_units.lookup` (هجرة 279) + جلب عبر `useBranchGeoUnits(branchId)` بفرع الزبون + `assertGeoUnitInScope` + قيد مستوى الحي (level 4) على المنتقي والخادم.
- **الزبون:** رؤية محكومة بنطاق `clients.view_list` (ASSIGNED=زبائنه، BRANCH=الفرع). درس مهم: اللوكب الاختياري (`employees.closers`) كان يُسقط تحميل النموذج كله بـ`403`؛ صار fail-soft.

- باقٍ للتعميم: تطبيق النمط نفسه على بقية النماذج (المهام، الزيارات، التسليم/التركيب) — مع قائمة التحقق في §5.1.
