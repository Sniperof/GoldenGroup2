## 1. الغرض والتعريف

كل جهاز فيزيائي له **صحة فنية** تُقاس مرارًا طوال حياته التشغيلية. هذا الملف يثبّت نموذج
**سجل الصحة الفنية للجهاز**: سجل تاريخي append-only يوثّق كل قراءة فنية أُخذت للجهاز،
في أي مهمة، ولأي سبب.

> **التعريف التنفيذي:** سجل الصحة الفنية هو توثيق تاريخي لقراءات الحالة الفنية لجهاز
> فيزيائي محدد، تُلتقط كل قراءة منها ضمن مهمة مصدر، وتصف وضع الجهاز الفني في لحظة الالتقاط،
> دون أن تكون السعر أو الحيازة أو الحالة التشغيلية.

### الطبقة الخامسة من تعريف الجهاز

[01f §العلاقة بين الجهاز والقطع](./01f-device-parts-and-components.md) أثبت أربع طبقات للجهاز
(الهوية، الحيازة، الحالة التشغيلية، القطع). سجل الصحة الفنية هو الطبقة الخامسة المكمّلة:

1. الهوية الفيزيائية
2. سجل الحيازة
3. الحالة التشغيلية
4. التركيب الفني والقطع
5. **الصحة الفنية عبر الزمن** ← هذا الملف

## 2. القاعدة المؤسِّسة

> **لا توجد حالة فنية بدون مهمة.**

كل قراءة صحة فنية تلزمها **مهمة مصدر** و**جهاز فيزيائي**. القراءة ليست كيانًا قائمًا بذاته،
بل أثرٌ فنيٌّ لمهمة جرت على جهاز.

ومن ذلك القاعدة الثانية:

> **مفتاح سجل الصحة هو الجهاز الفيزيائي، لا العقد.**

الصحة تتبع الأصل المادي (الطبقة 3 من [01](./01-what-is-a-device.md))، فتصمد أمام:
- الأجهزة الخارجية التي لا عقد لها.
- انتقال الحيازة أو تغيّر العقد.

## 3. الوضع الحالي والثغرات (نقطة الانطلاق)

الجدول `device_technical_states` (migration 116) موجود ويلتقط ~22 قراءة فنية، لكنه يحمل ثغرات:

| الثغرة | الوصف |
|---|---|
| **لا ربط بالجهاز الفيزيائي** | الجدول مربوط بـ `contract_id` فقط؛ لا عمود `installed_device_id`. الأجهزة الخارجية بلا عقد لا تملك صحة. |
| **مصدر التقاط وحيد** | يُكتب فقط من معالج الطوارئ (`emergencyResult.ts`)؛ النموذج الموحّد للزيارات لا يلتقط صحة. |
| **القراءة مقيّدة بالعقد** | `GET /emergency-result/device/:contractId/history` يستعلم `WHERE contract_id`، فلا قراءة بمعرّف الجهاز ولا للأجهزة الخارجية. |
| **`phase` ناقص** | مقيّد بـ `pre`/`post`/`standalone`؛ لا يميّز القراءة التشخيصية عن المرجعية. |
| **حراسة فضفاضة** | endpoint التاريخ محروس بـ `marketing_visits.view`. |

التعديلات تبدأ من معالجة هذه الثغرات وفق القرارات أدناه.

## 4. القرارات المحسومة

### القرار 1 — بنية القياسات: هيكل ثابت + تعقيم اختياري

القياسات تبقى **أعمدة ثابتة** (لا قوالب مرنة حسب نوع الجهاز — هندسة زائدة لعائلة أجهزة
متجانسة). مجموعة القياسات المعيارية:

| الحقل | النوع | القيم/الملاحظة |
|---|---|---|
| نوع مصدر المياه | enum | رئيسية · خزان |
| عيار مصدر المياه | رقم | قياس الفني (ppm) |
| الضغط الداخل | enum | قوي · جيد · وسط · ضعيف |
| وجود كاسر | منطقي | نعم · لا |
| عيار مي حنفية الجهاز قبل الصيانة | رقم | قياس الفني (ppm) |
| ضغط المضخة | رقم | قياس الفني (bar) |
| خرج الميمبرين | رقم | قياس الفني (ppm) |
| دخل الميمبرين | رقم | قياس الفني (ppm) |
| تدفق الميمبرين | enum | جيد · وسط · ضعيف |
| فلو الكب | enum | 300 · 450 |
| كفاءة الميمبرين | **محسوبة** | معادلة — لا تُخزَّن (راجع §5) |
| عيار الهاي برشر | رقم | قياس الفني (ppm) |
| لو برشر | منطقي (يعمل/لايعمل) | |
| عيار الخزان | رقم | قياس الفني (ppm) |
| نوع القسام | enum | ميكانيك · كهرباء |
| ترانس مضخة | enum | 3 أمبير · 1.5 أمبير |
| وجود صباب خامسة | enum | موجود · إلغاء |
| توصيلة الجهاز | enum | تشالنجر · RO |
| ملاحظات إضافية | نص | حر |

**كتلة التعقيم — اختيارية** (ليس لكل جهاز تعقيم):

| الحقل | النوع |
|---|---|
| ترانس التعقيم | منطقي (يعمل/لايعمل) |
| لمبة التعقيم | منطقي (يعمل/لايعمل) |
| سليفة التعقيم | منطقي (يعمل/لايعمل) |

اختيارية التعقيم تُفعَّل عبر عَلَم قدرة على مستوى **نوع الجهاز** (`device_model.has_sterilization`)
يقرّر إظهار الكتلة في الفورم — أنظف من الاعتماد على ترك الحقول فارغة.

### القرار 2 — السياق: بُعدان منفصلان

معلومتان مختلفتان لا تُخلطان:

1. **نوع المهمة** (لماذا حضر الفني): تشغيل · صيانة طارئة · صيانة دورية · تشييك جهاز ·
   صيانة داخل الشركة. خاصية المهمة المصدر، لا القراءة.
2. **دور القراءة داخل المهمة** (`phase`): متى/لماذا أُخذت هذه القراءة بالذات.

> **استثناء صريح — التركيب/التسليم ليسا نقطتَي التقاط.** مهمة التركيب (`device_delivery`)
> ومهمة التسليم لا تنتجان قراءة حالة فنية، لأن الجهاز لا يُشغَّل بالضرورة عند التركيب،
> فلا قراءة ذات معنى. أول قراءة تشغيلية مرجعية (`baseline`) تُؤخذ عند **التشغيل (activation)**.
> عند تعميم الالتقاط (ربط الأنواع بالـ registry) **يُمنع** إضافة عارض حالة فنية لمهمة التركيب/التسليم.

قيم `phase` المعتمدة (توسيع للقيمة `standalone` القديمة):

| القيمة | المعنى |
|---|---|
| `pre` | قبل العمل |
| `post` | بعد العمل (بعد الصيانة / بعد التشغيل) |
| `diagnostic` | قراءة تشخيصية لشكّ بعطل محدد |
| `baseline` | قراءة مرجعية مستقلة بلا عمل (تشييك توثيقي) |

### القرار 3 — المفاتيح الإلزامية ولقطة نوع المهمة

| الحقل | القاعدة | السبب |
|---|---|---|
| `installed_device_id` | **إلزامي (NOT NULL)** — المفتاح | الصحة تتبع الأصل الفيزيائي |
| `open_task_id` | **إلزامي (NOT NULL)** | تفعيل «لا حالة فنية بدون مهمة» |
| `contract_id` | **اختياري، مرجع سياقي** | الجهاز الخارجي بلا عقد؛ يُشتقّ من الجهاز ولم يعد مفتاحًا |
| `task_type_snapshot` | **لقطة تُخزَّن على السجل** | المهام تُعدَّل/تُحذف، والسجل تاريخي دائم؛ التقارير تحتاج قيمة مستقرة؛ نمط اللقطات المعتمد بالمشروع |

**مرجع المهمة الموحّد:** أي مسار التقاط (طوارئ/دورية/تشييك/تشغيل/صيانة داخلية) يكتب مرجع
المهمة الموحّد نفسه، حتى لا ينقسم السجل بين `open_tasks` و`visit_tasks`.

### القرار 4 — الثبات: append-only مع تجميد عند إغلاق المهمة

| الحالة | القاعدة |
|---|---|
| المهمة المصدر مفتوحة/قيد التنفيذ | القراءة قابلة للتعديل (تصحيح إدخال أثناء العمل) |
| المهمة المصدر مغلقة/مكتملة | القراءة **مجمَّدة** — لا تعديل بأثر رجعي |
| تصحيح بعد الإغلاق | **قراءة جديدة** (لا تعديل في المكان)؛ يبقى الأصل في التاريخ |

يحافظ هذا على نزاهة «الصحة طوال الحياة»، ويظل عمليًا للفني أثناء المهمة، ومتّسق مع منطق
المشروع (أحداث القطع والحيازة لا تُعدَّل بأثر رجعي).

### القرار 5 — الصحة الحالية مقابل التاريخ، والفصل عن الحالة التشغيلية

1. **الصحة الحالية = اشتقاق، لا عمود مخزَّن.** «الحالة الفنية الحالية» = أحدث قراءة زمنيًا
   (`ORDER BY created_at DESC LIMIT 1`) بصرف النظر عن دورها. لا عمود منسوخ على
   `installed_devices` تجنّبًا للانحراف (drift)؛ مصدر الحقيقة يبقى السجل.
2. **فصل الصحة الفنية عن الحالة التشغيلية:**
   - `installed_devices.status` = أين الجهاز تشغيليًا (active / faulty / in_workshop…).
   - سجل الصحة = القياسات الفنية التفصيلية.
   - **القراءة لا تغيّر `status` تلقائيًا.** قد تكون قراءة `diagnostic` سيئة سببًا للانتقال إلى
     `faulty`، لكن الانتقال يبقى فعلًا تشغيليًا صريحًا عبر المهمة، لا أثرًا جانبيًا آليًا للقراءة.

هذا يحترم تعريف الدستور: الحالة التشغيلية تتغيّر بحدث ميداني، والصحة قياس.

### القرار 6 — الصلاحيات والسكوب

1. **القراءة (عرض تاريخ الصحة):** تُحرَس بـ `installed_devices.view`، مع سكوب الفرع عبر
   `installed_devices.branch_id` — من يرى الجهاز يرى صحته. (يستبدل الحراسة الحالية
   الفضفاضة `marketing_visits.view`.)
2. **الكتابة (التقاط قراءة):** **لا صلاحية كتابة جديدة.** الالتقاط يحدث داخل مسار إكمال
   المهمة فيرث صلاحيتها — من يحقّ له تنفيذ المهمة يحقّ له تسجيل قراءتها الفنية.

(بديل مؤجَّل ومرفوض الآن لأنه زائد: صلاحية فرعية مخصّصة
`installed_devices.technical_state.view` لإخفاء التفاصيل الفنية عن بعض من يرى الجهاز.)

## 5. تفاصيل ملحقة

- **كفاءة الميمبرين** تبقى **محسوبة** من الدخل/الخرج (`(1 − inlet / outlet) × 100`)، لا تُخزَّن.
- **الالتقاط الموحّد:** أنواع المهام الخمس (تشغيل · طارئة · دورية · تشييك · صيانة داخلية)
  تكتب كلها في نفس الجدول — مصدر واحد للصحة.
- **هجرة البيانات:** ربط السجلات القديمة بـ `installed_device_id` عبر `contract_id`، ومعالجة
  السجلات اليتيمة (بلا عقد ولا مهمة) الموجودة حاليًا.

---

## 6. ما يمثّله سجل الصحة وما لا يمثّله

**يمثّل:**
- القياسات الفنية للجهاز عبر الزمن
- صحة الجهاز الحالية (اشتقاقًا)
- أثر المهام الفنية على وضع الجهاز
- مدخلًا لتحليل الأعطال المتكررة

**لا يمثّل:**
- السعر أو الذمم
- الحيازة (راجع [01e](./01e-device-possession-ledger.md))
- الحالة التشغيلية (`status`)
- القطع المركّبة (راجع [01f](./01f-device-parts-and-components.md)) — وإن كانت قراءة قد
  ترافق حدث قطعة في نفس المهمة
