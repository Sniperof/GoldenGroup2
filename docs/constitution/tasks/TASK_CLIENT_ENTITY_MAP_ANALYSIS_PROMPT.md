# تحليل خريطة كيانات الزبون — Client Entity Map Analysis

## السياق

Golden CRM فيه 81+ جدول بالـ DB. بدنا نعرف: **شو الكيانات اللي لازم تتوثّق من منظور الزبون؟** يعني أي جدول بيحمل بيانات عن الزبون أو بيخدم رحلته بالنظام.

## المطلوب

1. **افحص كل الجداول** اللي فيها `client_id` أو `customer_id` (FK للزبون)
2. **حدّد** أي جداول لسّاتها ما إلها دستور مستقل (`domains/XX.md`)
3. **صنّف** الكيانات حسب مرحلة رحلة الزبون
4. **رتّب** حسب الأولوية (الأهم للزبون → الأقل)
5. **اطبع** تقرير واضح ومفصّل

## مصادر البيانات

- `/opt/golden-crm/apps/staging/migrations/*.sql` — الهجرات (الـ schema الحقيقي)
- `/opt/golden-crm/apps/staging/docs/constitution/INDEX.md` — شو مُوثّق حالياً
- `/opt/golden-crm/apps/staging/docs/constitution/CROSS-REFERENCE.md` — العلاقات بين الجداول

## المعايير

- **⭐ أولوية عالية:** الجدول فيه بيانات مالية أو تعاقدية أو طوارئ — أي خلل بيأثر على الزبون مباشرة
- **🟡 أولوية متوسطة:** الجدول تشغيلي — بيأثر على كفاءة الفريق
- **🟢 أولوية منخفضة:** الجدول نظامي أو أرشيفي

## قواعد التسمية
- اكتب بالعربي (السوري)
- رتّب حسب الأولوية ضمن كل مرحلة
- إذا الجدول legacy/deprecated — حط ⚠️ وحدّد البديل
- إذا الجدول جزء من دستور تاني (مثلاً `contract_line_items` جزء من `contracts.md`) — حط "جزء من X.md"

## ملاحظات مهمة
- `customer_id` و `client_id` = نفس الشي تقريباً — `customer_id` قديمة بس لسّاتها مستخدمة
- `tasks` (القديم) = legacy → البديل `open_tasks`
- `visits` (القديم) = legacy → البديل `field_visits`
- `marketing_visits` = legacy → migration جارية للتحويل لـ `field_visits`
- بعض الجداول "فرعية" (junction/result) — إذا ما إلها منطق مستقل كبير، ما بدها دستور مستقل
