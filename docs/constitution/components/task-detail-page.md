## ١ — لماذا هذه الوثيقة؟

كل أنواع المهام (`device_demo`, `device_delivery`, `device_installation`, `device_activation`, `emergency_maintenance`, `periodic_maintenance`, `installment_collection`, `gift_delivery`, `golden_warranty`, ...) تُعرض عبر **مكوّن واحد**: [`TaskDetailLayout`](../../../packages/web/src/components/tasks/TaskDetailLayout.tsx).

المشكلة المتكررة: كل مرة يُضاف نوع مهمة جديد، يَظهر سلوك مختلف — تبويب فارغ، بيانات ناقصة، أو حقل لا يظهر — لأن لا يوجد **عقد مكتوب** يحدّد *ما الذي يجب أن يظهر لكل مهمة*.

> **المبدأ الحاكم:** أي نوع مهمة جديد يجب أن يَعرض **نفس البنية الموحّدة** أدناه دون استثناء. ما يختلف بين الأنواع هو **بيانات التنفيذ فقط**، ويُضاف عبر نقاط تمديد مُعرَّفة — لا عبر تفريع السلوك.

## ٢ — بنية الصفحة الموحّدة

كل مهمة، مهما كان نوعها، تُعرض بنفس هذا الهيكل:

```
TaskHeader  (الترويسة: النوع + الزبون + زر الرجوع)
└── شريط التبويبات
    ├── نظرة عامة      (overview)      ← إجباري لكل مهمة
    ├── بيانات الزبون   (client)        ← إجباري لكل مهمة
    ├── العقد والجهاز   (contract)      ← إجباري لكل مهمة
    ├── التواصل والمتابعة (communication) ← إجباري لكل مهمة
    ├── [تبويبات تنفيذ] (extraTabs)     ← اختياري حسب النوع
    └── النتيجة         (result)        ← إجباري لكل مهمة
```

التبويبات الأربعة + النتيجة **ثابتة لكل الأنواع**. التبويبات الإضافية (`extraTabs`) هي نقطة التمديد الوحيدة المسموحة لإضافة عرض خاص بنوع المهمة (مثل «تفاصيل الخدمة» لمهام ما بعد البيع).

## ٣ — قواعد حاكمة عامة

### ٣.١ — التواريخ للقراءة فقط (READ-ONLY)
`due_date` (التاريخ المطلوب) و`expected_date` (التاريخ المتوقع) **لا يُعدَّلان يدوياً من صفحة التفاصيل إطلاقاً**. يتحركان حصراً عبر دورة حياة المهمة (الحجز / انعكاس نتيجة الزيارة / مسارات النظام). تُعرض كنصّ ثابت مع شارة البُعد الزمني فقط.

### ٣.٢ — لا حقل فارغ صامت
أي حقل إجباري غير متوفّر يُعرض بـ `—` صراحةً، لا يُحذف. وأي تبويب بلا بيانات يَعرض `EmptyState` مفهوماً يشرح السبب (لماذا لا توجد بيانات لهذا النوع).

### ٣.٣ — مصدر البيانات = لقطات مجمّدة + سجل حيّ
- لقطات العقد/الجهاز/الزبون تُبنى **وقت الإنشاء** عبر `persistOpenTaskSnapshots` ويجب أن يستدعيها **كل** منشئ مهمة.
- كل تحوّل حالة فعلي للمهمة يجب أن يُسجَّل صفّاً في `task_activity_log` (بما فيها الانعكاس من نتيجة الزيارة) وإلا يتعطّل السجل ومشتقّاته (`completedAt`، عدّاد الأنشطة، شارة التأخّر).

### ٣.٤ — توحيد التسميات
كل قيمة مُرمَّزة (حالة، قرار، نوع حدث، قناة تواصل) يجب أن تملك تسمية عربية في خرائط الواجهة المركزية. القيمة غير المُعرَّفة تُعرض خاماً = خطأ توثيق يجب سدّه.

## ٤ — تبويب «نظرة عامة» — الحقول الإجبارية لكل مهمة

| البطاقة | الحقل | المصدر | إجباري؟ |
|--------|-------|--------|---------|
| **Hero** | الحالة + المرحلة المشتقّة | `status` / `phase` | ✅ لكل مهمة |
| Hero | الأولوية | `priority` | ✅ |
| Hero | عدّاد المحاولات | `attemptsCount` | ✅ |
| Hero | التاريخ المطلوب + شارة البُعد الزمني | `dueDate` | ✅ (للقراءة) |
| Hero | الموعد المتوقع + شارة البُعد الزمني | `expectedDate` | ✅ (للقراءة) |
| **تفاصيل المهمة** | سبب الإنشاء | `reason` | ✅ |
| تفاصيل المهمة | سبب الانتظار | `waitingReasonText` | عند وجوده |
| تفاصيل المهمة | آخر تحديث على محاولة | `lastAttempt.closedAt` | عند وجوده |
| تفاصيل المهمة | سبب الإلغاء | `cancellationReason` | عند الإلغاء |
| **الجدولة** | الموعد المتوقع (للقراءة) | `expectedDate` | ✅ |
| الجدولة | موعد الزيارة القادمة/آخر محاولة | `activeVisit` / `lastAttempt` | عند وجوده |
| الجدولة | الفرع | `branchName` | ✅ |
| **بيانات الإنشاء** | تاريخ الإنشاء + آخر تحديث | `createdAt` / `updatedAt` | ✅ |
| بيانات الإنشاء | منشئ المهمة | `createdByName` (أو «النظام») | ✅ |
| بيانات الإنشاء | مصدر الإنشاء | `creationOrigin` | ✅ |
| بيانات الإنشاء | **المهمة الأم / المصدر** | `sourceContextType` / `sourceContextId` | عند وجوده |

## ٥ — التبويبات الأخرى — الحدّ الأدنى الإجباري

**بيانات الزبون (`client`):** الاسم، الموبايل + أرقام التواصل (واتساب)، العنوان الكامل (محافظة › منطقة › حي › تفصيلي)، التصنيف (LEAD/FOP/OP)، التقييم، المُحيل. مصدرها لقطة الزبون أو حقول الزبون الحيّة.

**العقد والجهاز (`contract`):** بطاقة العقد (رقم/تاريخ/حالة/الفرع/مُغلِق) + بطاقة الجهاز (موديل/تسلسلي/حالة/تواريخ التسليم-التركيب-التشغيل) + الكفالة + عنوان التركيب (مع الخريطة) + البيانات المالية. **شرط:** يجب أن تكون اللقطة مبنيّة وقت الإنشاء؛ المهمة المرتبطة بجهاز يجب ألا تُظهر هذا التبويب فارغاً أبداً.

**التواصل والمتابعة (`communication`):** اتصالات التيلماركتر المرتبطة بالمهمة + السجل الكامل لتحوّلات الحالة والأحداث + **خيط الملاحظات**. خيط الملاحظات يبدأ دائماً بـ **ملاحظة منشئ المهمة** (`task.notes` منسوبةً إلى المنشئ وتاريخ الإنشاء) ثم ملاحظات المتابعة اللاحقة، مع محرّر لإضافة ملاحظة جديدة. ملاحظة المهمة لا تُعرض في «نظرة عامة».

**النتيجة (`result`):** ملخص نتيجة المهمة (عند حالة نهائية) + **تاريخ الإتمام** (مصدره `lastAttempt.closedAt` الموثوق، لا العمود المعطوب `completedAt`) + قائمة محاولات التنفيذ (`visit_tasks`) + عارض نتيجة خاص بالنوع عبر `ResultRenderer` (نقطة تمديد). تاريخ الإتمام يخصّ هذا التبويب حصراً ولا يُعرض في «نظرة عامة».

**تبويبات التنفيذ (`extraTabs`):** نقطة التمديد الوحيدة المسموحة لعرض خاص بالنوع.

## ٦ — قائمة تحقّق إضافة نوع مهمة جديد

عند إضافة `task_type` جديد، لا يُعتبر مكتملاً إلا بعد:

- [ ] صفحة تفصيل موصولة في [`App.tsx`](../../../packages/web/src/App.tsx) (مسار `/:id`) قبل الـ catch-all.
- [ ] رابط رجوع `backHref`/`backLabel` صحيح للمجموعة المصدر.
- [ ] منشئ المهمة يستدعي `persistOpenTaskSnapshots` (عقد/جهاز/زبون).
- [ ] انعكاس نتيجة الزيارة يكتب `status_change` في `task_activity_log`.
- [ ] كل قيم الحالة/القرار للنوع لها تسميات عربية في خرائط الواجهة.
- [ ] التبويبات الأربعة + النتيجة تعرض الحدّ الأدنى الإجباري (§٤، §٥).
- [ ] التواريخ معروضة للقراءة فقط (§٣.١).
- [ ] عند غياب عقد/جهاز يظهر `EmptyState` مفهوم، لا صفحة بيضاء.

## ٧ — فجوات معروفة (Backlog) — يجب سدّها

1. **اكتمال المهمة لا يُسجَّل في `task_activity_log`** عند الانعكاس من نتيجة الزيارة → السجل ناقص + `completedAt` دائماً NULL + شارة «متأخّر» تبقى للأبد على المهام المكتملة. (الأعلى أولوية)
2. **لقطات العقد/الجهاز لا تُبنى** لمهام ما بعد البيع (`createDeliveryTaskForContract` لا يستدعي `persistOpenTaskSnapshots`) → تبويب «العقد والجهاز» فارغ رغم ارتباط المهمة بجهاز.
3. **عدّاد الأجهزة = 0** لمهام ما بعد البيع لأن `getDevices` يقرأ `open_task_devices` فقط ولا يشمل `open_tasks.device_id`.
4. **«آخر تحديث على محاولة»** يقرأ العمود الميّت `last_attempt_at` بدل `lastAttempt.closedAt`.
5. **مكالمات التيلماركتر** لا تُربط بمهام ما بعد البيع (لا `call_task_links`).
6. **حقول ناقصة في «نظرة عامة»:** المهمة الأم/المصدر، والمرحلة المشتقّة كعرض صريح. (ملاحظة المهمة نُقلت إلى تبويب التواصل كأول ملاحظة من المنشئ — مُطبَّقة.)
