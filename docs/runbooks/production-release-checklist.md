# Golden CRM — Production Release Checklist

> **الحالة:** مرجع تشغيل
> **النطاق:** النشر على البرودكشن فقط
> **ملاحظة:** المايغريشنز على البرودكشن تبقى *يدوية* بعد المراجعة، وليست جزءًا من سكربت النشر التلقائي.

## 1) قبل النشر

- تأكد أن التغييرات المطلوبة موجودة ومجرّبة على `staging`.
- تأكد أن فرع البرودكشن هو `main`.
- تأكد أن الـ build ناجح على الستيجنج قبل رفع أي شيء.
- راجع المايغريشنز الجديدة وحدد إن كان أي واحد منها مطلوب على البرودكشن.

## 2) ملف النشر الرسمي

السكربت المعتمد للنشر موجود هنا:

- `scripts/deploy-production.sh`

ويقوم بالخطوات التالية:

1. `git fetch origin`
2. `git checkout -B main origin/main`
3. حذف ملفات البيئة المحلية (`.env`, `.env.production`, وغيرها)
4. تحميل إعدادات البرودكشن من:
   - `/etc/golden-crm/production.env`
5. `pnpm install --frozen-lockfile`
6. `pnpm build`
7. `pm2 restart golden-crm`
8. `pm2 save`
9. فحص الصحة عبر:
   - `http://127.0.0.1:3000/api/health`

## 3) المايغريشنز

- إذا في migration جديد مطلوب، شغّله **يدويًا** على قاعدة البرودكشن بعد التأكد من صحته.
- لا تشغّل migrations تلقائيًا داخل سكربت النشر.
- شغّل migration فقط إذا:
  - اعتمدت تغييرات schema
  - وتمت مراجعة أثرها على البيانات الحالية

## 4) تسلسل النشر المقترح

1. تحديث الكود على `main`.
2. بناء البرودكشن باستخدام سكربت النشر.
3. تشغيل المايغريشنز المطلوبة يدويًا إذا لزم.
4. إعادة تشغيل PM2.
5. التأكد من الـ health check.
6. مراجعة الشاشة الرئيسية والصفحات الحرجة يدويًا.

## 5) تحقق بعد النشر

- الصفحة الرئيسية تفتح بدون أخطاء.
- `api/health` يرجع 200.
- صفحات التشغيل الأساسية تفتح:
  - العملاء
  - التيلماركتر
  - المهام المفتوحة
  - الزيارات
- لا توجد أخطاء 500 في الـ logs.

## 6) Rollback سريع

إذا ظهر خلل بعد النشر:

- ارجع إلى آخر commit مستقر على `main`.
- أعد تشغيل السكربت.
- إذا كانت المشكلة من migration، لا تعكسه إلا بخطة رجوع واضحة ومراجعة البيانات أولًا.

## 7) ملاحظات مهمة

- هذا الملف **دليل تشغيل** وليس تنفيذًا آليًا.
- أي خطوة تمس البيانات على البرودكشن تحتاج مراجعة مستقلة.
- لا تعتمد على تعديل واحد فقط قبل العرض؛ تأكد من الواجهة + الـ API + المايغريشنز.
