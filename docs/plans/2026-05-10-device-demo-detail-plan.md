# خطة تنفيذ صفحة تفاصيل مهمة عرض الجهاز

> **للتنفيذ لاحقاً:** استخدم هذه الخطة لتعديل صفحة التفاصيل على الستيجنج فقط، مع الالتزام بالمصادر الفعلية الموجودة في الكود والملف التحليلي.

**الهدف:** إعادة تنظيم صفحة تفاصيل مهمة عرض الجهاز بحيث تعرض البيانات بشكل أوضح، وتفصل بين الزيارة والمهمة والعروض والتواصل والنتيجة، وبنفس المصطلحات والأنماط اللي ثبتناها في مودل إضافة العرض.

**المرجع التحليلي الأساسي:**
- `docs/marketing-visit-device-demo-fields.md`

**الصفحة الفعلية الحالية:**
- `packages/web/src/pages/tasks/DeviceDemoDetail.tsx`

**النمط المرجعي من مودل الإضافة:**
- `packages/web/src/components/clients/DeviceOfferModal.tsx`

---

## 1) صورة الوضع الحالي

الصفحة الحالية فيها تبويبات جاهزة:
- `overview`
- `client`
- `offer`
- `communication`
- `result`

والصفحة تسحب البيانات من:
- `api.openTasks.get(taskId)`
- `api.openTasks.getActivity(taskId)`
- `api.openTasks.getDevices(taskId)`
- `api.openTasks.getCalls(taskId)`

وهي حالياً تعرض:
- بيانات أساسية للمهمة والزبون
- جدول عروض مسبقة داخل تبويب العرض
- سجل نشاط ومكالمات
- قسم نتيجة يستعمل `TaskResultSection`

**المشكلة الأساسية:**
البيانات صحيحة غالباً، لكن العرض ما يزال خليطاً بين مصادر متعددة، وفيه تكرار/تشويش بين:
- الزيارة
- المهمة
- العروض المسبقة
- النتيجة النهائية

---

## 2) المبادئ اللي لازم نمشي عليها

1. **لا نختلق حقول جديدة** خارج الملف التحليلي إلا إذا كانت موجودة فعلاً بالكود.
2. **نعرّف طبقة تطبيع بيانات** قبل ما نلمس الواجهة.
3. **نفصل العرض عن المنطق**: كل tab يعرض مجموعة محددة من البيانات فقط.
4. **نحافظ على مصطلحات موحدة**:
   - العروض المسبقة
   - التسكير
   - سبب عدم التسكير
   - رد الزبون
   - تم البيع / رفض / مهلة
5. **نستخدم نفس أسلوب العرض المنظم** الذي طبقناه في مودل الإضافة:
   - جداول واضحة
   - نصوص قصيرة
   - حالات فارغة واضحة
   - تنبيهات نقص البيانات بدل الفراغ الصامت

---

## 3) الخطة التنفيذية المقترحة

### المرحلة 1: طبقة تطبيع بيانات موحدة

**الهدف:** منع تشتت الصفحة بين camelCase / snake_case وبين legacy / current fields.

**مطلوب داخل `DeviceDemoDetail.tsx` أو helper قريب منه:**
- توحيد:
  - `task.preOffers` و `task.pre_offers`
  - `task.offers` و `preOffers`
  - `task.result` و `task.outcome`
  - `task.closedByEmployeeName` و `closedByEmployeeId`
  - `task.noClosingReason`
- إنشاء دوال مساعدة لعرض:
  - اسم الجهاز
  - نوع العرض
  - الحسم
  - سبب التسكير / عدم التسكير
  - رد الزبون

**نتيجة هذه المرحلة:**
الصفحة تتعامل مع شكل موحّد للبيانات بدل التفرّع داخل JSX.

---

### المرحلة 2: إعادة ترتيب تبويب `overview`

**الهدف:** تحويله إلى لوحة ملخص واضحة بدل تجميع معلومات متفرقة.

**المحتوى المقترح:**
- بطاقة ملخص سريع:
  - رقم المهمة
  - الحالة
  - الأولوية
  - تاريخ الإنشاء
  - مصدر الإنشاء
- بطاقة الزيارة:
  - تاريخ الزيارة
  - الوقت
  - الارتباط بزيارة أو عدمه
- بطاقة المهمة:
  - السبب
  - الملاحظات
  - آخر تحديث
- بطاقة تنبيهات نقص البيانات:
  - حقول ناقصة فعلية حسب الملف التحليلي

**القاعدة:**
أي نقص لازم يظهر كتنبيه مرئي واضح، مو كسطر ناقص وسط النص.

---

### المرحلة 3: إعادة تنظيم تبويب `client`

**الهدف:** عرض بيانات الزبون بشكل نظيف ومقروء.

**المحتوى المقترح:**
- الاسم
- الهاتف
- المحافظة / المنطقة / الناحية / الحي
- العنوان التفصيلي
- أرقام التواصل
- اختصار إلى `ClientCardPopup` عند الحاجة

**القاعدة:**
اعتماد بيانات `clientSnapshot` أولاً، ثم fallback إلى حقول المهمة إذا كانت snapshot ناقصة.

---

### المرحلة 4: إعادة تشكيل تبويب `offer`

**الهدف:** جعل تبويب العرض هو المكان الأساسي لقراءة كل ما يتعلق بالعروض المسبقة.

**المحتوى المقترح:**
1. **بطاقة الفريق المكلف**
   - مشرف
   - فني
   - متدرب

2. **بطاقة العروض المسبقة** على شكل جدول
   - الجهاز
   - نوع العرض
   - الكمية
   - الإجمالي
   - الحسم
   - حالة رد الزبون
   - التسكير / السبب

3. **تنسيق موحّد للحالات**
   - accepted → تم البيع
   - rejected → رفض
   - extension_requested → مهلة
   - null → بانتظار

**القاعدة:**
نفس منطق التسمية اللي طبقناه في مودل الإضافة، لكن للعرض فقط، بدون حقول إدخال.

---

### المرحلة 5: إعادة ترتيب تبويب `communication`

**الهدف:** فصل التواصل عن النشاط عن الملاحظات بشكل واضح.

**المحتوى المقترح:**
- **مكالمات التيلماركتر**
- **سجل النشاط** كـ timeline
- **الملاحظات** كقائمة مستقلة

**القاعدة:**
إذا مافي بيانات في قسم معين، نعرض empty state واضح، مو فراغ.

---

### المرحلة 6: تحسين تبويب `result`

**الهدف:** جعل النتيجة النهائية لقطة واحدة مفهومة.

**المحتوى المقترح:**
- `outcome`
- `result`
- `completedAt`
- `noClosingReason`
- `contractId` إن وجد
- `saleReferenceNumber` إن وجد
- ملخص العروض المرتبطة

**القاعدة:**
هذا التبويب لازم يكون الخلاصة النهائية، وليس فقط تكراراً لما ظهر في تبويب العرض.

---

## 4) النقاط اللي لازم ننتبه لها من الملف التحليلي

الملف التحليلي يثبت أن الصفحة تتعامل مع 3 مستويات مختلفة:

### أ) طبقة الزيارة
- `status`
- `scheduledDate`
- `scheduledTime`
- `requestedDeviceModelId`
- `requestedDeviceName`
- `technicianNotes`
- `customerName`
- `customerAddress`
- `customerMobile`
- `ownership`

### ب) طبقة المهمة
- `reason`
- `status`
- `result`
- `outcome`
- `priority`
- `closedByEmployeeId`
- `noClosingReason`
- `saleReferenceNumber`
- `contractId`
- `sourceOpenTaskId`

### ج) طبقة العروض
- `deviceModelId`
- `offerType`
- `quantity`
- `totalAmount`
- `firstPaymentAmount`
- `installmentMonths`
- `currency`
- `discountPercentage`
- `closedByEmployeeId`
- `noClosingReason`
- `customerResponse`
- `rejectionReasonId`
- `extensionReasonId`
- `extensionDueDate`

**قاعدة التنفيذ:**
أي قسم في الصفحة لازم يوضح من أي طبقة عم يقرأ، حتى ما نخلط مصدر البيانات.

## 5) الترتيب المقترح للتنفيذ

### Task 1: إضافة طبقة تطبيع بيانات
- توحيد الأسماء والـ fallbacks
- تجهيز helpers لعرض الجهاز/العرض/السبب/الحالة

- ملخص سريع
- بيانات زيارة
- بيانات مهمة
- تنبيهات نقص البيانات

### Task 3: إعادة ترتيب `client`
- snapshot أولاً
- fallback من المهمة
- تحسين العرض البصري

### Task 4: إعادة بناء `offer`
- بطاقة الفريق
- جدول العروض المسبقة
- حالات الرد والتسكير

### Task 5: إعادة تنظيم `communication`
- calls
- activity
- notes

### Task 6: تحسين `result`
- outcome/result summary
- contract / sale reference / no closing reason
- legacy support

### Task 7: QA نهائي
- مهمة فيها عروض مسبقة متعددة
- مهمة بلا عروض
- مهمة فيها legacy data فقط
- مهمة فيها `accepted / rejected / extension_requested`
- مهمة فيها snapshot ناقص

## 7) معايير القبول

يعتبر التنفيذ ناجحاً إذا:
- كل tab صار واضح ومفصول عن غيره
- العروض المسبقة تظهر بشكل جدول منظم ومفهوم
- الحقول الناقصة تظهر كتنبّهات فعلية
- ما عاد في خلط بين البيانات القديمة والجديدة
- النصوص والمصطلحات متوافقة مع مودل الإضافة
- الصفحة تبقى شغالة على الستيجنج بدون أخطاء TypeScript أو build
